第6章《变更检测——Angular 如何知道界面需要更新》

你有没有遇到过这样的场景：你写了一个方法，调用 HTTP 接口，在 Promise 的 .then() 里把返回的数据赋给组件属性，然后你切回浏览器，等待了一秒……界面没有任何变化。你加上 console.log，数据确实更新了。但页面还是旧的。

这个 bug 几乎是每位 Angular 开发者的"成人礼"。第一次遇到它时，你可能会怀疑自己是不是哪里写错了；第二次遇到它时，你已经学会了加上 this.cdr.markForCheck()；但如果你想真正理解为什么要加这一行，以及为什么有时候加了也不管用，就需要深入变更检测的源码。

与浏览器原生 DOM API 不同，Angular 的视图更新是按需触发的。Angular 不会为每个对象属性安装监听器（那样内存和 CPU 开销会非常大），而是选择了一种不同的策略：在特定时机，遍历整个组件树，比较每个绑定值的新旧版本，按需更新 DOM。这个过程就是变更检测（Change Detection）。

正如第2章所分析的，ApplicationRef.tick() 是触发变更检测的"心跳"，NgZone 是负责报时的广播系统。但 tick() 内部究竟发生了什么？哪些组件会被检查？检查的依据是什么？OnPush 策略为什么能减少不必要的检测？Signals 又是如何做到"自动精确更新"的？

在剧院比喻体系里，变更检测就像舞台监督的巡场：在每一幕结束后（每次 tick），监督员（变更检测引擎）巡视所有舞台区域（组件视图），逐一检查演员的状态（绑定值）是否与剧本（数据模型）一致，不一致则立即发出"补妆换景"指令（DOM 更新）。

Default 策略的组件，就像一个每幕都要接受全面检查的演员——不管有没有换台词，监督员都会走一遍。OnPush 组件则是挂了"有事再叫我"牌子的演员——只有导演亲自来拍桌子（markForCheck()），才会进入检查流程。而 Signals 组件更聪明，它能主动举手示意——但只有真的换了台词，才会举手。

理解这三种策略的区别，是 Angular 性能优化的基础。这一章，我们从源码出发，把这套机制彻底讲清楚。

在阅读本章之前，你需要理解第5章建立的基础：每个组件实例对应一个 LView 数组，LView 的特定槽位存储着 DOM 节点引用、绑定值的历史快照，以及描述视图状态的标志位。变更检测引擎的工作，本质上是遍历所有 LView，比较绑定值的新旧版本，按需操作 DOM。整个过程是同步的（在单次 JavaScript 事件循环任务中完成），是可预测的（在确定的时间点触发），也是可调试的（标志位状态随时可以在调试器中查看）。

变更检测全景调用链

在深入任何具体机制之前，先建立对整条调用链的全局认知。从 ApplicationRef.tick() 到单个组件的 DOM 更新，Angular 经过了九层函数调用，横跨多个源文件：

链路分为两层：前五个函数（tick 到 synchronizeOnce）是调度层，负责"什么时候检测、检测哪些根视图、如何处理级联效应"；后四个函数（detectChangesInternal 到 refreshView）是执行层，负责"具体检测这个视图并更新 DOM"。调度层和执行层的分离，是 Angular 同时支持 Zone.js 和 Zoneless 的关键——不管是哪条触发路径，一旦进入执行层，代码路径完全相同。

调度层：synchronize 的稳定性保证

tickImpl() 做的事情比你想象的要多：

// packages/core/src/application/application_ref.ts（第 537-565 行）
private tickImpl(): void {
  this._runningTick = true;
  try {
    this.synchronize();

    // 仅 dev 模式：第二轮检查，验证单向数据流的稳定性
    if (typeof ngDevMode === 'undefined' || ngDevMode) {
      for (const view of this.allViews) {
        view.checkNoChanges();  // ← 产生 ExpressionChangedAfterChecked 错误的地方
      }
    }
  } finally {
    this._runningTick = false;
  }
}

dev 模式的第二轮检查（checkNoChanges）是本章后面要讲的 ExpressionChangedAfterItHasBeenCheckedError 的根源，先记住这里有它。

synchronize() 本身是一个 while 循环：

// packages/core/src/application/application_ref.ts（第 571-595 行）
private synchronize(): void {
  let runs = 0;
  while (this.dirtyFlags !== ApplicationRefDirtyFlags.None) {
    if (runs++ >= MAXIMUM_REFRESH_RERUNS) {
      // 防止无限循环：超过最大重跑次数则抛出错误
      throw new RuntimeError(RuntimeErrorCode.INFINITE_CHANGE_DETECTION, ...);
    }
    this.synchronizeOnce();
  }
}

这个循环保证应用达到稳定状态（dirtyFlags === None）后才停止：一次变更检测期间，effect() 的副作用可能再次写入 Signal，导致新视图需要更新，循环确保所有级联效应都被处理。MAXIMUM_REFRESH_RERUNS（默认 100）是安全阀，防止写入→更新→写入的无限级联。

synchronizeOnce() 是实际执行检测的地方（application_ref.ts:600-671）：先检查并冲刷 rootEffectScheduler（Signal effect 的根调度器，注意这里是 rootEffectScheduler 而非 effectScheduler）；再根据 dirtyFlags 决定是否需要全量或精确检测——使用 requiresRefreshOrTraversal() 跳过无需检测的视图——调用 detectChangesInternal()；最后执行 afterRender 钩子并通过 syncDirtyFlagsWithViews() 同步视图状态。Signal effect 总是先于模板更新被冲刷，确保 effect 里的写入在当前 tick 内得到响应。

LViewFlags：变更检测的"通行证"

在进入检测逻辑之前，我们需要理解一个核心前提：Angular 怎么知道"这个组件现在需不需要检测"？

答案存储在每个 LView 的固定槽位里：lView[FLAGS]。这是一个 32 位整数，每一个二进制位代表这个视图的一种状态。这就是 LViewFlags 枚举——变更检测的"通行证"体系。

把 lView[FLAGS] 想象成一张多格打卡卡，每格对应一个问题的"是/否"答案：这个视图需要全量刷新吗？它有子视图需要刷新吗？它是 Signals 组件吗？巡场时，监督员只需翻一下这张卡，就知道这块舞台今天需要什么处理。

关键标志位解析

// packages/core/src/render3/interfaces/view.ts（第 402-450 行）
export const enum LViewFlags {
  // 位 4：默认策略，Global 模式下无条件刷新
  CheckAlways = 1 << 4,

  // 位 6：OnPush 组件已被标记，等待刷新；刷新后自动清除
  Dirty = 1 << 6,

  // 位 7：视图是否附着在变更检测树上（detach() 会清除此位）
  Attached = 1 << 7,

  // 位 10：强制刷新请求，无论当前是何种检测模式都执行刷新
  // Signal 组件和"移植视图"（transplanted view）使用此位
  RefreshView = 1 << 10,

  // 位 12：Signals 组件标识，刷新完全依赖 consumer.dirty
  SignalView = 1 << 12,

  // 位 13：子代视图树中有视图需要刷新，驱动 Targeted 遍历算法向下导航
  HasChildViewsToRefresh = 1 << 13,
}

这六个标志位是理解整个变更检测算法的基础，它们并非互斥而是组合使用。一个典型 OnPush 组件在 markForCheck() 后，lView[FLAGS] 会同时拥有 Dirty（自身需刷新）+ RefreshView（强制刷新）两个位。它的祖先视图则会拥有 HasChildViewsToRefresh（路径导航标记）。

位操作的意义在于高效：读取 lView[FLAGS] 是一次内存访问，检查标志位只需一条 CPU 指令，同时设置或清除多个标志也是一次原子操作——lView[FLAGS] &= ~(LViewFlags.Dirty | LViewFlags.RefreshView) 一次操作清除两位。在几百甚至几千个组件的应用里，这种效率优势会积累成可感知的性能差距。

三种策略的初始状态

Angular 在创建视图时，根据 ComponentDef 的设置初始化 lView[FLAGS]：

// packages/core/src/render3/view/construction.ts（第 264-280 行）
export function getInitialLViewFlagsFromDef(def: ComponentDef<unknown>): LViewFlags {
  let flags = LViewFlags.CheckAlways;  // 默认策略：无条件刷新

  if (def.signals) {
    // Signals 组件：既不 CheckAlways，也不初始 Dirty
    // 完全由 producer-consumer 响应式图驱动刷新
    flags = LViewFlags.SignalView;
  } else if (def.onPush) {
    // OnPush：初始给 Dirty 允许首次渲染，但没有 CheckAlways
    // 首次 refreshView 后 Dirty 被清除，组件进入"休眠"
    flags = LViewFlags.Dirty;
  }

  return flags;
}
策略初始标志首次渲染后后续唤醒方式
Default（CheckAlways）CheckAlwaysCheckAlways 永久保持无需唤醒，每次 Global tick 都刷新
OnPushDirty（无 CheckAlways）Dirty 被清除，进入休眠markForCheck() 重新设置 Dirty
Signals（SignalView）SignalView（无 CheckAlways/Dirty）SignalView 永久保持signal.set() 设置 consumer.dirty

detectChangesInView() 的四条决策路径

所有的一切，最终汇聚在 detectChangesInView() 的这段决策逻辑里——这是全书最重要的逻辑之一，它决定了视图是否被刷新：

// packages/core/src/render3/instructions/change_detection.ts（第 452-510 行）
function detectChangesInView(lView: LView, mode: ChangeDetectionMode): void {
  const flags = lView[FLAGS];
  const consumer = lView[REACTIVE_TEMPLATE_CONSUMER];  // Signal 消费者节点

  // 路径1：Default 策略 + Global 模式 → 无条件刷新
  let shouldRefreshView = !!(
    mode === ChangeDetectionMode.Global && flags & LViewFlags.CheckAlways
  );

  // 路径2：OnPush 已标记（Dirty）+ Global 模式 → 刷新
  shouldRefreshView ||= !!(
    flags & LViewFlags.Dirty &&
    mode === ChangeDetectionMode.Global &&
    !isInCheckNoChangesPass  // dev 模式二次检查时不走此路径
  );

  // 路径3：RefreshView 标志 → 无论何种模式都强制刷新
  // Signal 组件和移植视图（transplanted view）走此路径
  shouldRefreshView ||= !!(flags & LViewFlags.RefreshView);

  // 路径4：Signal 消费者脏了，且生产者版本号确认发生变化 → 刷新
  // push 通知（consumer.dirty）+ pull 确认（consumerPollProducersForChange）双重机制
  shouldRefreshView ||= !!(consumer?.dirty && consumerPollProducersForChange(consumer));

  // 刷新前先清除标志，允许刷新期间重新被标记（如 effect 写入新值）
  if (consumer) consumer.dirty = false;
  lView[FLAGS] &= ~(LViewFlags.HasChildViewsToRefresh | LViewFlags.RefreshView);

  if (shouldRefreshView) {
    refreshView(tView, lView, tView.template, lView[CONTEXT]);
  } else if (flags & LViewFlags.HasChildViewsToRefresh) {
    // 自身不刷新，但子树有需要刷新的视图 → 继续向下遍历
    detectChangesInEmbeddedViews(lView, ChangeDetectionMode.Targeted);
    detectChangesInChildComponents(lView, components, ChangeDetectionMode.Targeted);
  }
}

请仔细读这段代码。四条路径各自服务一种策略，而 else if 分支（HasChildViewsToRefresh）是 Targeted 模式能够"穿透"中间层视图的关键——即便某个中间组件自身不需要刷新，只要有 HasChildViewsToRefresh 标志，遍历就会继续向下推进。

接下来的三节，我们分别深入这四条路径。

默认策略：脏检查的本质

"脏检查"（Dirty Checking）这个名字，乍一听让人以为是在检查"脏数据"，但实际的含义恰恰相反——它是说"假设什么都可能脏了（变了），统一扫一遍"。

这是 Default 策略（ChangeDetectionStrategy.Default）的工作方式：不管你的数据有没有变，只要持有 CheckAlways 标志，在每次 Global 模式 tick 时，Angular 都会执行 refreshView()。这是路径1在每次心跳里无条件触发的保证。

为什么选择"脏检查"而不是更精确的方式

这是一个务实的决策。Default 策略的价值在于：开发者可以用普通 JavaScript 对象直接修改属性，Angular 会自己发现；心智模型极简，不需要理解"哪些变化能被检测到"；调试友好，状态在已知时间点被检查，不存在"为什么这个变化没有触发更新"的困惑。代价是性能：在组件树深或节点多的应用里，每次 tick 全量扫描会消耗大量 CPU，这也是 OnPush 策略存在的理由。

refreshView() 的工作内容

当 detectChangesInView() 决定刷新某个视图，它调用 refreshView()。这个函数会重新执行组件的 template 函数，但仅执行其中的 Update 模式（rf & RenderFlags.Update）——创建阶段已经在首次渲染时完成，这里只需要更新绑定值：

// packages/core/src/render3/instructions/change_detection.ts（第 185-290 行）
export function refreshView<T>(
  tView: TView,
  lView: LView,
  templateFn: ComponentTemplate<{}> | null,
  context: T,
): void {
  // Signal 追踪开始（稍后第六节详细讲）
  const currentConsumer = getOrBorrowReactiveLViewConsumer(lView);
  consumerBeforeComputation(currentConsumer);

  try {
    if (templateFn !== null) {
      // 执行编译器生成的模板函数，Update 模式只处理绑定更新，不重建 DOM 结构
      executeTemplate(tView, lView, templateFn, RenderFlags.Update, context);
    }

    // 按顺序触发生命周期钩子：ngOnChanges → ngDoCheck → ngAfterContentChecked 等
    // ... 省略生命周期钩子相关代码 ...

    // 刷新嵌入视图（*ngIf、*ngFor 等结构型指令产生的视图）
    detectChangesInEmbeddedViews(lView, ChangeDetectionMode.Global);

    // 刷新子组件
    detectChangesInChildComponents(lView, components, ChangeDetectionMode.Global);

  } finally {
    consumerAfterComputation(currentConsumer, prevConsumer);  // Signal 追踪结束
  }

  // 刷新完成：清除 Dirty 和 FirstLViewPass 标志
  lView[FLAGS] &= ~(LViewFlags.Dirty | LViewFlags.FirstLViewPass);
}

注意最后的 lView[FLAGS] &= ~LViewFlags.Dirty——这正是 OnPush 组件完成首次渲染后进入"休眠"的机制：Dirty 在此被清除，下次 Global 扫描时不再满足路径2的条件。

bindingUpdated()：真正做 DOM 决策的地方

template 函数中的每次属性绑定（[title]="name"）最终都会调用 bindingUpdated()。这是 Angular 的最后一道"节流阀"：

// packages/core/src/render3/bindings.ts（第 44-80 行）
export function bindingUpdated(
  lView: LView,
  bindingIndex: number,
  value: unknown,
): boolean {
  const oldValue = lView[bindingIndex];  // 从 LView 读取上次记录的值

  // Object.is 语义：NaN === NaN，+0 !== -0，精确引用比较
  if (Object.is(oldValue, value)) {
    return false;  // 值没变，跳过所有 DOM 操作
  }

  // dev 模式下的二次检查：若值在两次检测间发生了变化，则抛出错误
  if (ngDevMode && isInCheckNoChangesMode()) {
    throwErrorIfNoChangesMode(oldValue === NO_CHANGE, oldValue, value, ...);
  }

  lView[bindingIndex] = value;  // 更新 LView 里缓存的旧值
  return true;  // 通知调用方：值已变化，需要操作 DOM
}

Object.is(oldValue, value) 是脏检查中真正的比较操作：对于基本类型（string、number、boolean），比较的是值；对于引用类型（对象、数组），比较的是引用地址——[] === [] 不同！这解释了为什么在 template 中直接写 [items]="getItems()" 而 getItems() 每次 return [] 会带来性能问题，以及为什么 OnPush 组件接收的对象应当使用不可变引用（修改属性不创建新引用时，Object.is 返回 true，子组件不会被标记刷新）。

全树扫描只是"决定检查哪些组件"的策略，bindingUpdated() 保证了"真正的 DOM 操作只在值变化时发生"。两者结合：粗粒度触发 + 细粒度比较。

OnPush 策略：精确标记与双轨冒泡

如果 Default 是"每幕都检查全部演员"，OnPush 就是"只检查贴了便利贴的演员"。这个"便利贴"就是 Dirty 标志。

OnPush 是 Angular 中使用最广泛的性能优化策略。但它经常被误解为"不检测变化"。正确的理解是：OnPush 组件是显式请求才检测，而不是永不检测。只要你知道数据变了，Angular 会比 Default 策略更高效地完成更新。

OnPush 组件的生命周期

OnPush 组件的状态变化分为三个阶段：

第一阶段：创建。getInitialLViewFlagsFromDef() 给视图设置初始标志 Dirty（不含 CheckAlways）。

第二阶段：首次渲染。Global 模式下，路径2条件满足（Dirty 为真），执行 refreshView()，渲染 DOM。refreshView() 末尾清除 Dirty。

第三阶段：休眠。Dirty 被清除后，下次 Global 扫描时路径2不再满足，组件被跳过。除非有人来重新设置 Dirty。

这就是"你调用了 API，数据更新了，但界面没有刷新"的根本原因。在 Promise.then() 这样的异步回调里直接赋值，Zone.js 虽然捕获到了事件触发了 tick，但 OnPush 组件的 Dirty 并没有被设置，四条路径中没有一条为真，组件被跳过。

markForCheck()：双轨冒泡机制

要唤醒休眠的 OnPush 组件，需要某种机制重新设置 Dirty，并且让遍历算法能够"找到"这个组件。这是 markForCheck() 的职责，它触发了两条独立的冒泡链：

markForCheck() 触发双轨冒泡：

轨道1（Dirty 冒泡）              轨道2（HasChildViewsToRefresh 冒泡）
ChildComponent                   ChildComponent
   ↑ Dirty + RefreshView            ↑ HasChildViewsToRefresh
  Parent                           Parent
   ↑ Dirty + RefreshView            ↑ HasChildViewsToRefresh
  Root                             Root（到根则停止）

没有轨道2，Targeted 遍历算法遇到中间层组件时，看到没有 HasChildViewsToRefresh，会直接跳过整棵子树——深埋在组件树第5层的 OnPush 组件就永远到达不了。

第一条冒泡链：Dirty 冒泡（markViewDirty()）

// packages/core/src/render3/instructions/mark_view_dirty.ts（第 29-72 行）
export function markViewDirty(
  lView: LView,
  source: NotificationSource,
): LView | null {
  // 正在刷新中：只标 Dirty（避免刷新重入）
  // 非刷新中：同时标 Dirty 和 RefreshView（让 Targeted 模式也能处理）
  const dirtyBitsToUse = isRefreshingViews()
    ? LViewFlags.Dirty
    : LViewFlags.RefreshView | LViewFlags.Dirty;

  // 通知调度器：有视图需要参与下次 tick
  lView[ENVIRONMENT].changeDetectionScheduler?.notify(source);

  // 向上冒泡：对当前视图及每个祖先都设置 dirtyBitsToUse
  while (lView !== null) {
    lView[FLAGS] |= dirtyBitsToUse;
    const parent = getLViewParent(lView);
    if (isRootView(lView) && parent === null) break;  // 到达根视图，停止
    lView = parent!;
  }

  return lView;
}

第二条冒泡链：遍历路径冒泡（markAncestorsForTraversal()）

// packages/core/src/render3/util/view_utils.ts（第 264-295 行）
export function markAncestorsForTraversal(lView: LView): void {
  lView[ENVIRONMENT].changeDetectionScheduler?.notify(
    NotificationSource.MarkAncestorsForTraversal,
  );

  // 向上冒泡：标记 HasChildViewsToRefresh，为遍历算法铺路
  let parent = getLViewParent(lView);
  while (parent !== null) {
    if (parent[FLAGS] & LViewFlags.HasChildViewsToRefresh) break;  // 已标记则停止
    parent[FLAGS] |= LViewFlags.HasChildViewsToRefresh;
    if (!viewAttachedToChangeDetector(parent)) break;  // 脱离 CD 树则停止
    parent = getLViewParent(parent);
  }
}
  • Dirty / RefreshView 回答的是："这个视图自身需要刷新"——满足 detectChangesInView() 的路径2或路径3
  • HasChildViewsToRefresh 回答的是："请沿这条路径继续向下走，子树里有视图需要刷新"——让 Targeted 模式能够导航到目标组件

HasChildViewsToRefresh 从目标组件往上一路冒泡，为 Targeted 模式的遍历算法铺了一条"导航路径"。中间层的 Default 组件同样获得了 Dirty，这意味着它也会在这次 tick 中被刷新——这是正确的行为，因为它的绑定值可能也有更新。

自动触发 markForCheck() 的场景

并非只有手动调用 cdr.markForCheck() 才会触发这个流程。Angular 在多个地方内置了自动调用：

@Input() 引用变化时：父组件刷新、执行 ɵɵproperty 渲染指令（instruction）传递新值时，Angular 编译器生成的 markDirtyIfOnPush() 调用会自动标记 OnPush 子组件：

// packages/core/src/render3/instructions/shared.ts（第 319-335 行）
export function markDirtyIfOnPush(lView: LView, viewIndex: number): void {
  // 如果子组件是 OnPush 策略，且不含 CheckAlways，则标记为 Dirty
  const childComponentLView = getComponentLViewByIndex(viewIndex, lView);
  if (!(childComponentLView[FLAGS] & LViewFlags.CheckAlways)) {
    childComponentLView[FLAGS] |= LViewFlags.Dirty;
  }
}

AsyncPipe（async 管道）：每当 Observable 或 Promise 发出新值，AsyncPipe 内部调用 ChangeDetectorRef.markForCheck()。这也是为什么在 OnPush 组件里，使用 async 管道代替手动订阅是推荐做法——它自动处理了标记逻辑。

手动调用：this.cdr.markForCheck() 或 this.cdr.detectChanges()（detectChanges 是立即同步执行，markForCheck 是标记后等待下次 tick）。

⚠️ 注意：markForCheck() 不会立即刷新视图。它只是设置标志位，并通知调度器"下次 tick 需要检测"。实际的 DOM 更新发生在下一个变更检测周期。这个区别在高频更新场景下很重要：批量写入后调用一次 markForCheck()，而不是写一次调一次。

💡 提示：ChangeDetectorRef.detach() 会清除视图的 Attached 标志，使该视图完全脱离变更检测树——不管是 Global 还是 Targeted 模式，都不会访问这个视图。但 detectChanges() 仍然有效（它是直接对这个视图执行检测，不走遍历逻辑）。这是实现自定义调度（如虚拟滚动、大型 table）的底层机制。

Global 与 Targeted：两种遍历模式

到目前为止，ChangeDetectionMode.Global 和 ChangeDetectionMode.Targeted 已经多次出现。现在是时候把它们讲清楚了。

两种模式的含义

这两种模式控制了 detectChangesInView() 如何决定是否刷新视图，以及遍历算法如何导航视图树：

// packages/core/src/change_detection/constants.ts
export const enum ChangeDetectionMode {
  // Global：全量检查——处理所有 CheckAlways 视图，以及当前标记了 Dirty 的 OnPush 视图
  Global = 0,

  // Targeted：精确检查——只沿 HasChildViewsToRefresh 标记的路径向下走
  // 跳过没有 HasChildViewsToRefresh 的子树，大幅减少遍历开销
  Targeted = 1,
}

Global 模式就像是"全场巡视"：遍历整棵视图树，对每个视图都问一遍"有 CheckAlways 吗？有 Dirty 吗？"任意一个为真就刷新。这是 Zone.js 路径的默认行为，保证即便开发者完全不使用 OnPush 或 Signals，应用也能正确更新。

Targeted 模式则是"精准导航"：只跟随 HasChildViewsToRefresh 标志向下走，没有这个标志的子树会被整体跳过。遍历开销与"被标记的视图数量"成正比，而不是与"总视图数量"成正比，在大型应用中性能优势显著。

synchronizeOnce() 如何选择模式

两种模式的选择逻辑在 synchronizeOnce() 里：

// packages/core/src/application/application_ref.ts（第 635-670 行）
private synchronizeOnce(): void {
  this.effectScheduler.flush();

  // dirtyFlags 记录了这次 tick 由什么触发：
  // ViewTreeGlobal：由 Zone.js / 手动 tick 触发 → 需要 Global 模式
  // ViewTreeTraversal：由 Signal 写入或 markForCheck 触发 → 只需 Targeted 模式
  const useGlobalCheck = !!(
    this.dirtyFlags & ApplicationRefDirtyFlags.ViewTreeGlobal
  );

  for (const view of this.allViews) {
    if (!view.dirty) continue;

    const mode = useGlobalCheck
      ? ChangeDetectionMode.Global
      : ChangeDetectionMode.Targeted;

    detectChangesInternal(view._lView, mode);
  }
}

两种模式并非互斥。在一次 synchronize() 的循环中，可能先以 Global 模式处理第一轮（Zone.js 触发，ViewTreeGlobal 标志为真），完成后 dirtyFlags 被清除；然后 Signal effect 的副作用写入了新值，再次触发 ViewTreeTraversal，进入下一轮 Targeted 模式。这就是为什么 synchronize() 是 while 循环而非单次调用。

Signals 集成：第三种范式

理解了 Default（全量扫描）和 OnPush（手动精确标记）之后，Signals 是第三种范式——自动精确。

OnPush 需要你（或框架）显式调用 markForCheck() 告诉 Angular"这个组件的数据可能变了"。Signals 则不需要：当你写入 signal 时，Angular 自动知道哪些组件读取过这个 signal，并且只刷新那些组件。两者的本质差异一目了然：

维度OnPush + markForCheck()Signals（SignalView）
唤醒的触发者外部显式调用（开发者或 Angular 自动）signal.set() 自动触发
依赖关系管理开发者负责（知道哪些数据影响哪些视图）框架自动追踪（模板读取建立订阅）
假脏过滤无（标记即认为需要刷新）consumerPollProducersForChange() 过滤
参与 Global 扫描是（Dirty 在 Global 模式下被检测）否（SignalView 不受 Global 影响）
适用于 Zone.js 应用是是（通知链最终使用相同的底层机制）
适用于 Zoneless 应用是（需 Targeted 模式）是（天然 Targeted）

这就像是在 OnPush 的基础上，把"举手示意"的工作从开发者手中接管——演员读了哪几幕的台词（读取了哪些 Signal），在台词更新时（signal.set()），Angular 知道只需要重演读过这些台词的场景。

SignalView：不参与 Global 扫描的视图

标记了 signals: true 的组件（Angular v19 的 signal-based components），其 LView 拥有 SignalView 标志，而不是 CheckAlways 或初始 Dirty。

这意味着 detectChangesInView() 的路径1和路径2永远不会为这类组件触发刷新。它的刷新完全依赖路径3（RefreshView）或路径4（consumer.dirty && consumerPollProducersForChange()）。

即便在 Zone.js 路径的 Global 模式 tick 中，SignalView 组件也不会被无条件刷新——它是 Signals 世界的居民，遵循不同的规则。

Signal 的三阶段工作机制

Signal 与变更检测的集成精确地分为三个阶段：追踪、通知、确认。理解这三个阶段，是理解 Signals 精确更新的关键。

阶段1：追踪（每次 refreshView 时重建依赖）

每次 refreshView() 开始时，Angular 把这个视图的 ReactiveLViewConsumer 设置为"当前活跃消费者"：

// packages/core/src/render3/instructions/change_detection.ts（refreshView 内部）
const currentConsumer = getOrBorrowReactiveLViewConsumer(lView);
// 激活：任何 signal() 调用都会向这个 consumer 注册依赖
consumerBeforeComputation(currentConsumer);

try {
  executeTemplate(tView, lView, templateFn, RenderFlags.Update, context);
} finally {
  // 还原：关闭追踪窗口，清理不再使用的旧依赖链接
  consumerAfterComputation(currentConsumer, prevConsumer);
}

当模板函数执行时，每次读取 mySignal() 都会触发 producerAccessed()，在 signal 节点和视图 consumer 之间建立一条 ReactiveLink——记录 signal 的当前版本（用于后续版本比较）和指向 consumer 的引用（用于发送通知）。追踪结束后，consumerAfterComputation() 会清理那些这次执行中没有被访问的旧链接（条件渲染场景：*ngIf="showA" 从 true 变为 false，那些只在 showA 时才读取的 signal 依赖会被移除）。

阶段2：通知（signal.set() 触发）

当某处代码调用 mySignal.set(newValue)，推进过程沿着 ReactiveLink 链条传播：

// packages/core/primitives/signals/src/signal.ts（set 方法）
set(newValue: T): void {
  this.rawValue = newValue;
  this.version++;           // 该 signal 的版本号自增
  producerIncrementEpoch(); // 全局 epoch 自增（逻辑时钟）
  producerNotifyConsumers(this);  // 通知所有订阅了此 signal 的消费者
}

producerNotifyConsumers 遍历所有 ReactiveLink，对每个消费者调用 consumerMarkDirty()。ReactiveLViewConsumer 的实现最终调用 markAncestorsForTraversal()——这与 OnPush markForCheck() 的第二条冒泡链完全相同。Signal 并没有发明全新的变更检测路径，它只是接管了"何时标记"的工作。

阶段3：确认（Pull 验证，防止假脏）

consumer.dirty = true 只是一个"可能有变化"的 push 通知，它无法排除假脏场景：

const a = signal(1);
const b = signal(-1);
const sum = computed(() => a() + b());  // sum = 0

a.set(2);   // sum consumer 被标记 dirty
b.set(-2);  // 取消了上面的变化，sum 依然 = 0

如果 push 通知直接触发刷新，依赖 sum 的视图会被刷新，但 sum 的值根本没有改变。consumerPollProducersForChange() 的 pull 确认解决了这个问题：

// packages/core/primitives/signals/src/graph.ts（第 350-385 行）
export function consumerPollProducersForChange(node: ReactiveNode): boolean {
  for (let link = node.producers; link !== undefined; link = link.nextProducer) {
    const producer = link.producer;
    // 快速路径：版本号不同 → 确认有变化
    if (link.lastReadVersion !== producer.version) return true;
    // 慢速路径：对于 computed，强制它重新求值
    producerUpdateValueVersion(producer);
    if (link.lastReadVersion !== producer.version) return true;
  }
  return false;  // 所有 producer 版本都没变 → 假脏，不刷新
}

"push 通知 + pull 确认"双重机制：push 是快速告知（O(1)，不需要遍历依赖图），pull 是精确验证（O(n)，n 是直接依赖数量，通常很小）。两者结合，避免假脏导致的无效渲染。

Signal 的依赖关系是动态的——每次模板执行都会重建。假设模板里有 {{ showDetails ? detail() : summary() }}，当 showDetails 为 false 时，detail signal 的依赖链接在模板执行结束后被清理；当 showDetails 变为 true，下次执行时重新建立订阅。这种动态依赖追踪不需要手动管理订阅生命周期，比静态订阅（如 RxJS Subject）更精确。

🔗 交叉引用：Signal 系统的完整实现（ReactiveNode、computed、effect、依赖图的数据结构）将在第7章详细展开。本章只涉及 Signal 与变更检测引擎的接口层。

ExpressionChangedAfterItHasBeenCheckedError

这是 Angular 开发者最常见的错误之一。它的全名让人望而生畏，但它实际上是 Angular 在帮你发现一个真实的数据流问题——一个在生产环境会静默出现、导致 UI 状态不一致的 bug。

根因：dev 模式的二次检查

回顾 tickImpl() 的结构。正常的变更检测（synchronize()）完成后，dev 模式下还会对每个视图再执行一次 checkNoChanges()：

// packages/core/src/application/application_ref.ts（第 537-565 行）
private tickImpl(): void {
  this._runningTick = true;
  try {
    this.synchronize();  // ← 第一轮：执行正常变更检测，更新 DOM

    if (ngDevMode) {
      for (const view of this.allViews) {
        // ← 第二轮：仅 dev 模式，不更新 DOM，只验证值是否稳定
        view.checkNoChanges();
      }
    }
  } finally {
    this._runningTick = false;
  }
}

第二轮检查走的是 checkNoChangesInternal()，它重新运行一遍 detectChangesInternal()，但进入了特殊模式：isInCheckNoChangesMode() 返回 true。在这个模式下，bindingUpdated() 发现新旧值不同时，不是更新 LView，而是抛出错误。

这个逻辑的含义是：如果两次检测之间，template 函数对同一个绑定的求值结果发生了变化，就说明有代码在"变更检测进行中修改了数据"——Angular 单向数据流（从数据模型流向 DOM）原则被破坏。

三种典型触发场景

场景一：在 ngAfterViewInit 里修改父组件的绑定值

// 错误写法：会触发 ExpressionChangedAfterChecked
@Component({
  template: `<h1>{{ title }}</h1> <child></child>`,
})
class ParentComponent {
  title = 'initial';
}

@Component({ selector: 'child' })
class ChildComponent implements AfterViewInit {
  constructor(private parent: ParentComponent) {}

  ngAfterViewInit() {
    // 子组件初始化时修改父组件属性
    // 此时父组件的变更检测已经完成，title 的旧值已经写入 LView
    this.parent.title = 'updated';  // ← 触发错误！
  }
}

修复：使用 Promise.resolve() 延迟到下一个微任务（下一次 tick）：

ngAfterViewInit() {
  Promise.resolve().then(() => { this.parent.title = 'updated'; });
}

场景二：绑定表达式每次调用返回新引用

// 错误写法
@Component({
  template: `<child [items]="getItems()"></child>`,
})
class ParentComponent {
  // 每次调用都返回一个新数组——Object.is 比较时永远不等！
  getItems() { return ['a', 'b', 'c']; }
}

修复：缓存为组件属性：

items = ['a', 'b', 'c'];  // 稳定引用

场景三：ngDoCheck 里修改了绑定

ngDoCheck 在每次变更检测时都会被调用，包括第二轮检查。如果在 ngDoCheck 里修改了任何绑定值，第二轮检查时会再次执行 ngDoCheck，发现值变了，触发错误。

场景根因推荐修复
ngAfterViewInit 修改父绑定父组件检测已结束后才修改Promise.resolve().then(...) 延迟到下一 tick
方法返回新引用每次调用创建新对象/数组缓存为稳定属性
ngDoCheck 修改绑定钩子在检测中被调用移到 ngAfterViewChecked，或用 ChangeDetectorRef.detectChanges()
双向绑定形成反馈A 更新 B，B 更新 A重审数据流，打破循环

修复方案优先级：

  1. 优先：重设计数据流，打破循环——让变化发生在变更检测开始前，这是真正修复
  2. 次选：使用 markForCheck() 配合 async 管道，或将方法调用缓存为稳定属性
  3. 最后手段：setTimeout / Promise.resolve() 延迟——治标不治本，只在无法重构时使用

⚠️ 注意：这个错误在生产环境（ngDevMode = false）下不会抛出。Angular 通过 tree-shaking 移除了 throwErrorIfNoChangesMode() 调用。但数据不同步的 bug 会以更隐蔽的方式存在（UI 显示的值可能是旧的）。发现这个错误应当视为修复数据流的机会，而不是通过 setTimeout 简单绕过。

🔍 深入：如果你想在项目里临时禁用这个错误检查（不推荐，但有时需要渐进式迁移），可以使用 provideCheckNoChangesConfig({ exhaustive: false }) 将检查模式改为"仅检查脏视图"而非全量检查。这个 API 在 Angular v17+ 可用，主要用于大型迁移场景。

设计思考

设计思考

脏检查的务实哲学

2016年，Angular 2.0 发布时，细粒度响应式（Vue 的 defineProperty、MobX 的 observable）已经被业界验证可行。Angular 团队选择 Zone.js + 全树扫描的脏检查路线，背后有几个务实考量：

零侵入性：开发者可以使用普通 JavaScript 对象，直接修改属性，Angular 会自己检测到变化。这大幅降低了从 AngularJS 迁移的学习成本，也让 Angular 与任意第三方 JavaScript 库的集成变得简单——不需要任何包装或 observable 化。可预测性：变更检测在已知的时间点（Zone tick 结束后）统一发生，开发者可以明确推理"哪个时机 DOM 会更新"。

从 2016 到 2024 的演进轨迹

Angular 2.0（2016）上线了 Default + OnPush 双策略，确立了"脏检查 + 可选精确标记"的路线。OnPush 本质上是脏检查的半步进化：保留了"只要调用 markForCheck() 就能正常工作"的简单性，同时允许对数据流清晰的组件主动退出全树扫描。这种渐进式优化非常 Angular：你可以在任意时候，对任意组件单独启用 OnPush，不需要改变整个应用的架构。

随后数年，社区对 Signals 模式的兴趣持续升温。Angular v16（2023）引入开发者预览版 Signals，v17 引入 signal-based components 实验特性，v18 引入 Zoneless 开发者预览，v19（2024）将 Zoneless 模式稳定化。这条路线之所以花了八年：Signals 的精确性要求数据从源头就以 signal 形式流动；早期缺乏编译器支持和生态工具处理混合模式；OnPush 是更现实的过渡方案。v21+ 计划将 Zoneless 设为默认模式。

Signals 是不是 Zone.js 的终结者？

从技术路线上看，Angular 团队的答案是"是"——但这是一个渐进式的、向后兼容的迁移。Zone.js 的根本问题在于它的全局性：它对 setTimeout、Promise、addEventListener 等异步 API 进行了全量 monkey-patch，在大型应用中，任何第三方库触发的异步操作都可能意外触发一次全树检测。

本章的源码分析已经揭示：Signal 的通知链（markAncestorsForTraversal）和 OnPush 的 markForCheck() 使用的是完全相同的底层机制。Angular 并没有为 Signals 构建全新的变更检测引擎，而是复用了 OnPush 的基础设施，让 Signal 成为了"自动化的 OnPush"。这意味着从 Zone.js 迁移到 Zoneless + Signals 的路径是平滑的：OnPush 组件不需要修改，Signal 组件直接参与相同的遍历算法，按组件粒度逐步迁移。

三种策略的权衡总结

Default 策略的价值不应该被低估：它让开发者专注于业务逻辑，在组件树不深、节点不多的应用里完全够用。OnPush 的价值在于中大型应用的局部精确控制，但它要求开发者对数据流有清晰的认知。Signals 则是最高精确度但也最"系统化"的方案——它要求数据通过 signal 流动，不能混用普通赋值。三种策略共存，是 Angular 在"易用性"与"性能"这个经典权衡上给出的多层次答案。

小结

本章从 ApplicationRef.tick() 出发，逐层解析了 Angular 变更检测的完整机制：

  • 调度层（tick → synchronize → synchronizeOnce）负责决定何时检测、检测多少轮、处理 Signal effect 的级联效应。synchronize() 的稳定性循环保证应用在每次 tick 后达到收敛状态。
  • 执行层（detectChangesInView → refreshView → bindingUpdated）负责具体的视图刷新和 DOM 比较。detectChangesInView() 的四条路径是整个算法的核心决策点，分别对应 Default、OnPush、移植视图和 Signals 四种场景。
  • 三种策略本质上是三种唤醒机制：Default 由 Zone.js 全局广播唤醒（每次都），OnPush 由 markForCheck() 手动精确唤醒（显式请求），Signals 由响应式图自动精确唤醒（读取建立订阅，写入自动通知）。
  • markForCheck() 的双轨冒泡（Dirty 链 + HasChildViewsToRefresh 链）是 OnPush 和 Signals 共用的底层路径标记机制。理解它是排查变更检测异常的关键。
  • ExpressionChangedAfterChecked 是数据流健康的早期预警，修复优先级：重设计数据流 > markForCheck()/async 管道 > setTimeout（最后手段）。
  • Signals 不是变更检测的替代品，而是触发机制的升级。refreshView() 仍然是最终执行模板更新的地方；Signal 改变的是"谁来告诉 Angular 何时该执行 refreshView"。这意味着你对变更检测的理解（LViewFlags、双轨冒泡、bindingUpdated 等）在 Signals 时代依然有效，不会过时。

性能优化的路径：Default → OnPush → Signals，越往后精确度越高，代码改动和心智成本也越大。对于新项目，Signals + Zoneless 是 Angular 官方推荐的方向；对于存量项目，OnPush 是最实用的渐进式优化手段。决定使用哪种策略之前，先用 Angular DevTools 的 profiler 测量，找到真正的性能瓶颈，再有针对性地优化，而不是无差别地给所有组件加 OnPush。

思考题

  1. 在一个 OnPush 组件中，父组件通过 @Input() 传递了一个对象，父组件随后修改了这个对象的某个属性（但对象引用没有改变），子组件的 OnPush 视图会刷新吗？为什么？如果想让子组件检测到这种变化，有哪些方案？
  1. ChangeDetectorRef.detach() 之后，再调用 markForCheck()，视图会参与下次 tick 的变更检测吗？查阅 detectChangesInView() 中 Attached 标志的检查逻辑，分析其行为。（提示：Attached 标志影响的是 viewAttachedToChangeDetector() 的返回值，它被 markAncestorsForTraversal() 用于决定是否继续冒泡。）
  1. 一个 Signals 组件（SignalView）在 ngAfterViewInit 里调用 this.count.set(1)（count 是 template 中读取过的 signal），会触发 ExpressionChangedAfterItHasBeenCheckedError 吗？结合 Signal 三阶段机制和 dev 模式二次检查的时序，分析其原因。
