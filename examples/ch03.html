第3章：依赖注入——框架的神经系统

你在一个组件的构造函数里写下 inject(UserService)，按下保存，应用正常运行。但有时候，你在一个非常普通的位置写了同样的调用，却收到一条令人困惑的错误：

NG0203: inject() must be called from an injection context...

或者更常见的：

NullInjectorError: No provider for UserService!

这两条错误背后，是同一套机制在工作——Angular 的依赖注入（Dependency Injection，DI）系统。当你理解了这套系统的内部结构，这些错误就不再神秘：每一条报错，都是 DI 系统在告诉你"我现在到哪一步了，然后卡在了哪里"。

本章的目标，是让你能追踪任意一次 inject() 调用的完整旅程——从调用发生的那一刻，到实例被返回（或错误被抛出）的每一步。

⚠️ 版本说明：本章源码引用基于 Angular main 分支（撰写时约为 v19.x）。由于 Angular 持续演进，实际行号可能与发布版本有出入——建议以本书提供的函数名在当前版本中搜索定位。

两棵注入器树

在进入代码之前，我们先建立一个全局认知：Angular 的 DI 系统不是一个单一的容器，而是两棵并行的树。

第一棵：EnvironmentInjector 树。 这棵树存在于组件树之外，是应用级别的 provider 仓库。树的根节点是 root 注入器（对应 providedIn: 'root' 的服务），往下可以有路由层级的子注入器（懒加载路由模块各自有一棵）。

第二棵：NodeInjector 树。 这棵树与组件树完全同构——每一个组件节点、每一个指令（directive）节点，都对应一个 NodeInjector。它存储着该节点 providers 和 viewProviders 中声明的 provider。

两棵树的关系如下：

flowchart TD subgraph ENV["EnvironmentInjector 树"] direction TB ROOT["Root Injector<br/>(R3Injector)<br/>providedIn:'root' 的服务"] LAZY["LazyRoute Injector<br/>(R3Injector)<br/>路由懒加载模块的服务"] NULL["NullInjector<br/>(最终兜底，抛错)"] ROOT --> LAZY NULL --> ROOT end subgraph NODE["NodeInjector 树（组件树）"] direction TB APP["AppComponent<br/>NodeInjector"] PARENT["ParentComponent<br/>NodeInjector"] CHILD["ChildComponent<br/>NodeInjector"] APP --> PARENT PARENT --> CHILD end CHILD -- "找不到时<br/>委托给" --> ROOT

这里引入本章的主比喻：DI 系统就像一套连锁餐厅的点单系统。你（组件）只需要报出菜名（token），不需要知道厨房（工厂函数）怎么做；餐厅先在本店找（NodeInjector），找不到再向总部仓库申请（EnvironmentInjector），总部仓库也没有，就报错（NullInjector）。

  • EnvironmentInjector = 连锁餐厅的总部仓库，存放全链共享的食材（应用级服务）
  • NodeInjector = 服务员的随身小推车，每个组件节点的服务员推着自己的小推车，车上只放本组件 providers 里的东西

🔗 NodeInjector 不是一个独立的对象，而是 LView 中的两个槽位——这个关键细节将在 NodeInjector 的秘密 一节展开。

💡 设计考量：Angular DI 的双层架构解决了一个根本矛盾：全局单例（应用级服务，如 HttpClient、Router）和局部隔离（组件级服务，如表单状态服务）需要共存。如果所有 provider 都在一个扁平容器里，组件级的 provider 会污染全局命名空间；如果每个组件有完全独立的容器，全局单例就需要复杂的传递机制。两层树的设计，优雅地切分了职责：NodeInjector 负责局部隔离，EnvironmentInjector 负责全局共享，查询时从局部到全局自动委托。

inject() 的完整旅程

现在我们来追踪一次 inject(UserService) 调用的完整路径。

第一关：注入上下文检查

inject() 函数在执行任何查找之前，先做一件事：确认自己是否在"注入上下文（injection context）"中。就像餐厅的点单机器会先验证你是否身在餐厅内部——在门外是无法下单的。

// packages/core/src/di/injector_compatibility.ts（第 88-120 行）
export function injectInjectorOnly<T>(token: ProviderToken<T>, flags = InternalInjectFlags.Default): T | null {
  const currentInjector = getCurrentInjector();

  if (currentInjector === undefined) {
    // ❌ 未在注入上下文中 → 抛出 NG0203
    throw new RuntimeError(RuntimeErrorCode.MISSING_INJECTION_CONTEXT, ...);
  } else if (currentInjector === null) {
    // ⚠️ Limp mode：启动早期，无活跃注入器
    return injectRootLimpMode(token, undefined, flags);
  } else {
    // ✅ 正常路径：调用当前注入实现
    return currentInjector.retrieve(token, options);
  }
}

getCurrentInjector() 返回的是一个全局变量。这个变量在三种情况下会被设置：

状态含义
undefined未在注入上下文中（构造函数外、异步回调后）
null应用启动极早期，进入 Limp mode
PrimitivesInjector 实例正常的注入上下文，可以查找

这就解释了那条 NG0203 错误：inject() 在检查上下文时，发现全局变量是 undefined——没有人在调用前设置它。

💡 Limp mode 是 Angular 的一个降级处理：当 providedIn: 'root' 的服务在无注入器的情况下被调用时，直接从服务类的 ɵprov.value 读取缓存的单例实例。这通常发生在单元测试的快速初始化场景中。

第二关：选择执行路径

通过上下文检查后，inject() 需要决定走哪条路。Angular 维护着一个全局的"注入实现（inject implementation）"变量：

  • 当值为 undefined 时（默认），走 EnvironmentInjector 路由
  • 当值为 directiveInject 函数时（在组件/指令构造时设置），走 NodeInjector 路由

这个切换机制是 Angular DI 双层架构的关键：在组件树渲染期间，Angular 会将注入实现切换为 directiveInject，这样 inject() 就能找到 NodeInjector。渲染引擎在每次组件构造前后，精确地设置和重置这个全局变量：

sequenceDiagram participant RE as 渲染引擎 participant DI as injectImplementation participant Ctor as 组件构造函数 RE->>DI: setInjectImplementation(directiveInject) RE->>Ctor: new ComponentClass() Note over Ctor: inject() 调用走 NodeInjector 路由 Ctor-->>RE: 实例已创建 RE->>DI: setInjectImplementation(undefined)

三个阶段的状态对照：

阶段injectImplementation 值inject() 行为
组件构造前directiveInject走 NodeInjector 路由，查找组件树 providers
组件构造中directiveInject可访问当前节点及所有祖先节点的 providers
组件构造后undefined（默认）走 EnvironmentInjector 路由

💡 设计考量：injectImplementation 全局变量的切换机制（undefined ↔ directiveInject），是 Angular 实现"同一个 inject() 函数，在不同上下文走不同路径"的核心手段。这种全局状态虽然看起来不够"纯粹"，但它避免了在 inject API 上暴露上下文参数，使用户代码保持简洁。

第三关：NodeInjector 链

如果走 NodeInjector 路由，调用链是：

directiveInject(token)
  → getOrCreateInjectable(tNode, lView, token, flags)
    → lookupTokenUsingNodeInjector(...)  // 在组件树向上搜索
      → lookupTokenUsingModuleInjector(...)  // 交给 EnvironmentInjector

NodeInjector 从当前组件节点开始，沿着组件树向上遍历，直到找到提供 token 的节点，或者到达树的顶端。到达顶端还没找到，就将查询委托给 EnvironmentInjector。

第四关：EnvironmentInjector 链

R3Injector.get() 是 EnvironmentInjector 的实现核心。它在自己的 records Map 中查找，找不到就委托给父级 R3Injector，最终到达 NullInjector。

// packages/core/src/di/r3_injector.ts（第 376-383 行）
// Self 标志：只查本注入器，禁止向上委托
const nextInjector = !(flags & InternalInjectFlags.Self)
  ? this.parent
  : getNullInjector();

return nextInjector.get(token, notFoundValue);

第五关：NullInjector 抛错

NullInjector 是整条链的终点。它的 get() 方法非常简单：如果传入了 Optional 标志，返回 null；否则，抛出 NullInjectorError。

那条看起来"莫名其妙"的报错，就是 NullInjector 在说："我是最后一个注入器，token 已经穿透了所有注入器，但谁都没提供它。"

flowchart LR START["inject(UserService)"] CTX{"注入上下文\n存在？"} NODE["NodeInjector\n(当前节点)"] NODE_UP["NodeInjector\n(父组件)"] ENV["EnvironmentInjector\n(Root)"] NULL["NullInjector\n抛出错误"] FOUND["✅ 返回实例"] START --> CTX CTX -- "undefined → NG0203" --> NULL CTX -- "有效" --> NODE NODE -- "未找到，向上" --> NODE_UP NODE_UP -- "穿透组件树" --> ENV ENV -- "找到" --> FOUND ENV -- "未找到" --> NULL

⚠️ @Self() 和 @SkipSelf() 装饰器（以及 inject() 的 self/skipSelf 选项）正是通过修改这次旅程的起点和终点来工作的：@Self() 让旅程只经过当前节点就结束；@SkipSelf() 则从父节点开始出发。

R3Injector 的 token 解析

当查询到达 EnvironmentInjector（即 R3Injector）时，它的 get() 方法需要做三件事：找到 provider、实例化它、缓存结果。

provider 的四种形式

Angular 的 provider 配置有四种写法，R3Injector 在初始化时会把它们全部转换成统一的工厂函数（factory function）：

// packages/core/src/di/r3_injector.ts（第 639-681 行）
// useValue：直接返回固定值，最简单
if (isValueProvider(provider)) {
  factory = () => resolveForwardRef(provider.useValue);
}

// useFactory：调用工厂函数，先注入依赖
if (isFactoryProvider(provider)) {
  factory = () => provider.useFactory(...injectArgs(provider.deps || []));
}

// useExisting：代理到另一个 token（别名机制）
if (isExistingProvider(provider)) {
  factory = (_, flags) => ɵɵinject(resolveForwardRef(provider.useExisting), flags);
}

// useClass / 直接提供类型：实例化指定类
const classRef = resolveForwardRef(provider.useClass || provider.provide);
factory = () => new classRef(...injectArgs(provider.deps));

这四种形式最终都变成一个 () => T 形式的函数，存储在 Record<T> 的 factory 字段中。

惰性实例化：hydrate()

R3Injector 不会在注册 provider 时立刻创建实例。它用两个特殊的哨兵值来控制惰性初始化：

  • NOT_YET：尚未实例化
  • CIRCULAR：正在实例化中（用于检测循环依赖）
// packages/core/src/di/r3_injector.ts（第 524-549 行）
private hydrate<T>(token: ProviderToken<T>, record: Record<T>): T {
  if (record.value === CIRCULAR) {
    // 检测到循环依赖：A 依赖 B，B 又依赖 A
    throw cyclicDependencyError(stringify(token));
  }
  if (record.value === NOT_YET) {
    record.value = CIRCULAR;  // 标记"正在处理"，防止递归
    record.value = record.factory!(undefined, flags);  // 执行工厂函数
    // 如果服务有 ngOnDestroy，注册清理钩子
    if (hasOnDestroy(record.value)) {
      this._ngOnDestroyHooks.add(record.value);
    }
  }
  return record.value as T;
}

实例一旦创建，就存储在 record.value 中。下次查询同一个 token，直接返回缓存值，不再执行工厂函数——这就是"单例"的实现原理。

multi provider：收集所有实现

{ provide: HTTP_INTERCEPTORS, useClass: MyInterceptor, multi: true } 这样的写法，是通过 multi 字段实现的。R3Injector 对 multi: true 的 provider 特殊处理：

// packages/core/src/di/r3_injector.ts（第 497-522 行）
if (!isTypeProvider(provider) && provider.multi === true) {
  let multiRecord = this.records.get(token);
  if (!multiRecord) {
    // 第一次：创建一个收集器 Record，工厂函数返回 multi 数组的所有实例
    multiRecord = makeRecord(undefined, NOT_YET, true);
    multiRecord.factory = () => injectArgs(multiRecord!.multi!);
    this.records.set(token, multiRecord);
  }
  // 将每个 provider 添加到收集器的 multi 数组
  multiRecord.multi!.push(provider);
}

当你调用 inject(HTTP_INTERCEPTORS) 时，收集器的工厂函数被调用，它依次实例化 multi 数组中的每一个 provider，返回实例数组。

NodeInjector 的秘密

NodeInjector 的实现有一个让很多人惊讶的细节：它不是一个真实的对象。

在 Angular 的 render3 渲染引擎中，LView 是一个大数组，存储视图的所有运行时状态（DOM 节点、组件实例、绑定值等）。每个组件节点的 NodeInjector，其实是这个数组中的特定位置——一个"TNODE 槽"和一个"PARENT 槽"。

LView 数组（示意）：
┌─────────┬──────────┬───────────────────────────────────────────┐
│ 固定槽  │ 状态槽   │ ... 节点数据 ...                          │
├─────────┼──────────┼─────────────────────────────────────────┤
│ INJECTOR│ TVIEW    │ [injectorIndex+0..7]  ← Bloom Filter 桶  │
│ (索引3) │ (索引1)  │ [injectorIndex+8]     ← TNODE 引用        │
│         │          │ [injectorIndex+9]     ← PARENT 位置        │
└─────────┴──────────┴─────────────────────────────────────────┘

继续用餐厅比喻：NodeInjector 不是服务员手里拿着的独立推车，而是餐厅地板上划出的一块区域——这块区域里有当前服务员的物品（providers），以及一张写着"找我上级服务员去哪片区域找"的纸条（PARENT 槽）。

向上遍历算法

NodeInjector 的查找算法是一个循环，沿着 PARENT 槽不断向上：

// packages/core/src/render3/di.ts（第 578-615 行）
while (injectorIndex !== -1) {
  // 先用 Bloom Filter 快速检查此节点是否可能有这个 token
  if (bloomHasToken(bloomHash, injectorIndex, tView.data)) {
    const instance = searchTokensOnInjector(injectorIndex, lView, token, ...);
    if (instance !== NOT_FOUND) return instance;
  }
  // 读取 PARENT 槽，跳到父节点的注入器
  parentLocation = lView[injectorIndex + NodeInjectorOffset.PARENT];
  injectorIndex = getParentInjectorIndex(parentLocation);
  lView = getParentInjectorView(parentLocation, lView);
}

PARENT 槽存储的 RelativeInjectorLocation 是一个整数，编码了两个信息：低 16 位是父注入器在 LView 中的索引，高 16 位是需要跨越的视图层数。这样一个整数就能导航到任意层级的父节点，而不需要维护对象引用链。

🔍 深入：Bloom Filter 快速检查

以下内容是可选的深层探讨。如果你只想理解 DI 的工作原理，可以直接跳到下一节。

NodeInjector 在详细查找前，先做一次 Bloom Filter 过滤。这是一种概率型数据结构：如果 Bloom Filter 说"没有"，则一定没有；如果说"可能有"，才进行真正的线性查找。

实现细节：每个注入器节点占用 TView.data 中的 8 个连续整数槽（共 256 位），构成 Bloom Filter。每当向节点注册一个 provider，就执行 bloomAdd()：

// packages/core/src/render3/di.ts（第 152-181 行）
function bloomAdd(injectorIndex: number, tView: TView, type: ProviderToken<any>): void {
  // 每个 token 有一个全局唯一 ID（存在 NG_ELEMENT_ID 属性上）
  let id = (type as any)[NG_ELEMENT_ID];
  if (id == null) id = (type as any)[NG_ELEMENT_ID] = nextNgElementId++;

  const bloomHash = id & 0xFF;  // 取低 8 位，得到 0-255 的 hash 值（实际代码使用 `BLOOM_MASK` 常量，此处展示等价逻辑）
  // 找到对应的 32 位桶（共 8 个桶，每桶 32 位）
  // 在桶中设置对应的位
  (tView.data as number[])[injectorIndex + (bloomHash >> 5)] |= (1 << (bloomHash & 31));
}

查询时，bloomHasToken() 计算同样的 hash，检查对应位是否为 1。如果为 0，当前节点绝对没有这个 token，直接跳到父节点——省去了线性扫描指令列表的开销。

Bloom Filter 存在误判（不同 token 可能产生相同的 hash），但不存在漏判。误判只会导致多做一次无谓的线性查找，不会影响正确性。在 Angular 的实际使用中，每个组件注册的 provider 数量通常很少，这个优化在大型应用的深层组件树中效果显著。

💡 设计考量：Bloom Filter 的引入是一个典型的"以空间换时间"优化。每个 NodeInjector 占用 TView.data 中 8 个额外的整数槽（32 字节），换来的是在深层组件树中跳过无关节点的能力。在大型企业应用中，组件树深度可以超过 20 层，Bloom Filter 可以显著减少线性查找的次数。

🔗 NodeInjector 存储在 LView 中的详细布局（特别是 TNODE 和 PARENT 槽的精确位置），将在第5章分析 LView 数据结构时深入展开。

providedIn 与摇树优化

理解了 DI 的查找机制后，我们来解答一个工程上的核心问题：为什么 providedIn: 'root' 可以被摇树优化（tree-shaking）掉，而 providers: [MyService] 不行？

ɵprov：作用域标记字段

当你写下 @Injectable({ providedIn: 'root' }) 时，Angular 编译器会在类上生成一个静态字段 ɵprov（正式名称是 NG_PROV_DEF）：

// 你写的代码
@Injectable({ providedIn: 'root' })
export class UserService { ... }

// 编译器生成的静态字段（简化）
UserService.ɵprov = ɵɵdefineInjectable({
  token: UserService,
  providedIn: 'root',
  factory: () => new UserService(inject(HttpClient)),
});

这个 ɵprov 字段不需要被任何 providers 数组引用。它孤立地挂在类上，只有当 R3Injector 在查找 token 时，才会主动去读它：

// packages/core/src/di/r3_injector.ts（行号随版本变化，约第 420 行附近）
if (record === undefined) {
  // 自动发现：查询 token 上的 ɵprov
  const def = couldBeInjectableType(token) && getInjectableDef(token);
  if (def && this.injectableDefInScope(def)) {
    // 只有作用域匹配，才动态注册这个 provider
    record = makeRecord(injectableDefOrInjectorDefFactory(token), NOT_YET);
    this.records.set(token, record);
  }
}

摇树优化器（Webpack、Rollup、esbuild）的工作是：追踪"被引用的代码"。providedIn: 'root' 的服务，其 ɵprov 字段没有被任何数组持有——如果 UserService 本身从未在任何模板或代码中被用到，整个服务类（含 ɵprov）都可以被删除。

providers 数组为什么不能摇树

// 方案 A：可以摇树
@Injectable({ providedIn: 'root' })
class AnalyticsService {}

// 方案 B：不能摇树
@NgModule({
  providers: [AnalyticsService]  // ← 这里有一个显式引用
})
class AppModule {}

在方案 B 中，AppModule 的 ɵinj（注入器定义）必须包含 AnalyticsService 的引用。AppModule.ɵinj 被 bootstrap 流程持有，因此整条引用链无法断开——即使 AnalyticsService 在应用中从未被 inject() 调用，它也会被打包进最终产物。

flowchart LR A["AnalyticsService\nɵprov"] B["AppModule\nɵinj.providers"] C["bootstrap()"] D["打包产物"] subgraph TREESHAKEABLE["providedIn: root（可摇树）"] A -- "若未被使用\n可删除" --> D end subgraph NOT_TREESHAKEABLE["providers 数组（不可摇树）"] B -- "被 bootstrap 引用" --> C --> D B -- "引用" --> A end

💡 standalone 组件的 providers: [] 和 bootstrapApplication() 的 providers: [] 也是同理：任何出现在 providers 数组中的 token，都会被打包工具认为是"被引用的代码"，无法删除。

providedIn 的作用域匹配

injectableDefInScope() 方法决定某个 EnvironmentInjector 是否应该接管一个 @Injectable 的创建：

// packages/core/src/di/r3_injector.ts（第 551-559 行）
private injectableDefInScope(def: ɵɵInjectableDeclaration<any>): boolean {
  if (!def.providedIn) return false;
  const providedIn = resolveForwardRef(def.providedIn);

  if (typeof providedIn === 'string') {
    // 'root' | 'platform' | 'any' | 'environment'
    return providedIn === 'any' || this.scopes.has(providedIn);
  } else {
    // NgModule 类型：检查此注入器是否关联该 NgModule
    return this.injectorDefTypes.has(providedIn);
  }
}

Root 注入器的 scopes Set 中包含 'root' 和 'environment'，Platform 注入器包含 'platform'。providedIn: 'any' 比较特殊，它匹配所有注入器——每个注入器都会为它创建一个独立的实例，适用于需要"每个懒加载模块各自一份"的场景。

小结

本章追踪了 Angular DI 系统的完整内部机制：

  • 两棵树：EnvironmentInjector 树（应用级）和 NodeInjector 树（组件树同构），查询时从组件节点出发，穿越 NodeInjector 链，最终委托给 EnvironmentInjector，到达 NullInjector 则抛错
  • inject() 的上下文：全局的 currentInjector 变量控制注入上下文；injectImplementation 变量决定走 NodeInjector 路由还是 EnvironmentInjector 路由
  • R3Injector 解析：四种 provider 形式（useClass/useFactory/useValue/useExisting）统一转换为工厂函数；hydrate() 实现惰性单例，用 NOT_YET/CIRCULAR 哨兵值检测循环依赖；multi: true provider 聚合为数组
  • NodeInjector 的本质：不是独立对象，而是 LView 中的特定槽位；Bloom Filter（256位，8个32位桶）提供快速存在性检查；PARENT 槽编码父节点位置，支持高效向上遍历
  • 摇树优化：providedIn: 'root' 将作用域信息隔离在 ɵprov 字段中，不被任何数组持有，构建工具可安全删除未引用的服务；providers 数组中的 token 总是被持有，无法删除

再看开场的两条错误，现在它们都有了精确的解释：

  • NG0203：inject() 在第一关发现 currentInjector 为 undefined——调用方不在任何注入上下文中（不是构造函数、生命周期钩子或工厂函数）
  • NullInjectorError：token 穿透了所有 NodeInjector 和 EnvironmentInjector，在第五关抵达 NullInjector——没有任何 provider 声明过这个 token

每一次 DI 报错，都是系统在精确地告诉你它卡在了五关旅程的哪个位置。
