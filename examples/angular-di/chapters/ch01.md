# 第一章：Angular 依赖注入系统源码解析

> **本章目标**：读完这一章，你不只是"知道 DI 怎么用"，而是能在脑子里放映出 `inject(MyService)` 从调用到返回实例的完整电影。

---

## 开篇引入：那个让你困惑了很久的问题

你在 Angular 项目里写过多少次这样的代码？

```typescript
@Injectable({ providedIn: 'root' })
export class UserService {
  private http = inject(HttpClient);
}
```

你知道它能工作，你也知道为什么这样配置。但有没有那么一个时刻，你突然想问：

*当 Angular 执行到 `inject(HttpClient)` 这一行的时候，它到底在做什么？*

这个问题听起来像是在钻牛角尖，但它其实是打开 Angular 架构大门的钥匙。一旦你真正理解了依赖注入系统的运作方式，以前那些莫名其妙的报错（"No provider found for..."）、那些让你困惑的配置（`providedIn: 'root'` 和 `providers: [SomeService]` 到底有什么区别？）都会突然变得通透。

这一章，我们就做一件事：跟着 Angular 源码（`packages/core/src/di/`），把 `inject(HttpClient)` 这个调用从头到尾走一遍。

我们会用"**城市政府**"来作为贯穿全章的比喻。这个比喻不是装饰，而是理解整个系统结构的脚手架。

---

## 宏观架构：一座城市的行政体系

想象一个国家有很多城市，每个城市有自己的政府，管辖自己城市里的事务。城市政府有上级——省政府；省政府有上级——国家政府。国家政府上面没有更高的了，如果国家政府也解决不了你的问题，你只能收到一封"无法受理"的回信。

Angular 的 DI 系统就是这样一个层级行政体系：

```
NullInjector（终点站：永远拒绝，永远报错）
  └── Platform Injector（平台政府，scopes: {'platform'}）
        └── Root Injector（国家政府，scopes: {'root', 'environment'}）
              └── 路由级 Feature Injector（省政府，可选）
                    └── ComponentA 的 NodeInjector（市政府）
                          └── ComponentB 的 NodeInjector（区政府）
```

Angular 实际上有**两棵**并行的注入器树：

**环境注入器树（Environment Injector Tree）**：由 `R3Injector` 实例组成，对应 Platform、Root、Route 等级别。它们在组件树之外，独立存在。

**节点注入器树（Node Injector Tree）**：每个组件都有一个对应的 NodeInjector，随着组件树的结构形成平行的注入器树。当一个 NodeInjector 找不到依赖时，它会一路向上，最终委托给环境注入器树。

这两棵树覆盖了 Angular 应用里几乎所有的注入需求，它们共享同一套"查户口"协议，但服务不同的区域。

核心文件一览：

| 源文件 | 类比 | 核心职责 |
|--------|------|----------|
| `primitives/di/src/injector.ts` | 行政系统底层规程 | 维护"当前受理窗口" `_currentInjector` |
| `di/injection_token.ts` | 居民证类型定义 | `InjectionToken<T>` 的实现 |
| `di/interface/defs.ts` | 市民登记表格式 | `ɵɵInjectableDeclaration` 接口和工厂函数 |
| `di/r3_injector.ts` | 政府大楼主体 | `R3Injector`，整个查找引擎 |
| `di/inject_switch.ts` | 前台分诊台 | 决定把请求交给哪个处理器 |
| `di/injector_compatibility.ts` | 市民热线 | 公开的 `inject()` / `ɵɵinject()` 函数 |
| `di/null_injector.ts` | 上访终点站 | `NullInjector`，永远返回"无法受理" |
| `di/contextual.ts` | 特别通行证 | `runInInjectionContext()` |

## Token 与 Provider：户口本和户籍登记

### Token 是什么：居民证号码

在这座城市里，每个"服务"都需要一个唯一的身份标识，才能被找到。这个标识就是 **Token**。

Token 有两种形式：

**第一种：直接用类名作为 Token**

```typescript
@Injectable({ providedIn: 'root' })
class UserService { }

inject(UserService);  // UserService 类本身就是 Token
```

**第二种：`InjectionToken<T>` 用于非类型值**

```typescript
const API_URL = new InjectionToken<string>('api-url', {
  providedIn: 'root',
  factory: () => 'https://api.example.com',
});

inject(API_URL);  // InjectionToken 实例作为 Token
```

让我们看看 `InjectionToken` 的构造器源码（`packages/core/src/di/injection_token.ts`）：

```typescript
constructor(
  protected _desc: string,           // 只用于调试显示，不影响 Token 身份
  options?: {
    providedIn?: Type<any> | 'root' | 'platform' | 'any' | null;
    factory: () => T;
  },
) {
  this.ɵprov = undefined;
  if (typeof options == 'number') {
    // 内部特殊用法：用负数设置 __NG_ELEMENT_ID__
    (this as any).__NG_ELEMENT_ID__ = options;
  } else if (options !== undefined) {
    this.ɵprov = ɵɵdefineInjectable({
      token: this,
      providedIn: options.providedIn || 'root',
      factory: options.factory,
    });
  }
}
```

这里有两个关键点：

**第一点**：Token 的身份是**对象引用**，不是描述字符串。两个 `new InjectionToken('same-name')` 是完全不同的 Token，就像两个同名的居民，只认证件号（内存地址），不认名字。

**第二点**：`ɵprov` 字段是 Token 的"户籍档案"。如果你传入了 `factory`，这个 Token 就是**自带工厂的 Token**——它知道自己怎么被创建。如果没有传 `factory`，`ɵprov` 是 `undefined`，这个 Token 就必须在某个注入器里被显式 provide。

### `ɵɵInjectableDeclaration`：那份户籍档案

当你写 `@Injectable({ providedIn: 'root' })` 时，Angular 编译器会在你的服务类上生成一个静态字段 `ɵprov`：

```typescript
// packages/core/src/di/interface/defs.ts
export interface ɵɵInjectableDeclaration<T> {
  providedIn: InjectorType<any> | 'root' | 'platform' | 'any' | 'environment' | null;
  token: unknown;           // 指向 Token 本身
  factory: (t?: Type<any>) => T;  // 工厂函数
  value: T | undefined;     // 单例缓存（用于 limp mode）
}
```

`value: undefined` 是一个懒加载缓存槽。在没有注入器的极端情况下（Angular 的 "limp mode"），工厂函数执行后的结果会被存在这里，以避免重复执行。

### Provider：户籍登记的六种方式

当你在 `providers` 数组里写东西时，你实际上在帮 Angular 登记"如何找到/创建某个依赖"。Angular 支持六种登记方式（`packages/core/src/di/interface/provider.ts`）：

```typescript
// 1. TypeProvider：直接登记类（最常见）
providers: [UserService]

// 2. ClassProvider：用另一个类替代
providers: [{ provide: UserService, useClass: MockUserService }]

// 3. ValueProvider：直接登记一个值
providers: [{ provide: API_URL, useValue: 'https://prod.api.com' }]

// 4. FactoryProvider：用工厂函数创建
providers: [{ provide: Logger, useFactory: () => new Logger('prefix') }]

// 5. ExistingProvider：别名——指向已有 Token
providers: [{ provide: OldService, useExisting: NewService }]

// 6. ConstructorProvider：指定构造依赖
providers: [{ provide: SomeClass, deps: [HttpClient, Router] }]
```

所有这六种，最终都会被 `providerToFactory()` 函数（`r3_injector.ts`）转换成一个统一格式：`() => 某个值`。

有意思的是 `useExisting` 的转换方式：

```typescript
// packages/core/src/di/r3_injector.ts
} else if (isExistingProvider(provider)) {
  factory = (_, flags) =>
    ɵɵinject(resolveForwardRef(provider.useExisting), flags);
}
```

它不是"把 NewService 的值复制给 OldService"，而是每次工厂被调用时，都通过 `ɵɵinject()` 实时去查 `NewService`。由于单例缓存机制，这个工厂只会被调用一次——但它是活的别名，而不是静态拷贝。

## 注入器的层级结构：政府大楼内部

### `R3Injector`：政府大楼的档案室

让我们走进最核心的类。`R3Injector` 是所有环境注入器的实现，它的内部结构非常清晰（`packages/core/src/di/r3_injector.ts`）：

```typescript
export class R3Injector extends EnvironmentInjector {
  // 核心：Token → Record 的 Map
  private records = new Map<ProviderToken<any>, Record<any> | null>();

  // 有 ngOnDestroy 的服务，注销时需要通知
  private _ngOnDestroyHooks = new Set<OnDestroy>();

  constructor(
    providers: Array<Provider | EnvironmentProviders>,
    readonly parent: Injector,   // ← 父级注入器引用
    readonly source: string | null,
    readonly scopes: Set<InjectorScope>,
  ) {
    super();
    // 把所有 providers 处理成 Record，但不立即创建实例
    forEachSingleProvider(providers, (provider) => this.processProvider(provider));

    // 自动注册：注入 INJECTOR token 返回自身
    this.records.set(INJECTOR, makeRecord(undefined, this));
  }
}
```

`records` 这个 Map 就是档案室的索引柜：键是 Token，值是一份 `Record`。

### `Record<T>`：每份档案的内容

```typescript
// packages/core/src/di/r3_injector.ts
interface Record<T> {
  factory: ((_: undefined, flags?: InternalInjectFlags) => T) | undefined;
  value: T | {};     // 单例实例，或者是哨兵值
  multi: any[] | undefined;  // multi: true 时的实例数组
}
```

每一份档案包含三样东西：工厂函数（如何创建）、当前值（已创建的单例）、以及可选的多值数组（`multi: true` 的场景）。

### 两个神奇的哨兵

```typescript
// packages/core/src/di/r3_injector.ts
const NOT_YET = {};    // 还没创建
const CIRCULAR = {};   // 正在创建中（用于检测循环依赖）
```

这两个普通的空对象 `{}` 承担着重要的逻辑意义。它们被用来标记 `record.value` 的三种状态：

```
record.value === NOT_YET    → 档案存在，但实例还未创建（懒加载）
record.value === CIRCULAR   → 工厂正在执行中（有人在创建这个服务）
record.value === 真实实例   → 单例已创建，直接返回
```

### 注入器层级示意图

```
                    ┌──────────────────────────┐
                    │       NullInjector        │
                    │  get() → 永远抛出错误      │
                    └────────────┬─────────────┘
                                 │ parent
                    ┌────────────▼─────────────┐
                    │    Platform R3Injector    │
                    │  scopes: {'platform'}      │
                    └────────────┬─────────────┘
                                 │ parent
                    ┌────────────▼─────────────┐
                    │      Root R3Injector      │
                    │  scopes: {'root','env'}   │
                    │  records: {               │
                    │    HttpClient → Record    │
                    │    UserService → Record   │
                    │    ...                    │
                    │  }                        │
                    └────────────┬─────────────┘
                                 │ parent
           ┌─────────────────────▼──────────────────────┐
           │          NodeInjector (ComponentA)          │

           └─────────────────────┬──────────────────────┘
```

每个 `R3Injector` 持有对 `parent` 的引用。查找时，如果当前层找不到，就向上委托——这正是"逐级上报"的行政逻辑。

## 依赖解析的调用链：一次查户口的全过程

现在是最精彩的部分。我们来一步步跟踪 `inject(UserService)` 的完整执行路径。

### 第一步：进入公开 API

你在代码里调用的 `inject()` 函数，最终会走到 `ɵɵinject()`（`packages/core/src/di/injector_compatibility.ts`）：

```typescript
export function ɵɵinject<T>(
  token: ProviderToken<T>,
  flags = InternalInjectFlags.Default,
): T | null {
  return (getInjectImplementation() || injectInjectorOnly)(
    resolveForwardRef(token),
    flags,
  );
}
```

这一行 `(getInjectImplementation() || injectInjectorOnly)` 是整个系统的**分诊台**。

- 如果 `_injectImplementation` 有值（比如在处理组件构造器时，Angular 内部会把它设为 `directiveInject`），就走组件注入器路径。
- 如果 `_injectImplementation` 是 `undefined`（普通环境注入上下文），就走 `injectInjectorOnly`。

### 第二步：确认受理窗口

`injectInjectorOnly()` 的第一件事是查当前的"受理窗口"：

```typescript
export function injectInjectorOnly<T>(token, flags = InternalInjectFlags.Default): T | null {
  const currentInjector = getCurrentInjector();   // 从全局变量取得当前注入器

  if (currentInjector === undefined) {
    // 根本不在注入上下文里，直接报错
    throw new RuntimeError(RuntimeErrorCode.MISSING_INJECTION_CONTEXT, ...);
  } else if (currentInjector === null) {
    // 在注入上下文里，但没有注入器——走 limp mode
    return injectRootLimpMode(token, undefined, flags);
  } else {
    const value = currentInjector.retrieve(token, options);
    // ...
    return value;
  }
}
```

`getCurrentInjector()` 返回的是 `primitives/di/src/injector.ts` 里那个全局变量 `_currentInjector`。这个变量在每次进入注入上下文时被设置，离开时被恢复——就像一个"当前服务大厅"的指示牌。

### 第三步：R3Injector.get() — 档案室查找

现在我们到了 `R3Injector.get()` 方法（`packages/core/src/di/r3_injector.ts`）。这是真正的查找逻辑：

```typescript
override get<T>(token, notFoundValue = THROW_IF_NOT_FOUND, options?: InjectOptions): T {
  assertNotDestroyed(this);

  // 特殊通道：内部 Angular token（ElementRef 等）有快速路径
  if (token.hasOwnProperty(NG_ENV_ID)) {
    return (token as any)[NG_ENV_ID](this);
  }

  // 设置上下文，保存并将在 finally 中恢复
  const previousInjector = setCurrentInjector(this);
  const previousInjectImplementation = setInjectImplementation(undefined);

  try {
    if (!(flags & InternalInjectFlags.SkipSelf)) {
      // 在本级档案室查找
      let record = this.records.get(token);

      if (record === undefined) {
        // 档案不存在，但这个 Token 可能是"tree-shakable"的
        const def = couldBeInjectableType(token) && getInjectableDef(token);
        if (def && this.injectableDefInScope(def)) {
          // Token 的 providedIn 属于本级作用域，动态登记
          record = makeRecord(injectableDefOrInjectorDefFactory(token), NOT_YET);
        } else {
          record = null;  // ← null 表示"已查，本级没有"（负缓存）
        }
        this.records.set(token, record);
      }

      if (record != null) {
        return this.hydrate(token, record, flags);  // 有档案，去取实例
      }
    }

    // 本级找不到，向上级委托
    const nextInjector = !(flags & InternalInjectFlags.Self) ? this.parent : getNullInjector();
    return nextInjector.get(token, notFoundValue);

  } catch (error: any) {
    // 错误路径：往错误对象里追加当前 Token，拼出完整的依赖路径
    if (/* CYCLIC or NOT_FOUND */) {
      if (ngDevMode) {
        prependTokenToDependencyPath(error, token);
        if (previousInjector) throw error;
        else throw augmentRuntimeError(error, this.source);
      }
    }
  } finally {
    setInjectImplementation(previousInjectImplementation);
    setCurrentInjector(previousInjector);
  }
}
```

注意 `records.set(token, null)` 这行：当确认本级没有某个 Token 之后，Angular 会把 `null`（而不是 `undefined`）存入缓存。下次同一 Token 来查时，`records.get(token)` 返回 `null`（而非 `undefined`），直接跳过 `getInjectableDef()` 的检查，提升性能。这是一个**负缓存优化**。

### 第四步：hydrate() — 取出实例（或者第一次创建它）

```typescript
// packages/core/src/di/r3_injector.ts
private hydrate<T>(token, record: Record<T>, flags): T {
  if (record.value === CIRCULAR) {
    // 发现循环依赖！
    throw cyclicDependencyError(stringify(token));
  } else if (record.value === NOT_YET) {
    // 首次访问，需要创建实例
    record.value = CIRCULAR;           // ① 先设为 CIRCULAR，作为循环检测守卫
    record.value = record.factory!();  // ② 调用工厂，创建真实实例
                                       //    factory 内部可能触发其他 inject() 调用
                                       //    如果它又 inject 了自己，就会在①处被捕获
  }
  // value 已经是真实实例，自动跳过上面两个分支
  if (hasOnDestroy(record.value)) {
    this._ngOnDestroyHooks.add(record.value);  // 注册销毁钩子
  }
  return record.value as T;
}
```

循环依赖检测的逻辑精妙而简洁：

1. 进入 `hydrate()` 时，把 `record.value` 设为 `CIRCULAR` 哨兵
2. 调用 `factory()` —— 工厂内部会 `inject()` 其他依赖
3. 如果某个依赖（直接或间接地）又 `inject()` 了当前 Token，就会再次进入 `hydrate()`，此时 `record.value === CIRCULAR`，立即抛出循环依赖错误
4. 如果工厂正常完成，`record.value` 被真实实例覆盖，`CIRCULAR` 消失

没有额外的 `Set<Token>` 来追踪"正在创建中的 Token"——只有 `record.value` 自身的值就完成了检测。

### 完整调用链总览

```
你的代码: inject(UserService)
  │
  ▼
ɵɵinject(UserService)              [injector_compatibility.ts]
  │  (getInjectImplementation() || injectInjectorOnly)(token)
  ▼
injectInjectorOnly(UserService)    [injector_compatibility.ts]
  │  getCurrentInjector() → Root R3Injector
  │  currentInjector.retrieve(UserService)
  ▼
R3Injector.retrieve()              [r3_injector.ts]
  │  (包装成 get() 调用)
  ▼
R3Injector.get(UserService)        [r3_injector.ts]
  │  records.get(UserService) → undefined（首次）
  │  getInjectableDef(UserService).providedIn → 'root'
  │  injectableDefInScope('root') → true（root 注入器有 'root' scope）
  │  record = makeRecord(factory, NOT_YET)
  │  records.set(UserService, record)  ← 缓存下来
  ▼
R3Injector.hydrate(UserService, record)
  │  record.value === NOT_YET → 进入创建流程
  │  record.value = CIRCULAR  ← 循环检测守卫
  │  record.value = factory()
  │    └── new UserService()  (内部可能触发更多 inject())
  │  record.value = UserService实例  ← 覆盖 CIRCULAR
  ▼
返回 UserService 实例
```

## 一个令人意外的设计：注入上下文是个"全局指针"

在学习这部分源码时，最让我意外的设计是这个：

Angular 的"注入上下文"本质上是一个**全局变量**（`_currentInjector`），被一个"保存-设置-恢复"的模式管理，形成了一个隐式的调用栈。

```typescript
// packages/core/primitives/di/src/injector.ts
let _currentInjector: Injector | undefined | null = undefined;
```

当你调用 `R3Injector.get()` 时，第一件事是：

```typescript
const previousInjector = setCurrentInjector(this);  // 设置当前注入器为 this
// ... 执行注入逻辑 ...
finally {
  setCurrentInjector(previousInjector);  // 恢复之前的注入器
}
```

`setCurrentInjector(this)` 返回**之前的值**并保存在 `previousInjector`。`finally` 块确保无论如何都恢复。这个模式在 Angular DI 代码里出现了至少 5-6 次（`get()`、`runInContext()`、`runInInjectionContext()`、`resolveInjectorInitializers()` 等）。

这实际上是在用全局变量模拟"注入上下文调用栈"。之所以这样设计，原因是 JavaScript 是单线程的——在一个同步调用栈里，不需要真正的线程本地存储，一个全局变量配合"保存-恢复"就足够了。

**这带来一个重要的实际含义**：`inject()` **只能在同步调用中使用**。一旦你 `await` 或者用了 setTimeout，调用栈被打断，`_currentInjector` 已经被恢复成之前的值，再调用 `inject()` 就会得到 `undefined`，从而抛出"not in injection context"错误。这不是一个偶然的限制——它是这个设计的直接推论。

顺带一提，还有一个类似的全局指针 `_injectImplementation`（在 `inject_switch.ts` 里），用来在"组件构造器注入"和"普通环境注入"两种模式之间切换。两个全局指针共同决定了 `inject()` 在任意时刻的行为。从这个角度看，Angular DI 是一个极其精妙的状态机。

## 本章小结：四个核心心智模型

读完本章，你需要建立四个新的心智模型：

**① Token 是地址，不是名字**  
`inject()` 靠对象引用（`===`）查找，不靠字符串。这解释了为什么相同描述的两个 `InjectionToken` 实例是不同的，也解释了为什么可以用类本身作为 Token。

**② 注入器就是带父指针的 Map**  
`R3Injector` 的核心是 `records: Map<Token, Record>`。查找时，先在自己的 Map 里查，找不到就委托 `parent.get()`，一路向上直到 `NullInjector` 抛错。

**③ 实例是懒加载的，工厂只执行一次**  
所有服务在注入器创建时只是被登记成 `NOT_YET` 的 `Record`，不会立即实例化。第一次被 `inject()` 时，工厂函数才执行，结果缓存在 `record.value` 里，后续调用直接返回缓存。

**④ 注入上下文是个"全局指针 + 保存恢复"的调用栈**  
`_currentInjector` 决定了 `inject()` 找谁要值。每进一个注入上下文，就保存旧值、设置新值；退出时恢复。这就是为什么 `inject()` 不能跨越异步边界使用——`await` 之后，上下文指针已经被恢复了。
