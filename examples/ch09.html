第9章：路由系统——URL 与组件树的映射

你点击了一个链接，URL 变了，页面的一部分更新了。这个过程看起来行云流水，但如果你打开 Angular Router 的源码，会发现这个看似简单的操作背后，是一台精密运转的状态机。

从用户点击 <a routerLink="/dashboard"> 到 DashboardComponent 出现在屏幕上，Angular Router 精确地执行了这些步骤：将 URL 字符串解析成内部树状结构，对路由配置数组做深度优先匹配，依序执行守卫和预加载解析器，最后通知 RouterOutlet 销毁旧组件、激活新组件。任意一步出错，导航就会取消，页面保持不动。

这台状态机的核心叫 NavigationTransition，它的驱动力是一条 RxJS Observable 管道：每一个导航阶段都是管道上的一个操作符，串联在一起，形成一条从 URL 变化到 DOM 更新的完整执行链。理解了这条管道，你就掌握了 Angular Router 的工作原理，也就能回答工程中那些令人困惑的问题：守卫为什么按这个顺序执行？懒加载到底在哪个时机触发 import()？ActivatedRoute 里的 params$ 何时才会推送新值？

本章将沿着一次导航请求的完整生命周期，把这台状态机的每一个齿轮展示给你看。

⚠️ 版本说明：本章源码引用基于 Angular main 分支（约 v19.x）。函数签名和行号可能随版本演进，建议以函数名在当前版本中搜索定位。

Router 架构总览

在深入状态机之前，我们先建立宏观认知：Angular Router 究竟由哪些部分组成，各自承担什么职责？

Router 的职责可以分为三层：

  1. URL 层：将浏览器地址栏的字符串解析成内部数据结构（UrlTree），以及反向序列化；处理浏览器历史记录（History API）和哈希路由。
  2. 匹配与状态层：将 UrlTree 与路由配置数组匹配，生成一棵 RouterStateSnapshot（组件树的快照）；管理当前激活状态（RouterState）。
  3. 激活层：将新的 RouterState 应用到视图——通知 RouterOutlet 销毁旧组件、创建新组件，更新 ActivatedRoute 中的 Observable 值。

核心流程是：Router 接收导航请求，将其交给 NavigationTransitions 处理；NavigationTransitions 内部维护一个 BehaviorSubject<NavigationTransition>，每次新导航就向其推送一个新的 transition 对象，外层 switchMap 自动取消前一次未完成的导航——这是 Angular Router 防止导航竞态的核心机制。

🔗 NavigationTransitions 依赖 RouterConfigLoader 实现懒加载，而 RouterConfigLoader 内部使用 EnvironmentInjector 来创建懒加载模块的独立注入器树——这和第3章分析的 R3Injector 是同一套机制。

导航状态机

这一节是本章的核心。 一次导航从触发到完成，要经历七个阶段，每个阶段都对应 Observable 管道中的一个或多个操作符。

让我们沿着这条管道，逐段拆解每个阶段做了什么。

管道的骨架：switchMap 与导航取消

导航管道位于 navigation_transition.ts 的 setupNavigations() 方法中。整条管道以 transitions$（一个 BehaviorSubject）为源头，关键结构如下：

// packages/router/src/navigation_transition.ts（setupNavigations 核心结构）
this.transitions$.pipe(
  filter(t => t !== null),
  // 外层 switchMap：新导航到来时，自动取消（unsubscribe）旧管道
  switchMap(overallTransitionState => {
    return combineLatest([...]).pipe(
      switchMap(t => {
        // 内层主管道：recognize → checkGuards → resolveData → activate
        return of(t).pipe(
          mergeMap(t => recognize(t)),    // [1] URL 识别
          mergeMap(t => checkGuards(t)),  // [2] 守卫检查
          mergeMap(t => resolveData(t)),  // [3] 数据解析
          // ... [4] 懒加载组件
          // ... [5] 激活路由
        );
      }),
    );
  }),
).subscribe();

外层 switchMap 的语义至关重要：当用户在第一次导航还未完成时触发第二次导航，switchMap 会取消对第一条管道的订阅，RxJS 自动调用其 unsubscribe，所有未完成的操作符都会停止。这就是"新导航自动取消旧导航"的底层保障。

💡 提示：switchMap 能取消 Observable 订阅，但无法取消已发出的 Promise（如 fetch 请求）。如果你的 resolver 内部使用 async/await 而非 Observable，即使导航被取消，HTTP 请求仍会继续（结果只是被忽略）。这是一个值得注意的实践细节。

每个阶段都会通过 events$ 发出对应的路由事件，这是 Router 对外通信的主要方式：

从 GuardsCheck 阶段开始，每一步失败都会让导航进入 Cancelled 状态并发出 NavigationCancel 事件；守卫返回一个 UrlTree 时，Angular 会在取消当前导航的同时，立即发起一次新的跳转导航。

URL 识别与路由匹配

UrlTree：URL 的内部表示

Angular Router 并不直接操作 URL 字符串，而是先将其解析成一棵树状数据结构——UrlTree：

// packages/router/src/url_tree.ts（核心数据结构）
class UrlTree {
  constructor(
    public root: UrlSegmentGroup,          // 根分组
    public queryParams: Params = {},       // ?key=value 查询参数
    public fragment: string | null = null  // #fragment 片段
  ) {}
}

class UrlSegmentGroup {
  constructor(
    public segments: UrlSegment[],                      // 路径段数组
    public children: {[outlet: string]: UrlSegmentGroup} // outlet名 → 子分组
  ) {}
}

class UrlSegment {
  constructor(
    public path: string,                          // 如 'team'、'33'
    public parameters: {[name: string]: string}   // 矩阵参数 ;id=33
  ) {}
}

以 URL /team/33/(user/victor//support:help)?debug=true 为例，其对应的 UrlTree 树形结构是：

flowchart TD ROOT["UrlTree<br/>queryParams: {debug:'true'}"] PRI["primary: UrlSegmentGroup<br/>segments: [team, 33]"] PRI2["primary: UrlSegmentGroup<br/>segments: [user, victor]"] SUP["support: UrlSegmentGroup<br/>segments: [help]"] ROOT --> PRI PRI --> PRI2 PRI --> SUP

children 对象的 key 是 outlet 名称——primary 是默认主 outlet，support 是命名 outlet。这棵树天然地反映了 Angular 路由的多 outlet 结构。

路由匹配算法：深度优先

路由匹配由 Recognizer 类负责（recognize.ts），核心是对路由配置数组的深度优先遍历：

// packages/router/src/recognize.ts（processSegmentAgainstRoute 精简）
// 遍历路由配置数组，对每个 route 执行：
// 1. outlet 不匹配 → 跳过
// 2. 有 redirectTo → 处理重定向（最多执行 31 次，防止循环）
// 3. 执行 matchWithChecks(segmentGroup, route, segments)
//    ① match()：基础路径匹配（正则 / 通配符）
//    ② runCanMatchGuards()：执行 canMatch 守卫
// 4. 匹配成功 → getChildConfig() 获取子路由配置
//    如果有 loadChildren → 触发懒加载（见第五节）
// 5. 递归处理子路由段

匹配规则取决于 pathMatch 配置：

  • pathMatch: 'prefix'（默认）：当前路由只需匹配 URL 的前缀部分，剩余段继续匹配子路由。适合有子路由的路径配置。
  • pathMatch: 'full'：URL 必须被完全消耗（无剩余段）。通常用于空路径重定向：{ path: '', redirectTo: '/home', pathMatch: 'full' }。如果不加 pathMatch: 'full'，这条规则会匹配所有 URL 的前缀（因为所有 URL 都以空字符串开头）。

通配符路由 { path: '**', component: NotFoundComponent } 匹配所有剩余路径段，因此必须放在路由数组的最后——路由匹配是按数组顺序先定义先匹配的。

⚠️ 注意：canMatch 守卫在路由匹配阶段（recognize）就已执行，早于 canActivate。它的用途是决定"这条路由配置是否参与匹配"——可以让同一 URL 根据条件匹配到不同的路由配置，而不仅仅是允许/拒绝访问。这是 canMatch 与 canActivate 最本质的区别。

守卫执行顺序

守卫是 Angular Router 中最容易混淆的部分，主要因为几种守卫的执行顺序和方向并不相同。我们用"列车调度中心"的比喻来理解：每次导航是一趟列车，NavigationTransition 是调度中心，守卫就是一道道关卡——检票员（CanActivate）、离站确认员（CanDeactivate）、乘客区域检查员（CanActivateChild）各司其职，任一关卡拒绝，列车就返回原站。

完整执行顺序

flowchart TD A["触发导航"] --> B subgraph RECOG["recognize 阶段"] B["① canMatch<br/>（路由匹配时，逐条路由执行）"] end subgraph GUARDS["checkGuards 阶段"] C["② CanDeactivate<br/>（当前激活组件，并发执行）"] D["③ CanActivateChild<br/>（父路由 → 子路由方向）"] E["④ CanActivate<br/>（由深到浅，每个路由检查自身）"] C --> D D --> E end subgraph RESOLVE["resolveData 阶段"] F["⑤ Resolve<br/>（路由级别顺序执行，<br/>单路由内并发执行）"] end B --> C E --> F F --> G["组件激活"] style D fill:#ffeaa7 style E fill:#fdcb6e

⚠️ 关键细节：CanActivateChild 与 CanActivate 的方向相反

底层机制源于两个相互配合的设计：① checks 数组按"从深到浅"（最内层路由在前）排列；② runCanActivateChild 内部对 check.path（从根到此路由的完整路径数组）调用 .reverse()，使 CanActivateChild 从外（根）到内执行。两者叠加，产生了整体上 CanActivateChild 从外到内、CanActivate 从内到外的相反方向。

源码中的执行逻辑

// packages/router/src/operators/check_guards.ts
// CanActivate 的执行：concatMap 保证路由间顺序执行
function runCanActivateChecks(futureSnapshot, checks, forwardEvent?) {
  return from(checks).pipe(
    concatMap((check: CanActivate) => {
      return concat(
        // ① 发出激活事件（用于路由动画）
        fireChildActivationStart(check.route.parent, forwardEvent),
        fireActivationStart(check.route, forwardEvent),
        // ② CanActivateChild：遍历 path（从根到此路由），path.reverse() 使方向从外到内
        runCanActivateChild(futureSnapshot, check.path),
        // ③ CanActivate：检查当前路由本身
        runCanActivate(futureSnapshot, check.route),
      );
    }),
    first((result) => result !== true, true), // 第一个返回非 true 值（false 或 UrlTree）时立即中止后续守卫
  );
}

// CanDeactivate 的执行：mergeMap 并发执行所有待注销路由的守卫
function runCanDeactivateChecks(checks, futureRSS, currRSS) {
  return from(checks).pipe(
    mergeMap((check) => runCanDeactivate(...)),  // 并发！
    first((result) => result !== true, true),    // 一旦某守卫返回 false 或 UrlTree，整个链立即中止
  );
}

checks 数组中的路由按从内（最深子路由）到外（父路由）的顺序排列。这样对每个 check，runCanActivateChild 在 check.path（从根到此路由的完整路径）上从外到内执行各父路由的 canActivateChild，而 runCanActivate 只检查 check.route 本身。由于 checks 是从深到浅排列的，CanActivate 整体上先检查内层路由、后检查外层路由，与 CanActivateChild 的方向相反。

执行轨迹示例

以导航到 /admin/users 为例，路由配置如下，checks 数组从深到浅排列为 [users 节点, admin 节点]：

// 路由配置
const routes = [
  {
    path: 'admin',
    canActivate: [adminGuard],
    canActivateChild: [membershipGuard],
    children: [{ path: 'users', canActivate: [usersGuard] }]
  }
];

// 守卫执行轨迹（checks = [users路由, admin路由]）：
//
// check[0]：users 路由（深）
//   ① membershipGuard：admin 的 CanActivateChild
//      （path=[admin,users]，slice 后=[admin]，reverse 后=[admin]，从外到内）
//   ② usersGuard：users 路由的 CanActivate（自身）
//
// check[1]：admin 路由（浅）
//   ③ adminGuard：admin 路由的 CanActivate（自身）
//
// 最终顺序：membershipGuard → usersGuard → adminGuard

守卫返回值类型

所有守卫（无论是函数式还是类式）的返回值都通过 wrapIntoObservable() 统一处理，支持：

// packages/router/src/operators/check_guards.ts
// 守卫可以返回以下任意类型，框架统一转为 Observable：
type GuardResult =
  | boolean              // 同步允许/拒绝
  | UrlTree              // 重定向到另一个 URL
  | Observable<boolean | UrlTree>
  | Promise<boolean | UrlTree>
  | Signal<boolean>;     // Angular v19+

返回 UrlTree 是守卫实现重定向的标准方式：Router 会取消当前导航，并立即发起一次新的 navigateByUrl(urlTree) 调用。

函数式守卫（推荐）

Angular v14+ 引入的函数式守卫已成为推荐写法，相比类守卫代码量更少、无需注册为 provider，也更易于摇树优化（tree-shaking）：

// 现代写法：函数式守卫（推荐）
export const authGuard: CanActivateFn = (route, state) => {
  const authService = inject(AuthService); // inject() 可在守卫函数中使用
  return authService.isLoggedIn()
    ? true
    : inject(Router).createUrlTree(['/login']);
};

// 路由配置
const routes: Routes = [
  { path: 'admin', component: AdminComponent, canActivate: [authGuard] }
];

函数式守卫在路由守卫的注入上下文中执行，可以直接调用 inject()，无需类和构造函数注入。

懒加载实现

懒加载（lazy loading）是现代前端应用的标配优化手段，Angular Router 通过动态 import() 实现代码分割。关键是理解两种懒加载的触发时机不同。

loadChildren：在 recognize 阶段加载

loadChildren 用于懒加载路由模块或路由数组，在 URL 识别阶段（recognize）触发，早于任何守卫：

// packages/router/src/recognize.ts（getChildConfig 精简）
if (route.loadChildren) {
  if (route._loadedRoutes !== undefined) {
    // 已缓存：直接返回（每条路由只加载一次，后续导航复用缓存不再触发 import()）
    return {routes: route._loadedRoutes, injector: route._loadedInjector};
  }
  // ① 先执行 canLoad 守卫（已废弃，建议改用 canMatch）
  const shouldLoad = await runCanLoadGuards(injector, route, segments, ...);
  if (shouldLoad) {
    // ② 调用 RouterConfigLoader.loadChildren() → 触发 import()
    const cfg = await this.configLoader.loadChildren(injector, route);
    route._loadedRoutes = cfg.routes;     // 缓存路由配置
    route._loadedInjector = cfg.injector; // 缓存注入器（NgModule 场景）
    return cfg;
  }
}

对于 NgModule 式懒加载，RouterConfigLoader 会为懒加载模块创建独立的 EnvironmentInjector，挂载到应用注入器树上。这棵注入器子树会在路由销毁时一同销毁。

对于 standalone 路由数组式懒加载：

// standalone 路由懒加载（推荐，无独立注入器）
const routes: Routes = [
  {
    path: 'admin',
    loadChildren: () => import('./admin/routes').then(m => m.ADMIN_ROUTES)
  }
];

standalone 方式不创建独立注入器，共用父级 EnvironmentInjector，更轻量。

loadComponent：在守卫之后加载

loadComponent 用于懒加载单个 standalone 组件，触发时机在守卫和解析器都通过之后，在组件激活之前：

// packages/router/src/navigation_transition.ts（loadComponents 精简）
const loadComponents = (route: ActivatedRouteSnapshot): Array<Promise<void>> => {
  const loaders: Array<Promise<void>> = [];
  if (route.routeConfig?.loadComponent && !route.routeConfig._loadedComponent) {
    loaders.push(
      // 守卫通过后才触发 import()
      this.configLoader.loadComponent(injector, route.routeConfig)
        .then(comp => { route.component = comp; })
    );
  }
  for (const child of route.children) {
    loaders.push(...loadComponents(child)); // 递归处理嵌套懒加载组件
  }
  return loaders;
};

两种懒加载时机的对比：

对比项loadChildrenloadComponent
触发阶段recognize（守卫之前）loadComponents（守卫之后）
加载内容路由配置 + 可选注入器组件类
独立注入器NgModule 场景有（standalone 无）无
加载时机URL 匹配即开始下载守卫通过后才下载

⚠️ 鉴权场景的时机选择

两种方案在需要鉴权的路由上有重要的实践差异：

  • loadChildren + canMatch：canMatch 在 recognize 阶段执行，与 loadChildren 的 import() 同阶段——若使用 canActivate 而非 canMatch，未登录用户访问时路由配置已开始下载，造成带宽浪费；改用 canMatch 可在守卫拒绝时跳过整条路由，但一旦 canMatch 通过，下载随即触发。
  • loadComponent + canActivate：canActivate 守卫在守卫阶段执行，loadComponent 的 import() 在守卫全部通过之后才触发——守卫拒绝时不会发起任何网络请求，是鉴权场景下更高效的选择。

预加载策略

预加载（preloading）允许在用户空闲时提前加载懒加载路由，提升导航速度：

// 应用配置
bootstrapApplication(AppComponent, {
  providers: [
    provideRouter(routes,
      withPreloading(PreloadAllModules) // 空闲时预加载所有懒加载路由
    )
  ]
});

内置策略有 NoPreloading（默认，不预加载）和 PreloadAllModules（全量预加载）。自定义预加载策略实现 PreloadingStrategy 接口即可，常见做法是通过路由的 data 字段打标签，只预加载标记了 preload: true 的路由。

RouterOutlet 与组件激活

RouterOutlet 是 Router 与 DOM 之间的桥梁——它是一个指令（directive），负责在当前位置动态创建和销毁路由匹配的组件。

注册与激活

RouterOutlet 在 ngOnInit 时向 ChildrenOutletContexts 服务注册自己：

// packages/router/src/directives/router_outlet.ts（ngOnInit 精简）
ngOnInit(): void {
  // ① 向父级 outlet 上下文注册自己（让 ActivateRoutes 能找到它）
  this.parentContexts.onChildOutletCreated(this.name, this);

  // ② 如果此时路由已经激活（RouterOutlet 在 *ngIf 后显示的场景），
  //    补执行激活（"懒激活"逻辑）
  const context = this.parentContexts.getContext(this.name);
  if (context?.route) {
    this.activateWith(context.route, context.injector);
  }
}

当导航完成、ActivateRoutes.activate() 被调用时，它会找到注册的 RouterOutlet，调用 activateWith() 动态创建组件：

// packages/router/src/directives/router_outlet.ts（activateWith 精简）
activateWith(activatedRoute: ActivatedRoute, environmentInjector: EnvironmentInjector): void {
  const snapshot = activatedRoute.snapshot;
  const component = snapshot.component!;

  // 创建局部注入器，提供 ActivatedRoute 和子 outlet 上下文
  const injector = new OutletInjector(activatedRoute, childContexts, location.injector);

  // 动态创建路由组件
  this.activated = this.location.createComponent(component, {
    injector,
    environmentInjector,
  });
  this.changeDetector.markForCheck();
  this.activateEvents.emit(this.activated.instance); // (activate) 事件
}

OutletInjector 是一个自定义注入器，让路由组件内部可以通过 inject(ActivatedRoute) 获取当前路由数据。

组件销毁与 ngOnDestroy

路由切换时，RouterOutlet 先调用 deactivate() 销毁旧组件，再调用 activateWith() 创建新组件：

// packages/router/src/directives/router_outlet.ts（deactivate 精简）
deactivate(): void {
  if (this.activated) {
    const c = this.activated;
    // 销毁组件，此时旧组件的 ngOnDestroy 被触发
    c.destroy();
    this.activated = null;
    this.deactivateEvents.emit(c.instance); // (deactivate) 事件
  }
}

🔗 deactivate() 调用 ComponentRef.destroy() 时，Angular 按第8章分析的生命周期顺序销毁组件树——先子组件、后父组件（LView 逆序遍历），ngOnDestroy 依次被调用。NgModule 懒加载路由的 EnvironmentInjector，也在对应 RouterOutlet 完成 deactivate 后随之销毁。

命名 RouterOutlet

Angular 支持在同一页面使用多个命名 outlet，实现辅助路由（secondary routes）：


<router-outlet></router-outlet>


<router-outlet name="sidebar"></router-outlet>
// 路由配置
const routes: Routes = [
  { path: 'chat', component: ChatComponent, outlet: 'sidebar' }
];

URL 会编码为 /(sidebar:chat) 形式，UrlTree 中 root.children['sidebar'] 就是侧边栏的段组。命名 outlet 的 name 属性是静态的，不支持动态绑定——这是源码中明确的设计限制。

🔍 深入：ChildrenOutletContexts 是一棵树形结构，每个 RouterOutlet 在 activateWith 时，通过 OutletInjector 为子组件提供独立的 ChildrenOutletContexts。嵌套路由的层级完全靠这个机制隔离，父组件内的 <router-outlet> 操作的是父级的 ChildrenOutletContexts，不会干扰根级的 outlets。

ActivatedRoute——路由数据的容器

每个被路由激活的组件，都能通过 inject(ActivatedRoute) 获取当前路由的所有信息。

Observable 字段

ActivatedRoute 的核心字段都是 Observable（底层是 BehaviorSubject），这意味着它们会在路由参数变化时自动推送新值：

// packages/router/src/router_state.ts（ActivatedRoute 核心字段）
export class ActivatedRoute {
  url: Observable<UrlSegment[]>;          // 路由的 URL 段
  params: Observable<Params>;            // 路由参数（:id 等）
  queryParams: Observable<Params>;       // 查询参数（?key=value）
  fragment: Observable<string | null>;   // 片段（#hash）
  data: Observable<Data>;                // resolve 数据 + 路由静态 data
  paramMap: Observable<ParamMap>;        // params 的 Map 视图
  queryParamMap: Observable<ParamMap>;   // queryParams 的 Map 视图

  // 当前时刻的快照（同步，不随时间变化）
  snapshot: ActivatedRouteSnapshot;
}

这些 Observable 在 advanceActivatedRoute() 中被更新。Angular 做了一个 shallowEqual 优化：只有当新值与旧值不相等时，才调用 BehaviorSubject.next() 推送——避免同路由参数未变化时触发不必要的订阅回调。

何时用 Observable，何时用 snapshot

场景推荐方式
组件在同一路由下参数变化时需要重新加载数据订阅 params$ 或 paramMap
只在组件初始化时读一次路由参数snapshot.params
在 resolver 中读取路由参数snapshot.params（resolver 每次导航都重新执行）
在守卫中做路由条件判断直接使用 route 参数（ActivatedRouteSnapshot）

一个常见的陷阱是在 ngOnInit 中用 snapshot.params.id 读取参数，然后发现当用户点击从 /user/1 导航到 /user/2 时，组件没有更新——因为 Angular 复用了同一个组件实例，不会再次调用 ngOnInit，而 snapshot 也不会自动更新。订阅 activatedRoute.params 才是处理这种"参数变化复用"场景的正确方式。

路由数据继承

子路由默认不继承父路由的 params 和 data，触发继承的条件有三种：① 开启全局配置 paramsInheritanceStrategy: 'always'；② 子路由路径为空路径（path: ''）；③ 父路由无组件且无 loadComponent（纯容器路由，router_state.ts L263-269）。可以通过 withRouterConfig({ paramsInheritanceStrategy: 'always' }) 开启全量继承。

这是一个常见 bug 来源——在嵌套路由中，snapshot.params 可能读不到父路由的 :id，需要向上遍历 snapshot.parent，或确认父路由是否满足上述继承条件之一。

设计思考

设计思考：为什么用 Observable 管道而非 Promise 链？

导航状态机最核心的需求是"新导航取消旧导航"。用 Promise 链实现这一点相当繁琐——需要手动维护一个"当前导航 ID"，在每一步之前检查 ID 是否仍然有效。而 RxJS 的 switchMap 语义天然支持这个模式：切换到新 Observable 时，旧 Observable 被自动取消订阅。每一个导航阶段都是一个操作符，"导航被取消"等价于 Observable 被 unsubscribe——这是声明式地描述状态机行为，比命令式的 if/else 更清晰，也更难出错。

设计思考：守卫顺序设计的逻辑

守卫顺序遵循"先确认能否离开，再确认能否进入"的原则，这和现实世界的逻辑一致：你不会在没确认旧房间门能否打开之前就去开新房间的门。CanDeactivate 先于 CanActivate 执行，确保了"离开"操作得到确认后才开始"进入"的检查流程。

CanActivateChild 和 CanActivate 方向相反的设计，源于两者的语义不同：CanActivateChild 问的是"我（父路由）是否允许我的子路由激活？"，父路由有权优先否决；CanActivate 问的是"我（子路由）自己是否可以激活？"，子路由的自检可以在父路由的许可确认后再进行。

设计思考：函数式守卫的演进

Angular v14 引入函数式守卫，背后是整个框架向"函数 + inject()"范式迁移的大趋势（与 standalone 组件同步推进）。类守卫需要声明一个类、实现接口、注册为 provider，引入了不必要的间接层。函数式守卫直接使用 inject() 获取依赖，配合 TypeScript 的类型系统，代码量更少，也更易于摇树优化（tree-shaking）。

小结

一次导航请求的完整旅程：

  • URL 解析：DefaultUrlSerializer（内置 URL 序列化器，负责 URL 字符串 ↔ UrlTree 的双向转换）将字符串解析为 UrlTree（UrlSegment → UrlSegmentGroup → UrlTree）
  • 路由识别：Recognizer 对路由配置做深度优先匹配，canMatch 守卫在此阶段执行，loadChildren 在此阶段触发懒加载
  • 守卫检查：顺序为 CanDeactivate → CanActivateChild（父→子）→ CanActivate（由深到浅，每个路由检查自身），任一返回 false 即取消导航
  • 数据解析：Resolve 在守卫全部通过后执行，路由间顺序、路由内并发
  • 组件激活：RouterOutlet.activateWith() 动态创建组件，advanceActivatedRoute() 更新 ActivatedRoute 的 Observable
  • 状态机骨架：外层 switchMap 保证新导航自动取消旧导航，整条链路是 Observable 管道

ActivatedRoute 的 Observable 字段在路由参数变化时自动推送新值；snapshot 适合单次读取，但在同路由参数变化场景下需要改用 Observable 订阅。

🔗 路由懒加载为每个 NgModule 路由创建独立的 EnvironmentInjector，构成第3章分析的注入器树的一个子节点。理解注入器树对调试"懒加载模块内的服务为什么是新实例"至关重要。

💡 前向引用：路由识别和组件激活在服务端渲染（SSR）场景下需要同步完成——第10章将分析 platform-server 如何处理导航、懒加载策略如何影响首屏性能，以及 TransferState 如何避免客户端重复执行 Resolve。

思考题

1. 守卫顺序实践题

有如下路由配置：

const routes = [
  {
    path: 'workspace',
    canActivate: [workspaceGuard],
    canActivateChild: [membershipGuard],
    children: [
      {
        path: 'project/:id',
        canActivate: [projectGuard],
        component: ProjectComponent
      }
    ]
  }
];

当用户从 /workspace/project/1 导航到 /workspace/project/2 时（假设 workspace 路由被复用，只有 project 路由需要重新激活），守卫执行的顺序是什么？workspaceGuard、membershipGuard、projectGuard 各执行几次？

2. loadChildren 与 loadComponent 的时机差异

假设有一个需要鉴权的懒加载路由，你有两种选择：

  • 方案 A：loadChildren: () => import('./admin.routes') + canMatch: [authGuard]
  • 方案 B：loadComponent: () => import('./admin.component') + canActivate: [authGuard]

从加载时机和用户体验的角度，两种方案有什么区别？哪种方案在未登录用户访问时，不会触发 import() 的网络请求？
