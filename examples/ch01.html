第1章：宏观架构——Angular 的世界观

当你在编辑器里敲下这几行代码，按下保存键：

@Component({
  selector: 'app-hello',
  template: `<h1>你好，{{ name }}</h1>`
})
export class HelloComponent {
  name = 'Angular';
}

你可能以为只是声明了一个简单的组件。实际上，这几行字符在构建阶段和运行时各触发了一段复杂的协作链：编译器把模板翻译成一串 JavaScript 函数调用；运行时引擎用这串调用创建真实的 DOM 节点；DI 系统在整个过程中静静等待，准备向任何需要依赖的地方递送服务实例；变更检测机制则持续监视着 name 的值，一旦它改变就精准地更新那个 <h1> 里的文本。

这背后牵涉的，不是一个 package，而是整整一个精心分层的 monorepo——Angular 的源码世界。

读完本章，你应该能够回答两个问题：当你写下 @Component(...) 时，哪些 package 在参与工作？Angular 为什么要设计成这种分层结构，而不是"一个大 angular.js"？

一个仓库，二十个包——monorepo 的全景图

Angular 的所有源码（包括框架核心、路由、表单、动画、SSR、语言服务……）都住在同一个 Git 仓库：github.com/angular/angular。这就是 monorepo（单体仓库）策略——多个独立发布的 npm 包统一在一个代码仓库里管理。

packages/ 目录是整个仓库的心脏。打开它，你会看到 20 多个子目录，每个对应一个独立的 npm 包。按职责划分，它们大致分为三层：

核心层：框架的基础设施

Packagenpm 包名职责
core@angular/core框架核心：render3 引擎、依赖注入（Dependency Injection，DI）、变更检测、组件生命周期
compiler@angular/compiler模板编译器：HTML 模板 → JavaScript 渲染指令序列
compiler-cli@angular/compiler-cliAOT 编译 CLI 插件（ngtsc），嵌入 TypeScript 编译流水线

这三个包之间的关系，是理解 Angular 架构的第一把钥匙——后面的章节会不断回到这里。

💡 为什么 Angular 需要自有编译器：React 的 JSX 转换由 Babel/swc 这样的通用工具完成，不需要框架专属的编译器包。Angular 选择自有编译器，是因为模板中的 [property] 绑定、(event) 事件处理、DI 装饰器等语义超出了通用工具的理解范围。自有编译器的代价是更重的工具链，收益是类型安全的模板检查、精准的 IDE 语言服务，以及针对 Angular 语义的高效代码生成。

平台层：适配不同运行环境

Packagenpm 包名职责
platform-browser@angular/platform-browser浏览器平台：DOM 渲染器、bootstrapApplication 入口
platform-browser-dynamic@angular/platform-browser-dynamicJIT 路径的浏览器平台（开发调试场景）
platform-server@angular/platform-server服务端渲染（SSR）平台：Node.js 环境下运行 Angular
ssr@angular/ssrSSR 工具包：服务端渲染的高层封装

Angular 把"框架核心"与"运行平台"彻底分离。@angular/core 本身不知道自己运行在浏览器还是 Node.js——它只描述行为；平台包负责告诉它具体如何操作 DOM（或虚拟 DOM）。

功能层：开箱即用的特性

Packagenpm 包名职责
router@angular/router路由系统：URL 解析、路由守卫、懒加载
forms@angular/forms表单：响应式表单与模板驱动表单
common@angular/common通用 pipe 与 directive（NgIf、NgFor、AsyncPipe 等）
animations@angular/animations动画引擎
elements@angular/elementsAngular 组件 → Web Components
service-worker@angular/service-workerPWA Service Worker 支持
localize@angular/localizei18n 国际化

功能层的每个包都仅依赖 @angular/core，彼此之间几乎没有依赖关系。这种辐射状拓扑（而非链式依赖）确保了各功能的独立性——你可以只安装你需要的包，不用的包不会进入应用的包体积，摇树优化（tree-shaking）可以精准地将其移除。

除了这三层之外，还有一类特殊的"隐形包"：packages/core/primitives/ 下的子目录（signals/、di/、event-dispatch/ 等）。它们不独立发布为 npm 包，而是直接编译进 @angular/core 中。以后当我们深入 Signals 系统（第7章），就会在 packages/core/primitives/signals/ 里找到那些响应式原语的真身。

💡 关于 Bazel：Angular monorepo 内部使用 Google 的 Bazel 构建系统（而非 npm workspace）管理这二十多个包的构建、测试和依赖。每个目录下都有一个 BUILD.bazel 文件描述该目录的构建目标。对应用开发者来说，Bazel 完全透明——你只需要 npm install @angular/core，深入理解 Bazel 对阅读本书并非必要。

💡 monorepo 的工程代价：将所有 package 放在同一仓库，对 Angular 团队有显著的工程优势：一个 Pull Request 可以同时修改 @angular/core 和 @angular/router，不需要跨仓库协调版本；Bazel 通过文件内容哈希精确追踪变更，只重建受影响的包，CI 速度极快。代价也是真实的：克隆仓库需要下载整个 Angular 代码库，本地运行完整构建需要配置 Bazel 环境，门槛不低——但这不影响你阅读本书的任何章节。

编译时与运行时的分离

三个核心包中，最值得注意的是 compiler-cli、compiler、core 之间的分工。这个分工，是读懂后续所有章节的基础。

两条时间线

Angular 的工作横跨两个完全不同的时间点：

构建阶段（你在开发机上运行 ng build 时）：

  • @angular/compiler-cli（又名 ngtsc）作为 TypeScript 编译器插件被加载
  • 它扫描你的代码，找到所有 @Component、@Directive、@Injectable 装饰器
  • 调用 @angular/compiler 的模板编译 API，将每个组件的 HTML 模板转换为一段 JavaScript 渲染指令序列
  • 在每个类上添加静态字段 ɵcmp（ComponentDef，组件运行时定义）等
  • 最终输出的 .js 文件里，已经没有模板字符串，只有编译好的 JavaScript

运行阶段（用户打开浏览器时）：

  • 浏览器加载编译后的 .js 文件
  • @angular/core 中的 render3 引擎读取各组件的 ɵcmp 字段
  • 按照其中记录的模板函数（ComponentDef.template）执行渲染指令（instruction），创建 DOM
  • 依赖注入（Dependency Injection，DI）、变更检测、生命周期钩子在应用运行期间持续工作

这就是 AOT（Ahead-of-Time，预编译）的本质：把"解析模板"这件事从运行时提前到构建时。

运行时不需要 compiler

这里有一个关键设计原则：@angular/compiler 这个包在生产应用中不应该被打包进去。

AOT 编译后，所有模板翻译工作已在构建阶段完成。运行时的 @angular/core 只需读取已翻译好的 ComponentDef，完全不需要再调用 @angular/compiler 的任何 API。如果你发现生产包里包含了 @angular/compiler，那几乎一定是某处代码触发了 JIT 路径（运行时编译，Just-in-Time compilation），是一个需要修复的配置问题。

@angular/core 里确实有一个叫 compiler-facade 的轻量桥接层（packages/core/src/compiler/compiler_facade.ts），允许少量必须在运行时编译的场景（如 TestBed）动态引入 compiler，但它只是一个接口约定，不含 compiler 的实际实现。

这个设计让功能层的包（router、forms 等）在运行时只依赖 @angular/core，避免了无谓的包体积膨胀。

Ivy 革命——从全局编译到局部编译

理解了编译时与运行时的分离，下一个问题是：Angular v9 之前的 View Engine 引擎也有编译器，为什么还要推倒重来搞 Ivy？

View Engine 的症结

View Engine 时代（Angular v2—v8），模板编译产物高度依赖 NgModule 的全局命名空间。编译一个组件时，编译器需要知道"这个组件所在的 NgModule 里声明了哪些指令（directive）和 pipe 可供使用"。这导致了几个连锁问题：

  • 无法独立编译：编译组件 A 之前必须先知道它所在模块的完整依赖图，不能把一个组件单独送进编译器出来就能用
  • 无法摇树优化：引入某个 NgModule，它里面所有声明的组件、指令（directive）、pipe 都会被打包——即使你的模板只用了其中一个
  • 包体积居高不下：全局编译的特性使得未使用的代码路径难以被静态分析工具识别和移除

局部编译原则（Locality Principle）

Ivy 的根本改变是引入了局部编译原则（Locality Principle）：每个组件/指令（directive）/pipe 的编译产物必须是自给自足的，不依赖任何外部的全局状态。

编译一个组件，只需要看这个组件本身的源文件，加上它 imports 数组中直接列出的依赖——不需要知道任何 NgModule 上下文。编译产物中的 ɵcmp（ComponentDef）包含了运行时所需的一切信息。

这个原则的直接后果是：

  • 可独立编译：每个组件可以单独编译、单独缓存，增量重编译速度大幅提升
  • 精准摇树优化：组件的依赖关系被精确编码在 ɵcmp.dependencies 字段里，只有真正使用到的组件才会被打包
  • standalone 成为可能：没有全局命名空间的依赖，NgModule 就不再是必须的

这个设计是有意识的权衡。局部编译放弃了编译器的"全知视角"——某些需要跨组件信息的分析（如精确的未使用代码检测）无法在单组件编译时完成，留到打包工具（Webpack/esbuild）的链接（linking）阶段处理，这也是 Angular 构建仍然需要打包工具的原因之一。换来的是组件级别的独立编译缓存和精准摇树优化，对大型项目的增量构建速度和产物体积意义重大。

ComponentDef：ɵcmp 是什么

编译器为每个组件类生成的 ɵcmp 字段，其类型就是 ComponentDef<T>。这个对象贯穿本书所有章节，是理解 Angular 运行时的核心数据结构（第5章 LView 解剖时，我们会与它逐字段打交道）：

// packages/core/src/render3/interfaces/definition.ts（节选）
export interface ComponentDef<T> extends DirectiveDef<T> {
  readonly template: ComponentTemplate<T>;           // 编译后的模板函数（核心！）
  readonly dependencies: ReadonlyArray<...>;         // 直接依赖列表（局部编译的关键）
  readonly standalone: boolean;                      // 是否为 standalone 组件
  // ... 更多字段见下表 ...
}

字段类型说明
typeType<T>组件类引用（如 HelloComponent）
selectorsCssSelectorListCSS 选择器（如 [app-hello]）
declsnumber视图中节点数量（决定 LView 大小）

outputsRecord<string, string>@Output 映射
tView`TView \null`关联的静态视图元数据（框架运行时填充，在实例间共享）

ComponentDef.template 字段就是编译后的模板函数——一段 JavaScript 函数，通过调用一系列 ɵɵelementStart、ɵɵproperty、ɵɵtext 等渲染指令（instruction）来描述如何创建和更新视图。我们在第5章会逐字解剖这个函数。

ɵɵ 前缀的含义

你在源码中会看到大量 ɵɵ 前缀的函数。下面是编译器为 <h1>你好，{{ name }}</h1> 生成的代码示例：

// 编译器生成，非手写
// 对应模板：<h1>你好，{{ name }}</h1>
function HelloComponent_Template(rf: RenderFlags, ctx: HelloComponent) {
  if (rf & RenderFlags.Create) {    // 创建阶段：构建 DOM 结构
    ɵɵelementStart(0, 'h1');        // 创建 <h1> 节点，索引 0
    ɵɵtext(1);                      // 创建文本节点，索引 1
    ɵɵelementEnd();                 // 闭合 <h1>
  }
  if (rf & RenderFlags.Update) {   // 更新阶段：响应数据变化
    ɵɵadvance(1);                   // 移动指针到索引 1（文本节点）
    ɵɵtextInterpolate1('你好，', ctx.name, ''); // 更新文本内容
  }
}

Angular 核心文档对这些函数的定位说得很清楚：

"Ivy exports a number of functions prefixed with ɵɵ. These functions are part of API required for code generation by the Angular compiler. They are not meant for developer consumption."
——packages/core/src/render3/CODE_GEN_API.md

ɵɵ 双前缀是一个强信号：这是编译器生成代码专用的 API，框架可以随时修改其签名而不视为破坏性变更。单 ɵ 前缀则是框架内部 API，不属于公开稳定 API，但未必是编译器专用的。

Standalone 的演进

理解了局部编译原则，再回头看 standalone API 的出现就水到渠成了。

NgModule：历史的产物

在 Ivy 之前，NgModule 是 Angular 编译模型的必要组成部分——它为全局编译提供命名空间，告诉编译器"哪些组件/指令（directive）/pipe 可以在哪个模板里使用"。

Ivy 的局部编译原则让每个组件的依赖关系直接编码在自身的 ɵcmp.dependencies 字段里。NgModule 作为"编译上下文提供者"的角色，理论上已经可以退场了。但一套被广泛使用了多年的 API，退场需要时间。

从 NgModule 强制必须（v2—v13）到 standalone 开发者预览（v14，2022年），再到成为新项目默认模式（v17，2023年），直至 NgModule 进入维护模式（v19，2024年）——这条路走了整整八年，体现了 Angular 架构演进的一贯哲学：新特性尽可能复用现有基础设施，给生态充足的迁移时间。

standalone 的实现层面

standalone 在框架内部的实现出奇地简单——它只是 ComponentDef 上的一个布尔字段：

// packages/core/src/render3/def_getters.ts
export function isStandalone(type: Type<unknown>): boolean {
  // Try to get component def, directive def or pipe def
  const def = getComponentDef(type) || getDirectiveDef(type) || getPipeDef(type);
  // standalone is a simple boolean field on ComponentDef
  return def !== null && def.standalone;
}

当你写 @Component({ standalone: true })，编译器在生成 ɵcmp 时会把 standalone 字段设为 true；框架运行时通过 isStandalone() 检查这个字段，决定是否需要走 NgModule 的解析逻辑。

standalone 并不是一个全新的底层机制——它是编译器和运行时在现有 Ivy 架构上的一层判断分支。这也说明 Angular 的架构演进是渐进式的：新特性尽可能复用现有基础设施，而不是推倒重建。

⚠️ 版本注意：v19 之前的项目在迁移到 standalone 时，NgModule 可以与 standalone 组件混用。isStandalone() 是判断运行时行为的关键函数，在第3章（DI 作用域）中我们会再次看到它的使用。

全书路线图——走进这座剧院

我们已经勾勒出了 Angular 架构的轮廓。在进入每个模块的深度解析之前，让我们用一个统一的比喻来给全书建立共同的语言框架。

把 Angular 想象成一座剧院。

每当你的应用渲染一个页面，就是在上演一出戏。这座剧院里有几个不可或缺的角色：

  • 编译器（@angular/compiler）= 剧本作者：在演出开始之前（构建阶段），把声明式的剧本（HTML 模板）翻译成精确的舞台动作指令（渲染指令序列）。演出时观众永远看不到剧本，看到的只是演员按指令完成的动作。
  • render3 引擎（@angular/core 内部）= 台上的演员：拿到剧本作者提前写好的动作指令（ɵɵelementStart、ɵɵproperty……），在舞台上（DOM）忠实执行。每一帧渲染，都是在按动作指令演出。
  • 依赖注入（DI）= 幕后的道具组：负责在演出过程中按需向演员递送道具（服务实例）。演员不需要关心道具从哪里来、怎么制作，报出道具名（token）就会有人按需送来。
  • 变更检测 = 舞台监督：在每一幕演出结束后（每个事件处理完毕、每个异步操作完成后），舞台监督检查哪些道具被换过了（哪些数据绑定值变了），通知演员重新表演需要更新的那几个动作。
  • Zone.js = 计时广播系统：后台每当有人喊"好了！"（一个异步操作完成），广播系统自动通知舞台监督去做例行检查——这就是 Zone.js 触发变更检测的方式。Zoneless 模式则是关掉广播系统，由演员自己主动举手示意何时需要检查。

小结

回到本章开头的两个问题：

当你写下 @Component(...) 时，哪些 package 在参与工作？

两个阶段，两批参与者。构建阶段：compiler-cli（ngtsc）作为 TypeScript 编译器插件，调用 compiler 完成模板翻译，在每个组件类上挂载 ɵcmp（ComponentDef）字段。运行阶段：只有 @angular/core（render3 引擎 + DI + 变更检测）与对应的平台包（如 platform-browser）在工作——compiler 不在其中。这正是 AOT 编译的本质：把"理解模板"这件事从运行时提前到构建时。

Angular 为什么要设计成分层结构，而不是"一个大 angular.js"？

三个理由相互加强：① 独立性——核心层、平台层、功能层彼此解耦，更换运行平台（浏览器/SSR）不影响业务代码；② 精准摇树优化——局部编译原则（Locality Principle）使每个组件的依赖精确可知，未使用的代码可被安全移除；③ 演进灵活性——standalone API 在现有 Ivy 基础设施上平滑推进，NgModule 的退场走了八年却没有破坏任何现有应用。

本章涉及的核心数据结构 ComponentDef（ɵcmp）将在后续章节反复出现：第4章解析模板如何编译成它的 template 函数；第5章解剖它与 LView 的关系；第8章逐字段分析它对生命周期的影响。

📝 练习：在你的 Angular 应用中，打开浏览器 DevTools 控制台，输入 YourComponent.ɵcmp（将 YourComponent 替换为实际类名）。你将看到 ComponentDef 对象的全貌——对照本章介绍的字段，找找 template、dependencies、standalone 分别在哪里，值是什么。
