第7章：Signals——细粒度响应式的实现

第6章告诉了我们 Angular 如何知道需要更新——Zone.js 监听异步边界，ApplicationRef.tick() 触发全树脏检查，LViewFlags 决定哪些视图要被刷新。但"如何知道需要更新"的背后，是一套精密的响应式图。这一章，我们进入 Signals 的内部。

想象超市里的电子价签系统：每一块货架标签都与后台数据库实时同步，批发商调整了进价，所有显示零售价的价签会自动计算新的利润空间并更新显示——没有人需要逐一刷新，没有轮询，数据变了，价签自动更新。这就是 Angular Signals 的直觉模型。signal() 是后台数据库里的记录，computed() 是根据进价自动计算零售价的公式格，effect() 是当价格变化时自动触发的门店广播。

这个比喻还可以继续延伸：当某款商品从促销价恢复到原价，如果恢复后的价格恰好和上次的普通价相同，聪明的价签系统不会浪费电力刷新显示——这对应 Signals 的等值优化。整个店里的价签通过一张"布告栏"彼此感知变化：signal 把消息钉到布告栏上（标脏），computed 和 effect 等着被通知，而不是主动轮询。这就是"发布订阅的活动布告栏"比喻的来源。

这套系统有几个让人印象深刻的特性：它是懒惰的——不读计算格就不算；它是精确的——只有真正用到你的视图才会被更新；它是glitch-free 的——即便在复杂的钻石依赖中也不会读到中间状态。这些特性如何从源码中涌现，是本章要回答的问题。

Angular Signals 的实现分两层：原语层（packages/core/primitives/signals/src/）是与框架完全解耦的纯响应式图，理论上可以独立运行；集成层（packages/core/src/render3/reactivity/）将原语层接入 DI、组件生命周期和变更检测。我们会在这两层之间来回游走，看清楚每个抽象的意义，以及分层设计如何让 Angular 得以在不破坏现有应用的前提下引入全新的响应式机制。

三个核心原语

Signals 系统对外暴露三个原语，它们的关系如同一套生产链：

signal() 是最简单的原语：一个带版本号的值容器。你写入，它存着；你读取，它返回值并顺手记下"有人在追踪我"。与 RxJS 的 BehaviorSubject 不同，signal 没有流的语义——它只是当前时刻的一个值，不记录历史，不存在"未订阅就错过的"问题。

computed() 是派生信号。它的计算函数只有在被读取时才执行（懒计算），并缓存结果，只有当依赖的 signal 版本变化时才重新计算。它同时是消费者（读取上游 signal）和生产者（被下游读取）。由于它是只读的，你无法直接修改 computed 的值，这保证了数据流向的单向性——值只能由依赖的 signal 推动变化。

effect() 是副作用节点。当其依赖的 signal 或 computed 发生变化时，effect 的回调会被重新执行。与 computed() 不同，effect 不返回值，而是产生外部可见的副作用——打日志、发网络请求、更新非 Angular 管控的 DOM。effect 是整个响应式链路的"终点"，没有其他节点消费它的输出。

三者的核心区别可以用一句话概括：computed 是拉（pull）模式——有人读我，我才计算；effect 是推（push）模式——依赖一变，我就被调度执行。这个区别在第五节会深入展开。在数据流图中，signal 是源头，computed 是中间层，effect 是出口。数据流的方向永远是从 signal 流向 effect，不存在逆流。

理解这三个原语的使用边界，是写出良好 Signals 代码的前提：状态用 signal，派生用 computed，副作用用 effect。一个常见的错误是用 effect 来同步两个 signal 的值——这本质上是在"订阅变化然后手动更新"，而 computed 才是"自动同步派生值"的正确工具。effect 应该留给真正的副作用：持久化到 localStorage、调用第三方库 API、更新 canvas 等脱离 Angular 控制的操作。

💡 提示：注意 computed() 返回的是只读信号。你无法对它调用 .set()——这不是限制，而是约束：computed 应该是纯函数，其值完全由依赖决定。如果你需要一个"可覆盖的 computed"（例如切换数据源时重置选中项，但用户也可以手动选择），Angular v19 引入的 linkedSignal() 是正确答案。

ReactiveNode——响应式图的节点

如果说三个原语是 Signals 系统的外表，那么 ReactiveNode 就是骨架。每一个参与响应式图的节点——不管是 signal、computed 还是 effect——都实现了这个接口。理解 ReactiveNode，就等于理解了整个响应式图运作的底层逻辑。

ReactiveNode 的设计遵循一个朴素的原则：角色不是节点的属性，而是节点的行为方式。signal 之所以是"生产者"，是因为它有值让别人读；computed 之所以同时是"消费者"和"生产者"，是因为它既读别人也被别人读；effect 之所以是"消费者"，是因为它只读不产出。这种设计使整个图的实现极为统一——所有节点共享同一套接口，差异通过多态方法（producerRecomputeValue、consumerMarkedDirty 等）体现。

// packages/core/primitives/signals/src/graph.ts（第 112-199 行）
export interface ReactiveNode {
  version: Version;              // 此节点产出值的版本号，值变化时递增
  lastCleanEpoch: Version;       // 上次验证为"干净"时的全局 epoch（优化用）
  dirty: boolean;                // 作为消费者，是否需要重新计算
  recomputing: boolean;          // 是否正在重建生产者列表（循环检测用）

  producers: ReactiveLink | undefined;     // 依赖的生产者链表头
  producersTail: ReactiveLink | undefined; // 生产者链表尾
  consumers: ReactiveLink | undefined;     // 活跃消费者链表头
  consumersTail: ReactiveLink | undefined; // 消费者链表尾

  consumerAllowSignalWrites: boolean;   // 消费者上下文中是否允许写 signal
  readonly consumerIsAlwaysLive: boolean; // 是否始终活跃（effect 和模板为 true）
  kind: ReactiveNodeKind;               // 节点类型标识

  // 多态方法——不同节点类型有不同实现
  producerMustRecompute(node: unknown): boolean;
  producerRecomputeValue(node: unknown): void;
  consumerMarkedDirty(node: unknown): void;
  consumerOnSignalRead(node: unknown): void;
}

节点之间的边：ReactiveLink

生产者和消费者之间的关系通过 ReactiveLink 表示。每条边是一个独立的对象，同时存在于两条链中：

// packages/core/primitives/signals/src/graph.ts（第 82-89 行）
interface ReactiveLink {
  producer: ReactiveNode;
  consumer: ReactiveNode;
  lastReadVersion: number;        // 消费者上次读取时，生产者的 version
  prevConsumer: ReactiveLink | undefined;
  nextConsumer: ReactiveLink | undefined; // 同一生产者的消费者链
  nextProducer: ReactiveLink | undefined; // 同一消费者的生产者链
}

这是一个十字链表结构：每个 ReactiveLink 对象同时属于两条链——从消费者出发沿 nextProducer 可以遍历该消费者依赖的所有生产者；从生产者出发沿 nextConsumer 可以遍历该生产者的所有活跃消费者。

举个具体例子：假设有 computed d = b() + c()，d 的 producers 链就包含两个 ReactiveLink——一个指向 b，一个指向 c。与此同时，b.consumers 链和 c.consumers 链各自也包含指向 d 的 ReactiveLink。同一个链接对象，同时在两个方向被引用，这就是"十字"的含义。

每条链接还记录了 lastReadVersion：消费者上次读取这个生产者时，生产者的版本号。这个字段是后面版本号检查的关键——如果生产者当前 version > link.lastReadVersion，说明自上次读取以来生产者的值已经变化，消费者需要重新计算。

🔍 深入：Angular 选择链表而不是数组来管理边，是因为 computed 的依赖是动态的。如果 computed 函数有条件分支（if (a()) return b(); else return c();），当 a 的值从 true 变成 false 时，b 这条边需要被删除，c 这条边需要被添加。链表的增删操作是 O(1)，而数组则需要移位——对于频繁变化的动态依赖，链表是更好的选择。

版本号与 Epoch 机制

ReactiveNode 有两个版本相关的字段，它们承担不同的职责：

version：节点独有的版本号。只有当该节点的值真正发生变化时才递增。消费者通过比较自己记录的 lastReadVersion 与生产者当前的 version，来判断"我上次读它之后，它有没有变"。

epoch（全局变量）：每当任意 source signal 被 .set() 时递增。这是一个全局快捷路径：

// packages/core/primitives/signals/src/graph.ts（第 26 行）
let epoch: Version = 1 as Version;

export function producerIncrementEpoch(): void {
  epoch++;
}

当一个消费者验证完自己是"干净的"，会把当前 epoch 记录到 lastCleanEpoch。下次被读取时，如果全局 epoch 没有变化（意味着在此期间没有任何 signal 被写入），就可以跳过所有脏检查直接返回缓存值。这对模板渲染中大量并发信号读取的性能至关重要——在没有写入的情况下，每次渲染的脏检查开销接近于零。

// packages/core/primitives/signals/src/graph.ts（第 304-309 行）
if (!node.dirty && node.lastCleanEpoch === epoch) {
  return; // 快速路径：全局没有变化，无需检查任何依赖
}

生产者与消费者的角色

ReactiveNode 并不区分"生产者"和"消费者"——每个节点同时具备两种能力。区分角色的是节点实际的使用方式：

  • signal：永远不读取其他节点，因此只充当生产者
  • effect：永远不被其他节点读取，因此只充当消费者
  • computed：读取上游（是消费者），同时被下游读取（是生产者）

这一设计使得图的中间层（computed）能够透明地参与依赖传播，无需特殊处理。ReactiveNode 的这种"角色由行为定义"的设计哲学，也使得未来扩展新的原语（如 linkedSignal）成为可能——只需实现相应的多态方法，即可无缝融入现有的响应式图机制。

活跃消费者（Live Consumer）机制

并非所有 computed 都会始终待在生产者的消费者链表中。Angular 区分了活跃消费者和非活跃消费者：

// packages/core/primitives/signals/src/graph.ts（第 548-550 行）
function consumerIsLive(node: ReactiveNode): boolean {
  return node.consumerIsAlwaysLive || node.consumers !== undefined;
}

effect 和模板消费者设置 consumerIsAlwaysLive: true，因此永远活跃，始终注册在生产者的消费者链表中。computed 的 consumerIsAlwaysLive 为 false，只有当它自身也被某个活跃消费者（直接或传递地）订阅时，才会被加入生产者的链表——这种活跃性是传递的。

这个设计有一个重要的实际含义：如果你创建了一系列 computed，但没有将它们绑定到任何 effect 或模板，这些 computed 形成的子图不会在响应式图中注册任何边，不会占用内存，也不会在 signal 变化时被遍历。这避免了"创建了大量未使用的 computed 导致内存泄漏"的问题。

活跃性的传递：图遍历如何动态建立依赖链

活跃性的传递不是元数据标记，而是在依赖追踪时动态建立的。以 effect → computed A → computed B → signal 这条链为例：

  • effect 首次执行时，将自身设为 activeConsumer，读取 computedA()，触发 producerAccessed(computedA)，在 effect 和 A 之间建立 ReactiveLink——此时 A.consumers !== undefined，consumerIsLive(A) === true；
  • A 求值时将自身设为 activeConsumer，读取 computedB()，在 A 和 B 之间建立边——B.consumers !== undefined，consumerIsLive(B) === true；
  • B 求值时读取 signal()，signal 的消费者链表中出现 B。

当 signal.set() 触发 producerNotifyConsumers(signal) 时，B 被标脏；B 传播到 A，A 传播到 effect——整条链都参与了推阶段。

反过来，若 computedA 从未被任何 effect 或模板读取，A.consumers === undefined，consumerIsLive(A) === false。producerAccessed() 在尝试建立 A 到 B 的边时，因 A 不是活跃消费者，B 不会将 A 加入自己的消费者链表。于是整条 A → B → signal 子图与响应式图完全解耦，signal 的推阶段不会遍历到任何非活跃节点，不占用任何计算资源。

signal() 的实现

signal() 的原语实现在 packages/core/primitives/signals/src/signal.ts，Angular 公共 API 只是薄薄的一层包装。

创建与数据结构

// packages/core/primitives/signals/src/signal.ts（第 53-73 行）
export function createSignal<T>(initialValue: T, equal?: ValueEqualityFn<T>) {
  const node: SignalNode<T> = Object.create(SIGNAL_NODE);
  node.value = initialValue;
  if (equal !== undefined) { node.equal = equal; }

  // getter 函数本身就是 signal 的"门面"
  const getter = (() => signalGetFn(node)) as SignalGetter<T>;
  (getter as any)[SIGNAL] = node;  // 把 node 挂在函数对象上，供框架访问
  return [getter, set, update];
}

signal 的 getter 是一个闭包函数。当你写 count() 时，实际上是调用了 () => signalGetFn(node)，而 node 通过 getter[SIGNAL] 属性附在函数对象上。这个设计让 signal 既能像函数一样调用（读值），又能通过 .set() / .update() 方法修改——同一个函数对象同时是"读接口"和"写接口"的载体。

读取路径：producerAccessed

// packages/core/primitives/signals/src/signal.ts（第 81-84 行）
export function signalGetFn<T>(node: SignalNode<T>): T {
  producerAccessed(node);  // 关键：注册依赖
  return node.value;
}

producerAccessed() 是依赖追踪的入口。它检查当前是否存在活跃消费者（activeConsumer 全局变量），如果有，就在生产者和消费者之间创建或更新一条 ReactiveLink。

// packages/core/primitives/signals/src/graph.ts（第 204-283 行，节选）
export function producerAccessed(node: ReactiveNode): void {
  if (activeConsumer === null) {
    return;  // 没有活跃消费者，忽略（如在模板外裸调用 signal()）
  }
  // 建立或复用 ReactiveLink，记录 lastReadVersion
  // ... 链表操作
}

activeConsumer 何时被设置？当 computed 或 effect 开始执行其计算函数时，会先调用 consumerBeforeComputation(node) 将自己设为 activeConsumer，执行完毕后再调用 consumerAfterComputation() 恢复。因此，计算函数内部读取的所有 signal，都会自动注册为该 computed/effect 的依赖。

这个"全局变量充当上下文"的设计有一个重要限制：依赖追踪是单线程同步的。在异步回调（setTimeout、Promise.then）中读取 signal，因为计算函数已经返回、activeConsumer 已被清除，所以不会建立依赖。这也是为什么 effect 的回调函数必须是同步的——如果 effect 内部有异步操作，只有 await 之前同步读取到的 signal 才会被追踪。

📝 练习：在 effect 内部创建一个 setTimeout，在其回调中读取一个 signal。观察修改该 signal 时，外层 effect 是否会重新执行。答案是不会，原因正是上述机制。

写入路径：完整传播链

// packages/core/primitives/signals/src/signal.ts（第 86-125 行）
export function signalSetFn<T>(node: SignalNode<T>, newValue: T) {
  if (!producerUpdatesAllowed()) {
    throwInvalidWriteToSignalError(node); // computed 内部禁止写入
  }
  if (!node.equal(node.value, newValue)) { // 默认用 Object.is 比较
    node.value = newValue;
    signalValueChanged(node);
  }
}

function signalValueChanged<T>(node: SignalNode<T>): void {
  node.version++;                  // ① 递增自身版本号
  producerIncrementEpoch();        // ② 递增全局 epoch
  producerNotifyConsumers(node);   // ③ 通知所有活跃消费者（标脏）
                                   //    非活跃消费者（consumerIsLive = false）不在消费者链表中，
                                   //    自动跳过——这是非活跃子图的第二层快速路径
  postSignalSetFn?.(node);         // ④ 可选钩子（Angular 集成层用于同步调度）
}

producerNotifyConsumers 遍历消费者链表，对每个未标脏的消费者调用 consumerMarkDirty()——这是一个递归操作：

// packages/core/primitives/signals/src/graph.ts（第 359-363 行）
export function consumerMarkDirty(node: ReactiveNode): void {
  node.dirty = true;
  producerNotifyConsumers(node);     // 继续向下传播，computed 也作为生产者通知其消费者
  node.consumerMarkedDirty?.(node);  // 多态回调：模板/effect/根effect 各有不同响应
}

这里有一个关键设计：脏标记是推的，值计算是拉的。signal.set() 只将整条依赖链上的所有节点标记为脏，并不触发任何重算。当需要读取最终值时，才从末端往前拉取最新值。这个"推脏/拉值"模式是 Angular Signals 保证 glitch-free 的根本机制，第六节会详细展开。

值得注意的是 postSignalSetFn 这个可选钩子（signalValueChanged 第四步）。Angular 集成层在初始化时会设置这个钩子，用于在 Zoneless 模式下（Angular v18+）通知 ChangeDetectionScheduler 有 signal 发生了变化。在有 Zone.js 的应用中，Zone 的 async task 机制也会在合适时机触发 tick，两条路径最终都会进入 ApplicationRef._tick()。这个钩子是原语层与框架层之间唯一的"跨层通知"通道，保持了原语层的纯洁性。

update() 与 mutate() 的差异

signal.update(fn) 是 signal.set(fn(currentValue)) 的语法糖：

// packages/core/src/render3/reactivity/signal.ts（第 72-89 行）
export function signal<T>(initialValue: T, options?: CreateSignalOptions<T>): WritableSignal<T> {
  const [get, set, update] = createSignal(initialValue, options?.equal);
  const signalFn = get as SignalGetter<T> & WritableSignal<T>;
  signalFn.set = set;
  signalFn.update = update;  // update = fn => set(fn(get()))
  signalFn.asReadonly = signalAsReadonlyFn.bind(signalFn as any);
  return signalFn;
}

⚠️ 注意：Angular v17 之前存在 signal.mutate() 方法，允许直接修改对象内部而不替换引用，调用后强制通知消费者。该方法在 v17.1 已被移除，因为它绕过了等值检查，容易导致意外的重新渲染。如果你需要更新数组或对象，推荐的方式是用 update() 返回一个新引用：items.update(arr => [...arr, newItem])。

computed() 的懒计算

computed() 是 Signals 中最精妙的原语。它解决了一个看似矛盾的需求：自动追踪依赖、值变化时自动更新，但又不做不必要的计算。懒计算（lazy evaluation）这个词在这里不只是性能优化的说辞，它是 computed 语义的核心——一个未被读取的 computed，其计算函数在任何情况下都不会执行，哪怕依赖的 signal 已经变化了无数次。

三个哨兵值

ComputedNode 用三个特殊的 Symbol 来表示值的状态，而不是用 null 或 undefined（它们可能是合法的计算结果）：

// packages/core/primitives/signals/src/computed.ts（第 99-113 行）
export const UNSET: any = Symbol('UNSET');       // 从未计算过（初始状态）
export const COMPUTING: any = Symbol('COMPUTING'); // 正在计算中（循环检测）
export const ERRORED: any = Symbol('ERRORED');     // 上次计算抛出了异常

初始状态下 node.value === UNSET，这意味着首次读取时必须执行计算。COMPUTING 状态用于检测循环依赖：如果 computed A 的计算函数中读取了 computed A 自身，就会看到值为 COMPUTING 并抛出错误。

读取路径：懒惰求值

// packages/core/primitives/signals/src/computed.ts（第 70-83 行）
const computed = () => {
  producerUpdateValueVersion(node);  // 确保值是最新的（如有必要，触发重算）
  producerAccessed(node);            // 注册到上游消费者的依赖列表
  if (node.value === ERRORED) { throw node.error; }
  return node.value;
};

producerUpdateValueVersion 是懒计算的核心：

// packages/core/primitives/signals/src/graph.ts（第 297-322 行）
export function producerUpdateValueVersion(node: ReactiveNode): void {
  // 快速路径1：活跃且干净 → 直接返回
  if (consumerIsLive(node) && !node.dirty) { return; }
  // 快速路径2b：非活跃消费者（consumerIsLive 为 false）且脏 → 直接跳过脏检查
  // 非活跃节点从未进入任何生产者的消费者链表，推阶段不会通知它们，
  // 因此无需通知任何消费者，节省递归遍历开销
  if (!consumerIsLive(node) && node.dirty) { return; }
  // 快速路径2：全局 epoch 未变 → 直接返回
  if (!node.dirty && node.lastCleanEpoch === epoch) { return; }
  // 检查是否首次计算，或依赖是否真的变了
  if (!node.producerMustRecompute(node) && !consumerPollProducersForChange(node)) {
    producerMarkClean(node);  // 依赖没变，标记为干净并返回
    return;
  }
  // 真正的重算
  node.producerRecomputeValue(node);
  producerMarkClean(node);
}

依赖验证由 consumerPollProducersForChange() 完成——它遍历所有生产者，比较记录的 lastReadVersion 与生产者当前的 version，若不一致则说明依赖真的变了：

// packages/core/primitives/signals/src/graph.ts（第 448-472 行）
export function consumerPollProducersForChange(node: ReactiveNode): boolean {
  for (let link = node.producers; link !== undefined; link = link.nextProducer) {
    const producer = link.producer;
    const seenVersion = link.lastReadVersion;
    if (seenVersion !== producer.version) { return true; }
    // 递归验证：确保生产者自己也是最新的（关键！glitch-free 的来源）
    producerUpdateValueVersion(producer);
    if (seenVersion !== producer.version) { return true; }
  }
  return false;
}

动态依赖追踪与等值优化

每次 computed 重算时，consumerBeforeComputation() 先清除旧的依赖列表，然后在计算函数执行过程中重新建立依赖。这意味着依赖是动态的，不是静态订阅。

动态依赖追踪对有条件分支的计算非常重要。考虑：

const mode = signal<'a' | 'b'>('a');
const valueA = signal(10);
const valueB = signal(20);

const result = computed(() => {
  if (mode() === 'a') return valueA();  // 只依赖 valueA
  return valueB();                       // 只依赖 valueB
});

当 mode() 为 'a' 时，result 的依赖列表只包含 mode 和 valueA——修改 valueB 不会触发 result 重算，因为它根本没被读取过。这种按实际执行路径动态建立的依赖关系，比静态声明依赖更精确，减少了不必要的重算。

// packages/core/primitives/signals/src/computed.ts（第 132-173 行，节选）
producerRecomputeValue(node: ComputedNode<unknown>): void {
  const oldValue = node.value;
  node.value = COMPUTING;  // 防止循环依赖

  const prevConsumer = consumerBeforeComputation(node); // 设为 activeConsumer
  try {
    newValue = node.computation();  // 执行用户函数，自动追踪依赖
    setActiveConsumer(null);        // equal() 调用不应追踪信号
    wasEqual = oldValue !== UNSET && node.equal(oldValue, newValue);
  } finally {
    consumerAfterComputation(node, prevConsumer); // 恢复 activeConsumer，清理过期边
  }

  if (wasEqual) {
    node.value = oldValue;  // ← 关键！值相等时保留旧引用，不递增 version
    return;
  }
  node.value = newValue;
  node.version++;  // version 递增 → 下游才能感知到变化
}

等值优化是 Signals 的一个重要特性：如果 computed 重算后的新值与旧值相等（通过 equal 函数判断，默认 Object.is），则 node.version 不递增。下游的 effect 或模板在做 consumerPollProducersForChange() 时，会发现版本没有变，从而阻断传播，不触发重新渲染。这意味着即使 signal 的某个字段变了，如果 computed 提取的那个子集没变，整条依赖链上的 UI 就不会更新。

这在列表过滤、聚合计算等场景中非常有价值——用 computed 提取感兴趣的派生值，天然获得细粒度的更新隔离：

// 用户列表中，只关心"是否有管理员"
const users = signal<User[]>([]);
const hasAdmin = computed(() => users().some(u => u.role === 'admin'));

// 即使 users 数组频繁更新（添加普通用户），
// 只要 hasAdmin 的结果不变，依赖它的视图就不会重渲染

💡 提示：等值判断默认使用 Object.is（严格相等 + 处理 NaN）。如果 computed 返回数组或对象，每次新建引用即使内容相同也不相等，等值优化就会失效。这种情况下，可以传入自定义 equal 函数：computed(() => [...ids()].sort(), { equal: (a, b) => a.length === b.length && a.every((v, i) => v === b[i]) })。

effect() 的调度

effect 的本质：始终活跃的消费者

effect 与 computed 最根本的区别在于 consumerIsAlwaysLive 标志：

// packages/core/primitives/signals/src/effect.ts（第 36-43 行）
export const BASE_EFFECT_NODE = {
  ...REACTIVE_NODE,
  consumerIsAlwaysLive: true,     // effect 始终注册在生产者的消费者链表中
  consumerAllowSignalWrites: true, // effect 内允许写 signal（与 computed 不同）
  dirty: true,                    // 初始为脏，确保首次执行
  kind: 'effect',
};

consumerIsAlwaysLive: true 的含义是：即使没有其他消费者订阅这个 effect，它也始终存在于其依赖的 signal 的消费者链表中。这是"推"模式的基础——signal 变化时才能主动通知到 effect。

相比之下，computed 默认 consumerIsAlwaysLive: false，只有当它被某个 consumerIsAlwaysLive 的节点（直接或间接）订阅时，才会被加入生产者的消费者链表。未被任何 effect 或模板使用的 computed 链，不会在图中注册任何边，避免内存泄漏。

这个机制还有一个微妙之处：computed 的活跃性是传递的。若 effect → computed A → computed B → signal，那么 signal 变化时，B 也会在 signal 的消费者链表中注册——因为 B 被 A 消费，A 被 effect 消费，effect 是活跃的，活跃性沿链向上传递。整个响应式子图在没有 effect 根节点时能够"安静"下来，不占用任何资源。

两种 effect 的执行时机

Angular 的 effect API 在集成层分为两种，理解它们的差异对调试很重要：

组件 effect（ViewEffectNode）：在 @Component 内通过 effect() 创建，绑定到组件的生命周期。其 consumerMarkedDirty 实现如下：

// packages/core/src/render3/reactivity/effect.ts（第 275-282 行）
consumerMarkedDirty(this: ViewEffectNode): void {
  this.view[FLAGS] |= LViewFlags.HasChildViewsToRefresh;
  markAncestorsForTraversal(this.view);  // 沿视图树向上标记祖先
  this.notifier.notify(NotificationSource.ViewEffect);
}

组件 effect 在变更检测期间由 runEffectsInView() 执行，作为视图刷新的前置步骤，在模板渲染前执行——与 ngDoCheck 处于同一调度批次。它的生命周期与组件绑定，组件销毁时 effect 自动清理。

根 effect（RootEffectNode）：在注入上下文外或通过 inject() 创建（如在服务中）。其 consumerMarkedDirty 实现：

// packages/core/src/render3/reactivity/effect.ts（第 254-260 行）
consumerMarkedDirty(this: RootEffectNode) {
  this.scheduler.schedule(this);                        // 加入调度队列
  this.notifier.notify(NotificationSource.RootEffect);  // 通知触发 tick
}

根 effect 通过 EffectScheduler 独立调度，在 ApplicationRef.synchronizeOnce() 的开头通过 effectScheduler.flush() 批量执行——早于模板变更检测。这意味着根 effect 的执行先于视图更新，若 effect 内修改了 signal，该变化能在同一个 tick 内被视图捕获，不需要等到下一个 tick。

正如第6章分析的 synchronizeOnce() 代码所示，执行顺序是：effectScheduler.flush()（根 effect 批量执行）→ 遍历所有视图执行变更检测 → afterRender 钩子。这个顺序保证了视图看到的是 effect 执行后的最新状态。

📝 防护机制：在执行 effect 回调之前，runEffect() 会先对 effect 节点执行 version++。这个自增防止了"effect 内写入 signal → signal 脏标记传播回 effect → effect 被再次调度"的无限循环：当 signal 的脏标记传播回同一 effect 时，consumerPollProducersForChange 发现 effect 的版本已被递增，从而跳过重复调度。这是一个低开销的防循环保护，比 MAXIMUM_REFRESH_RERUNS 硬性限制更早触发。

⚠️ 注意：effect() 不是同步执行的。调用 signal.set(value) 后，effect 不会立即运行——它会被调度到下一个 microtask（或 requestAnimationFrame）。在同一同步块中多次调用 set()，effect 只会运行一次，拿到最终值。这就是 Angular Signals 的自动批处理。

untracked()：读而不订阅

有时候你需要在 effect 中读取 signal 的值，但不希望将其建立为依赖——比如在日志记录中，你只是想打印当前值，不希望每次变化都重新触发整个 effect。这时可以用 untracked()：

// packages/core/primitives/signals/src/untracked.ts（第 15-24 行）
export function untracked<T>(nonReactiveReadsFn: () => T): T {
  const prevConsumer = setActiveConsumer(null); // 暂时清除 activeConsumer
  try {
    return nonReactiveReadsFn();
  } finally {
    setActiveConsumer(prevConsumer); // 恢复
  }
}

untracked() 在 toObservable() 的实现中也有使用：subject.next() 在 effect 上下文中被调用，但如果 next() 内部触发了其他 signal 读取，这些读取不应该成为 effect 的依赖。

钻石问题与 glitch-free 保证

这是本章最值得细品的部分。"钻石问题"（Diamond Problem）这个名字来自依赖图的形状：一个节点有两条路径流向另一个节点，形成菱形（钻石形）。它是响应式系统设计中的经典难题，Angular 的解法颇具代表性。

什么是钻石问题

考虑以下依赖结构：

const a = signal(1);
const b = computed(() => a() * 2);  // b 依赖 a
const c = computed(() => a() * 3);  // c 依赖 a
const d = computed(() => b() + c()); // d 依赖 b 和 c
flowchart TD A["a = signal(1)"] B["b = computed<br/>a() * 2 = 2"] C["c = computed<br/>a() * 3 = 3"] D["d = computed<br/>b() + c() = 5"] E["effect / 模板"] A -->|"写入: a.set(2)"| B A -->|"写入: a.set(2)"| C B --> D C --> D D --> E style A fill:#ffecb3,stroke:#f57f17 style B fill:#e3f2fd,stroke:#1976d2 style C fill:#e3f2fd,stroke:#1976d2 style D fill:#f3e5f5,stroke:#7b1fa2 style E fill:#e8f5e9,stroke:#388e3c

当 a.set(2) 时，如果系统的实现是纯推（pure push）——每当生产者变化就立即重算消费者——会发生什么？

  • a 变化 → 先通知 b → b 重算为 4 → b 变化 → 通知 d → d 尝试读取 b() + c()，此时 c 还是旧值 3 → d 得到 7，而不是正确的 10

这种中间状态读取就是 glitch（毛刺）。它不只是性能问题，而是正确性问题。

Angular 的解决方案：推脏拉值

Angular Signals 使用 push-pull 混合模型彻底消除 glitch：

推阶段（signal.set() 触发）：只传播脏标记，不重算任何值。所有直接和间接消费者的 dirty 标志被设为 true，不涉及任何计算。

a.set(2)
  → b.dirty = true（b 作为消费者）
  → b 的 producerNotifyConsumers 传播：d.dirty = true
  → c.dirty = true（c 作为消费者）
  → c 的 producerNotifyConsumers 传播：d.dirty = true（已为 true，幂等）

推阶段结束后，b、c、d 都被标脏，但没有任何计算发生。a.version 递增，全局 epoch 递增，这是推阶段唯一的"副作用"。

拉阶段（effect 或模板读取 d() 时触发）：从末端节点出发，递归地验证并重算依赖链：

d()
  → producerUpdateValueVersion(d)
    → d.dirty = true → 检查依赖是否真的变了
      → consumerPollProducersForChange(d)
        → 检查 b：b.version vs lastReadVersion 不同
          → producerUpdateValueVersion(b) ← 先保证 b 最新
            → b 重算：b() = 2 * 2 = 4，b.version++
        → 检查 c：c.version vs lastReadVersion 不同
          → producerUpdateValueVersion(c) ← 先保证 c 最新
            → c 重算：c() = 2 * 3 = 6，c.version++
      → 依赖确实变了，d 执行重算
    → d 重算：d() = b() + c() = 4 + 6 = 10 ✓

关键在 consumerPollProducersForChange() 中的递归调用：在判断 d 的依赖是否变化之前，会先调用 producerUpdateValueVersion(b) 和 producerUpdateValueVersion(c)，确保它们都基于最新的 a 值重算完毕。这形成了一个深度优先的按需重算过程，保证任何中间节点被读取时其依赖都已是最新状态。

加上等值优化，还有第三重保障：如果 b 重算后值与旧值相等（wasEqual = true），b.version 不递增，d 看到版本没变，不触发重算，整条链就此截断。这在实际应用中非常有价值——例如一个根据用户状态计算权限的链：userState → role → hasAdminPermission → UI，如果 userState 的某个字段变化但 role 没变，hasAdminPermission 和 UI 都不会重算，精准地避免了不必要的刷新。

与变更检测的集成

Signals 的精妙之处不仅在于响应式图的内部机制，还在于它如何无缝接入 Angular 已有的变更检测体系。正如第6章所分析的，变更检测引擎通过 LViewFlags 位掩码来决定哪些视图需要刷新。Signals 的集成，本质上是将"signal 变化"这一事件翻译成"设置正确的 LViewFlags"这一操作。

模板消费者：ReactiveLViewConsumer

每个 Signal 组件（标有 SignalView 标志的组件）的模板渲染，都有一个对应的 ReactiveLViewConsumer 节点挂载在 LView 上。这个节点充当模板的代理消费者：

// packages/core/src/render3/reactive_lview_consumer.ts（第 43-53 行）
consumerMarkedDirty: (node: ReactiveLViewConsumer) => {
  markAncestorsForTraversal(node.lView!);
  // markAncestorsForTraversal 会沿视图树向上给每个祖先设置
  // LViewFlags.HasChildViewsToRefresh 标志
}

当模板读取到某个 signal 时，该 signal 与 ReactiveLViewConsumer 之间建立 ReactiveLink。当 signal 变化时，consumerMarkedDirty 被调用，向上标记祖先视图，最终触发 ChangeDetectionScheduler 调度一次 tick。

这里有一个值得注意的细节：ReactiveLViewConsumer 在每次模板渲染开始时被设置为 activeConsumer，渲染过程中读取到的所有 signal 都会自动成为该视图的依赖。渲染结束后，旧的依赖边被清理，新的依赖边被建立。这与 computed 的动态依赖追踪机制完全相同——模板渲染，从 Signals 的角度看，就是一次特殊的 computed 求值过程，只是其"输出"是 DOM 更新而不是一个值。

从 signal 写入到视图更新的完整路径如下：

注意图中的microtask 边界：signal.set() 是同步的，但从调度通知到 _tick() 执行之间存在一个异步边界。这就是 Angular Signals 实现自动批处理的机制——在同一同步执行块中多次写入 signal，只会触发一次 tick。

input() 与传统 @Input() 的底层差异

input() signal-based 输入在底层复用了 SignalNode：

// packages/core/src/authoring/input/input_signal_node.ts（第 42-51 行）
export const INPUT_SIGNAL_NODE: InputSignalNode<unknown, unknown> = {
  ...SIGNAL_NODE,
  transformFn: undefined,
  applyValueToInputSignal<T, TransformT>(node: InputSignalNode<T, TransformT>, value: T) {
    signalSetFn(node, value);  // 框架内部通过此方法写入值
  },
};

与传统 @Input() 的本质区别在于通知机制：传统 @Input() 的属性变化通过 ngOnChanges 钩子通知组件，依赖 Zone.js 或 markForCheck() 才能触发视图更新；input() signal 的值变化直接触发响应式图的传播，精准地只更新依赖该 input 的计算和视图。

对于 input()，框架调用 applyValueToInputSignal() 写入新值，这与 signal.set() 走的是完全相同的路径——signalSetFn → signalValueChanged → producerNotifyConsumers。视图中所有依赖该 input 的 computed 会被标脏，相关部分的 DOM 会在下次 tick 时精确更新。

此外，input() 对外是只读的——InputSignal<T> 类型不暴露 set() 和 update() 方法。只有框架通过 applyValueToInputSignal() 这个内部 API 写入。这个约束在类型层面强制了"数据从父到子单向流动"的原则，避免了子组件意外修改父组件传入的值。

🔗 交叉引用：第6章分析的 LViewFlags.HasChildViewsToRefresh 标志和 markAncestorsForTraversal() 函数，正是 Signals 与变更检测的连接点。Signal 组件的精准更新，就是通过设置这个标志，告诉变更检测引擎"我这个分支需要刷新"。

RxJS 互操作

Signal 和 Observable 各有长处——Signal 同步、始终有值、天然懒计算，非常适合表达 UI 状态；Observable 擅长表达时序、多值和异步流，在处理 HTTP 请求、WebSocket、用户输入防抖等场景仍然是更自然的选择。Angular 提供了两个互操作函数，让两个世界可以互联，开发者可以在新旧两种模式之间渐进过渡。

toSignal()：Observable → Signal

// packages/core/rxjs-interop/src/to_signal.ts（第 154-224 行，节选）
// 1. 内部状态用一个包含 kind/value/error 的 State 对象存储
let state: WritableSignal<State<T | U>>;
state = signal<State<T | U>>({ kind: StateKind.Value, value: options?.initialValue as U });

// 2. 订阅 Observable，每次发射更新 state signal
const sub = source.subscribe({
  next: (value) => state.set({ kind: StateKind.Value, value }),
  error: (error) => state.set({ kind: StateKind.Error, error }),
});

// 3. 返回一个 computed，解包 State 对象
return computed(() => {
  const current = state();
  switch (current.kind) {
    case StateKind.Value: return current.value;
    case StateKind.Error: throw current.error;
    case StateKind.NoValue: throw new Error('No value available yet.');
  }
});

toSignal 返回的实际上是一个 computed，不是 WritableSignal。内部有一个 WritableSignal 存储 Observable 推来的最新状态，外层 computed 负责解包。这意味着等值优化同样适用：如果 Observable 连续发出相同的值，computed 的 equal 函数会阻止下游更新。

为什么用 State 对象包装而不是直接存储值？因为 Observable 有三种终态：正常值、错误和无值（初始状态）。Signal 本身没有错误状态的概念，但通过 computed 内部的 throw，错误可以被缓存在 ComputedNode.error 字段中，下次读取时重新抛出。这是一个优雅的语义桥接。

⚠️ 注意：toSignal() 必须在注入上下文中调用（构造函数或 runInInjectionContext() 内），因为它内部需要通过 inject(DestroyRef) 注册清理订阅的回调。如果 Observable 可能在订阅时没有立即发射值，需要提供 { initialValue: ... } 或使用 requireSync: true 配合同步 Observable。

toObservable()：Signal → Observable

// packages/core/rxjs-interop/src/to_observable.ts（第 48-75 行）
export function toObservable<T>(source: Signal<T>, options?: ToObservableOptions): Observable<T> {
  const subject = new ReplaySubject<T>(1); // 确保新订阅者获得最新值

  const watcher = effect(() => {
    let value: T;
    try {
      value = source();  // 在 effect 中读取 signal → 建立依赖
    } catch (err) {
      untracked(() => subject.error(err));
      return;
    }
    untracked(() => subject.next(value)); // untracked：subject.next 内部不应追踪
  }, { injector, manualCleanup: true });

  injector.get(DestroyRef).onDestroy(() => {
    watcher.destroy();
    subject.complete();
  });

  return subject.asObservable();
}

实现极为简洁：用一个 effect 追踪 signal，signal 变化时 effect 重新运行，通过 ReplaySubject 推送给 Observable 订阅者。untracked() 包裹 subject.next() 是必要的——否则 next() 内部如果触发了任何 signal 读取（比如订阅者的 pipe 操作中），会被错误地记录为这个 effect 的依赖。

ReplaySubject(1) 的选择也有讲究：它会缓存最新一个值，使得新订阅者能立即获得当前 signal 的值，而不需要等待下次变化。这保证了 toObservable() 返回的 Observable 具有与 signal 相似的"始终有当前值"语义。

⚠️ 注意：toSignal() 和 toObservable() 都必须在注入上下文中调用，因为它们内部需要通过 inject(DestroyRef) 注册销毁回调。如果在路由守卫或服务的工厂函数中使用，需要手动传入 injector 参数。

toSignal() 的实战陷阱

陷阱1：多组件共用 toSignal() 导致重复订阅

// ❌ 危险写法：每个组件实例各自调用 toSignal()，持有独立订阅
@Component({ ... })
export class ProductListComponent {
  products = toSignal(this.http.get<Product[]>('/api/products'));
}

若同一页面渲染了多个 ProductListComponent 实例，HTTP 请求会被发起多次——每个 toSignal() 内部调用 source.subscribe()，订阅同一个 Cold Observable（HTTP 请求是 Cold 的）会独立触发一次请求。正确做法是将 toSignal() 提升到单例服务中，所有组件共享同一信号：

// ✅ 正确：在服务中创建一次，所有组件读取同一个 Signal
@Injectable({ providedIn: 'root' })
export class ProductService {
  products = toSignal(this.http.get<Product[]>('/api/products'), { initialValue: [] });
}

陷阱2：initialValue 缺失时的类型安全问题

不提供 initialValue 时，toSignal() 返回 Signal<T | undefined>——在 Observable 同步发射值之前，信号的初始状态是 undefined。若消费侧未处理 undefined 分支，运行时可能抛出错误：

const data = toSignal(http.get<User>('/api/user'));
// data 类型：Signal<User | undefined>

const userName = computed(() => data()?.name ?? '加载中...');  // ✅ 安全消费

// 若 Observable 保证同步发射（如 of(defaultUser)），可用 requireSync 消除 undefined
const data2 = toSignal(of(defaultUser), { requireSync: true }); // Signal<User>，无 undefined

设计思考

为什么选择 push-pull 混合模型，而不是纯 push？

纯 push 的实现（如早期的 Vue 2 Object.defineProperty）在依赖关系简单时效率高，但在 DAG（有向无环图）中很容易产生 glitch，必须引入拓扑排序来保证更新顺序。MobX 就是这一路线的代表：它维护一个全局更新队列，按拓扑排序执行重算，保证 glitch-free，但这意味着必须在每次写入时更新整个图的拓扑状态，有持续的内存和 CPU 开销。Solid.js 的 Signals 也选择了纯 push + 调度队列的方式，确实代码路径更短，但为 glitch-free 付出了维护队列的代价。

Angular 选择了 push-pull：推阶段只传播脏标记（成本极低，没有任何值计算），拉阶段在读取时递归按需重算（可以理解为"惰性拓扑排序"）。这带来几个优势：如果没有人读取某个 computed，它的计算函数永远不会执行，在服务端渲染或懒加载场景中节省了大量不必要的计算；等值优化可以在链路中间截断传播，避免大量下游重算；响应式图不需要全局的排序状态，可以并发地读写不同的子图而不产生干扰。

glitch-free 的代价是什么？

Push-pull 模型的代价是读取时的递归验证开销。当读取一个深层 computed 时，需要递归验证其所有依赖的依赖是否最新，这在极端情况下（超深依赖链、高扇入）可能造成较高的调用栈开销。Angular 用两个机制缓解了这个问题：lastCleanEpoch 提供了全局快捷路径——如果全局 epoch 没变，所有验证在 O(1) 时间内完成；consumerIsLive 过滤掉未被任何活跃消费者订阅的子图，使这些子图完全不参与推阶段的传播。综合来看，在典型的应用中（依赖链深度有限，大多数 tick 没有大量写入），push-pull 的开销是可接受的。

Signals 与 Zone.js 的共存策略

Angular v18-v19 处于从 Zone.js 到 Zoneless 的过渡期，两种机制可以共存。Zone.js 触发的全局 tick 会遍历所有视图，Signal 组件通过 markAncestorsForTraversal() 额外获得精准的 HasChildViewsToRefresh 标记，两者都能工作。对于 Zoneless 应用，Signals 是唯一的变更检测触发源，Zone.js 的全局轮询完全退出。

推荐的迁移路径：OnPush + Signals（中间状态）→ Zoneless + Signals（目标状态）。在第一阶段，OnPush 减少了全树检查的范围，Signals 进一步精确到组件内部的 signal 级别；在第二阶段，Zone.js 完全退出，变更检测由 Signals 精准驱动，性能达到最优。Angular 团队在 v19 将 Zoneless 标记为稳定，预示着这一迁移路径已经足够成熟，可以在生产环境中使用。

小结

本章从 ReactiveNode 数据结构出发，拆解了 Angular Signals 的完整实现链路。Signals 系统的设计美在于：复杂的响应式语义（懒计算、自动追踪、glitch-free）都从几个简单的原则中涌现——版本号、脏标记、活跃消费者机制、递归按需重算。以下是关键结论：

  • 响应式图是一个 DAG，节点同时具备生产者和消费者角色，通过 ReactiveLink 十字链表连接；version（节点维度）和 epoch（全局维度）协同提供高效的脏检测
  • signal 是值容器：读取时通过 producerAccessed() 自动注册依赖，写入时通过 signalValueChanged() 递增版本、传播脏标记
  • computed 是懒的：dirty = true 不等于立刻重算——只有被读取时才通过 producerUpdateValueVersion() 按需重算，等值优化（wasEqual）可在中间节点截断传播
  • effect 永远活跃（consumerIsAlwaysLive: true），是响应式链路的终点，分为组件 effect（CD 期间执行）和根 effect（microtask 调度执行）两种
  • glitch-free 保证来自 push-pull 混合模型：producerNotifyConsumers 只推脏标记，consumerPollProducersForChange 在读取时递归验证依赖树，确保最终消费者看到一致快照
  • 与变更检测的集成通过 ReactiveLViewConsumer 实现：模板渲染即是一次特殊的 computed 求值，依赖自动追踪，变化自动调度 tick
  • RxJS 互操作：toSignal() 用 WritableSignal + computed 桥接，toObservable() 用 effect + ReplaySubject 桥接，两者均通过 untracked() 避免依赖污染

思考题

【基础】掌握核心传播链

  1. 在模板中写 {{ count() }}，ReactiveLViewConsumer 会建立对 count 的依赖。现在连续调用两次 count.set(1)（第一次从 0 改为 1，第二次仍然是 1）——这两次写入各自会触发哪些步骤（版本号、epoch、脏标记、视图调度）？两次调用在行为上有何不同？

答案提示：第一次写入：count.version++ → epoch++ → consumerMarkDirty(ReactiveLViewConsumer) → markAncestorsForTraversal() → ChangeDetectionScheduler.notify() → 下一 tick 更新 DOM。第二次写入：signalSetFn 中 Object.is(1, 1) === true → 相等检查阻止 signalValueChanged() → 不执行任何后续步骤，无脏标记传播，无视图调度。

【中级】深入等值优化与批处理

  1. 创建 const isEven = computed(() => count() % 2 === 0)，在同一同步块中连续调用 count.set(1)、count.set(3)、count.set(5)。依赖 isEven 的 effect(() => console.log(isEven())) 最终会执行几次，打印什么值？请从批处理合并（自动批处理）、version 递增和等值优化的角度逐步分析——注意拉取（pull）发生在调度批次结束之后，而不是每次 set() 之后。

【高级】架构权衡与实战设计

  1. 你在服务中写了 data = toSignal(this.http.get<User[]>('/api/users'))。(a) 若该服务是 providedIn: 'root'，5 个组件同时注入并在模板中读取 data()，会发起几次 HTTP 请求？若该服务改为组件级 providers，答案是否不同？(b) 现在需要在路由参数（userId）变化时自动重新请求用户数据，请设计一个基于 computed + toSignal 或 toSignal(rxjs.switchMap(...)) 的响应式数据流方案，并分析其与变更检测的集成方式（提示：参数变化如何触发 ReactiveLViewConsumer 脏标记传播）。
