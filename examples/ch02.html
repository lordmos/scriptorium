第2章：从零启动——bootstrapApplication 的旅程

你在 main.ts 里敲下这一行：

// main.ts
bootstrapApplication(AppComponent);

保存文件，浏览器标签页刷新，熟悉的界面出现了。这个过程看似普通——它每天都在发生，在全球无数个 Angular 项目的开发环境里。但如果我们把这几十毫秒放大来看，会发现 Angular 在其中完成了一套精心设计的初始化序列：创建平台单例、组装注入器、运行初始化函数、挂载根组件、触发第一帧渲染。

这中间每一步都有充分的设计理由。这一章，我们跟随这段旅程，看看一行代码背后的完整故事。

调用链全景

在深入细节之前，先建立全局视图。bootstrapApplication 的调用链跨越了三个不同的源文件，最终抵达 ApplicationRef.tick() 完成第一帧渲染：

sequenceDiagram participant main as "main.ts" participant browser as "platform-browser<br/>browser.ts" participant create as "create_application.ts" participant bootstrap as "bootstrap.ts" participant ngZone as "NgZone" participant appInit as "ApplicationInitStatus" participant appRef as "ApplicationRef" main->>browser: bootstrapApplication(AppComponent, config) browser->>create: internalCreateApplication(...) create->>create: "① 创建/重用 PlatformRef" create->>create: "② 组装 appProviders" create->>create: "③ 创建 R3Injector" create->>bootstrap: bootstrap({ r3Injector, rootComponent }) bootstrap->>ngZone: ngZone.run(async () => {...}) ngZone->>ngZone: resolveInjectorInitializers() ngZone->>appInit: runInitializers() appInit-->>ngZone: "await donePromise（等待 APP_INITIALIZER）" ngZone->>appRef: bootstrap(AppComponent) appRef->>appRef: "ComponentFactory.create()" appRef->>appRef: attachView(compRef.hostView) appRef->>appRef: "tick() ✨ 第一帧渲染" bootstrap-->>main: "Promise<ApplicationRef>"

这条链路上有五个值得深讲的节点，我们逐一拆解。

节点一：入口薄层——bootstrapApplication

bootstrapApplication 的公开实现非常简洁，它只是一个薄薄的适配层：

// packages/platform-browser/src/browser.ts（第125-140行）
export async function bootstrapApplication(
  rootComponent: Type<unknown>,
  options?: ApplicationConfig,
  context?: BootstrapContext,
): Promise<ApplicationRef> {
  // 将用户配置与浏览器平台内置 providers 合并
  return internalCreateApplication({
    rootComponent,
    appProviders: [...(options?.providers ?? [])],
    platformProviders: BROWSER_MODULE_PROVIDERS,
  });
}

BROWSER_MODULE_PROVIDERS 包含了一批浏览器平台必需的 provider，如 DomRendererFactory2（负责操作 DOM 的渲染工厂）和一批浏览器特有的服务。这些 provider 对用户透明，但没有它们，Angular 就无法在浏览器环境中正常运行。值得注意的是，bootstrapApplication 本身返回的是 Promise<ApplicationRef>——整个启动过程是异步的，这让 APP_INITIALIZER 中的异步初始化成为可能。

节点二：三步建立应用容器

internalCreateApplication 完成了应用启动最关键的基础工作——建立注入器体系：

// packages/core/src/application/create_application.ts（第38-91行）
export function internalCreateApplication(config: {
  rootComponent?: Type<unknown>;
  appProviders?: Array<Provider | EnvironmentProviders>;
  platformProviders?: Array<Provider | EnvironmentProviders>;
}): Promise<ApplicationRef> {
  // ① 获取或创建平台级单例（整个浏览器 tab 共享）
  const platformInjector = createOrReusePlatformInjector(
    config.platformProviders as StaticProvider[],
  );

  // ② 在平台注入器之上，创建应用级 EnvironmentInjector
  //    用户提供的所有 providers 都注册在这一层
  const injector = createEnvironmentInjector(
    config.appProviders ?? [],
    platformInjector as EnvironmentInjector,
  ) as R3Injector;

  // ③ 使用这个注入器启动应用
  return bootstrap({ r3Injector: injector, rootComponent: config.rootComponent });
}

PlatformRef 是整个注入器层级的根，生命周期比任何 Angular 应用都长。即使调用 applicationRef.destroy() 销毁了应用，PlatformRef 仍然存在，可以用来启动新的应用实例。这个设计主要服务于微前端场景——在同一个页面上运行多个 Angular 应用时，它们共用同一个平台层，避免重复初始化。

在 PlatformRef 之上，createEnvironmentInjector 创建了应用级 R3Injector。你在 app.config.ts 里提供的所有 provider（provideRouter、provideHttpClient 等），以及框架内置的 ApplicationRef、ApplicationInitStatus、NgZone 等服务，都注册在这一层。

🔗 注入器的两层结构（EnvironmentInjector 与 NodeInjector）以及 token 的解析算法，将在第3章详细展开。

节点三：NgZone 的包裹

最终的启动逻辑在 bootstrap() 函数中执行，所有后续工作都在 ngZone.run() 的包裹下运行：

// packages/core/src/platform/bootstrap.ts（第75-181行）
async function bootstrap(options: {
  r3Injector: R3Injector;
  rootComponent?: Type<unknown>;
}): Promise<ApplicationRef> {
  const { r3Injector, rootComponent } = options;
  const ngZone = r3Injector.get(NgZone);

  return ngZone.run(async () => {
    // 触发通过 ENVIRONMENT_INITIALIZER 注册的同步初始化函数
    r3Injector.resolveInjectorInitializers();

    const appRef = r3Injector.get(ApplicationRef);
    const initStatus = r3Injector.get(ApplicationInitStatus);

    // 运行 APP_INITIALIZER，等待所有异步初始化器完成
    initStatus.runInitializers();
    await initStatus.donePromise;  // ← 白屏时间的来源

    setLocaleId(r3Injector.get(LOCALE_ID, DEFAULT_LOCALE_ID));

    if (rootComponent !== undefined) {
      appRef.bootstrap(rootComponent);  // 挂载根组件，触发第一次 tick()
    }
    return appRef;
  });
}

ngZone.run() 保证所有代码在 Angular Zone 内执行，让 Zone.js 能够追踪这期间发起的异步操作。在 Zoneless 模式下，NgZone 被替换为 NoopNgZone（一个直接调用传入函数的占位实现），接口一致，行为不同。这个替换发生在应用启动阶段的依赖注入配置中，由 provideExperimentalZonelessChangeDetection() 在注入器层面完成切换。

💡 提示：resolveInjectorInitializers() 触发的是通过 ENVIRONMENT_INITIALIZER token 注册的初始化函数——这是模块级初始化的底层机制，与 APP_INITIALIZER 不同，它不支持异步，不会阻塞 UI 挂载，主要用于注入器首次使用时执行某些同步副作用。

理解了完整的调用链，我们深入最核心的参与者：ApplicationRef。

ApplicationRef——应用的心脏

如果把 Angular 应用比作一个生命体，ApplicationRef 就是它的心脏：它维持着应用的"生命体征"（视图管理），决定何时跳动（tick() 的调度触发），以及每次跳动要完成什么工作（脏标志位系统）。

ApplicationRef.tick() 是这颗心脏的每一次跳动——它驱动整个变更检测流程。但在 Angular 的实现中，tick() 并不是一个简单的"检查所有组件"，而是一套精确的分级机制。

脏标志位系统：精确控制每次心跳

ApplicationRef 用一组位掩码（bitmask）记录"这次心跳需要做什么工作"，这是整个变更检测系统的核心调度数据结构：

// packages/core/src/application/application_ref.ts
export const enum ApplicationRefDirtyFlags {
  None              = 0b00000000,  // 无需操作，应用处于稳定状态
  ViewTreeGlobal    = 0b00000001,  // Zone.js 触发，需扫描整棵视图树
  ViewTreeTraversal = 0b00000010,  // 需遍历子树（如 Signals 依赖的下游节点）
  ViewTreeCheck     = 0b00000100,  // 有被 markForCheck() 精确标记的视图
  ViewTreeAny = ViewTreeGlobal | ViewTreeTraversal | ViewTreeCheck,  // 任意视图树变更（0b0000111）
  AfterRender       = 0b00001000,  // 有 afterRender / afterNextRender 钩子
  RootEffects       = 0b00010000,  // 有 Signals effect 待执行
}

每个标志对应一类不同来源的变更请求：

  • ViewTreeGlobal：粗粒度变更——Zone.js 触发的全树扫描，等同于"告知所有视图都需要检查"
  • ViewTreeTraversal：中粒度遍历——某条视图分支需要向下遍历，常由 Signals 依赖图变化驱动；区别于 ViewTreeCheck：ViewTreeCheck 是"某组件调用了 markForCheck()，精确标记到叶节点"；ViewTreeTraversal 则是"某个祖先节点的依赖发生变化，需要向下遍历寻找真正的脏节点"（详见第6章视图树遍历算法）
  • ViewTreeCheck：精确变更——某个组件显式调用了 markForCheck()，只遍历被标记的分支
  • ViewTreeAny：上述三种视图树标志的组合掩码，源码中用于一次性判断"是否有任何视图树工作需要处理"
  • AfterRender：由 afterRender() / afterNextRender() API 产生，在每帧渲染结束后执行
  • RootEffects：由 Signals 系统的 effect() 驱动，在变更检测前优先执行，确保 signal 副作用不影响检测结果（effect() 是 Signals API，第7章详述；此处只需理解"某些副作用在变更检测之前优先执行"即可）

位掩码的好处在于：多个标志可以在一次位运算中同时设置和读取。一次 Zone.js 全局触发时，dirtyFlags 可能同时包含 ViewTreeGlobal | AfterRender；一次 signal 更新可能同时产生 RootEffects | ViewTreeCheck。

🔍 ViewTreeGlobal 与 OnPush 策略的关系：ViewTreeCheck 对应"被 markForCheck() 显式标记的视图"，变更检测遍历时会跳过未标记的 OnPush 子树。但 ViewTreeGlobal 会绕过这个保护——Zone.js 触发的全树扫描下，OnPush 优化效果大打折扣。这是第6章变更检测性能优化的一个核心知识点。

tick() 与 synchronize()：稳定的双重保证

tick() 的实现看起来简单，内部却有精巧的稳定性机制：

// packages/core/src/application/application_ref.ts
export class ApplicationRef {
  tick(): void {
    this.synchronize();
  }

  private synchronize(): void {
    // 执行一轮完整的同步
    this.synchronizeOnce();
    // 检测过程中可能产生新的变更（如 effect 触发了 signal 更新）
    // 循环执行，直到应用进入稳定状态
    while (this.dirtyFlags !== ApplicationRefDirtyFlags.None) {
      this.synchronizeOnce();
    }
  }
}

synchronize() 中的 while 循环是一个"最终一致性"保证。设想这个场景：一个 effect() 在执行时更新了某个 signal，这个 signal 变化会新增 ViewTreeCheck 标志。如果只执行一次 synchronizeOnce()，这个新产生的变更就会被遗漏，留到下一帧才更新，造成视觉撕裂。while 循环确保当前帧内的所有"涟漪"都被消化完毕。

synchronizeOnce() 按固定顺序处理三类工作：

// packages/core/src/application/application_ref.ts（简化示意）
private synchronizeOnce(): void {
  // 步骤一：执行所有待处理的 Signals 根 effect（优先于检测执行）
  if (this.dirtyFlags & ApplicationRefDirtyFlags.RootEffects) {
    this.dirtyFlags &= ~ApplicationRefDirtyFlags.RootEffects;
    this.rootEffectScheduler.flush();  // 冲刷根 effect 队列
  }

  // 步骤二：变更检测——按脏标志粒度遍历视图树
  if (this.dirtyFlags & ApplicationRefDirtyFlags.ViewTreeAny) {
    this.dirtyFlags &= ~ApplicationRefDirtyFlags.ViewTreeAny;
    for (const view of this._views) {
      detectChangesInternal(view, /* detectMode 由 dirtyFlags 决定 */);
    }
  }

  // 步骤三：执行 afterRender 钩子（DOM 更新完成后才执行）
  if (this.dirtyFlags & ApplicationRefDirtyFlags.AfterRender) {
    this.dirtyFlags &= ~ApplicationRefDirtyFlags.AfterRender;
    this.runAfterRender();
  }
}

⚠️ 注意：Angular 对 synchronize() 的循环有内置的最大迭代次数保护。在开发模式下，如果你写出了"A 变化导致 B 变化，B 变化又导致 A 变化"的逻辑，Angular 会抛出 ExpressionChangedAfterItHasBeenCheckedError，帮助你在开发阶段尽早发现这类问题。

视图管理：_views 是遍历的根

// packages/core/src/application/application_ref.ts
export class ApplicationRef {
  _views: InternalViewRef[] = [];   // 变更检测的起始点列表
  components: ComponentRef<any>[] = [];  // 引导的根组件

  // appRef.bootstrap(AppComponent) 调用后，根视图通过此方法注入
  attachView(viewRef: ViewRef): void {
    const view = (viewRef as InternalViewRef).internalView;
    this._views.push(view);  // 加入变更检测遍历列表
  }

  detachView(viewRef: ViewRef): void {
    const view = (viewRef as InternalViewRef).internalView;
    remove(this._views, view);  // 从遍历列表移除，后续 tick() 不再检测此视图
  }
}

_views 是 tick() 遍历的入口。detectChangesInternal 从这些根视图出发，按深度优先顺序遍历整棵视图子树，检查每个节点的绑定值是否发生变化。通常一个应用只有一个根视图（AppComponent 的宿主视图），但如果你通过 ApplicationRef.attachView() 手动附加了额外的 ViewRef（如动态创建的组件），它们也会出现在这个数组里。

心脏的工作机制清楚了，下一个问题是：谁来告诉心脏"该跳了"？

两种心跳节拍——Zone.js 与 Zoneless

tick() 不会自动调用自己，它需要外部机制驱动。Angular 支持两种截然不同的心跳触发策略，代表着两种对"如何感知变更"的不同哲学。

Zone.js 模式：剧场计时广播系统

Zone.js 是 Angular 的传统心跳驱动机制。回到第1章建立的剧场比喻：Zone.js 是剧场的计时广播系统——每当后台有人喊"好了！"（异步操作完成），广播系统自动通知舞台监督（变更检测器）去巡场。

Zone.js 的核心能力是 monkey-patch：在应用启动前，它替换掉浏览器原生的 setTimeout、Promise.then、addEventListener 等 API，将这些操作包裹进自己管理的"Zone"中，从而感知每一个异步操作的开始与结束。

Angular 通过 NgZone 封装 Zone.js，并注册了两个关键钩子：

// packages/core/src/zone/ng_zone.ts
class NgZone {
  private forkInnerZoneWithAngularBehavior(zone: ZoneType): void {
    // fork() 创建一个子 Zone，继承父 Zone，但注入自定义钩子
    zone._inner = zone._inner.fork({
      name: 'angular',

      // 钩子一：每当一个异步任务被执行时触发（setTimeout 回调、Promise.then 等）
      onInvokeTask: (delegate, curr, target, task, applyThis, applyArgs) => {
        try {
          return delegate.invokeTask(target, task, applyThis, applyArgs);
        } finally {
          checkStable(zone);  // 任务结束后检查稳定性
        }
      },

      // 钩子二："有任务/无任务"状态切换时触发（任务队列变空或新增任务）
      onHasTask: (delegate, curr, target, hasTaskState) => {
        delegate.hasTask(target, hasTaskState);
        checkStable(zone);  // 状态切换后检查稳定性
      },
    });
  }
}

checkStable 在判断"所有微任务队列已空且无待处理宏任务"时，触发 onMicrotaskEmpty 事件，进而启动变更检测调度链：

flowchart LR A["异步操作完成<br/>setTimeout / Promise / XHR"] --> B["Zone.js 钩子触发<br/>onHasTask"] --> C["checkStable()"] --> D["onMicrotaskEmpty.emit()"] --> E["ChangeDetectionSchedulerImpl<br/>.notify(ViewTreeGlobal)"] --> F["scheduleCallbackWithRafRace()<br/>rAF 对齐下一帧"] --> G["ApplicationRef.tick()"]

这条链路的含义很明确：任何在 Angular Zone 内发生的异步操作——无论是你的 HTTP 请求、第三方库内部的定时器，还是用户点击事件的回调——完成后都会驱动一次变更检测。 Zone.js 不需要你做任何配合，广播系统自动运作。

💡 runOutsideAngular 优化：Zone.js 模式下，如果你有一个高频的 setInterval（如动画帧、性能计数器），把它放进 ngZone.runOutsideAngular() 可以让它的回调运行在 Angular Zone 之外，完成后不触发变更检测，显著减少无意义的检测次数。

Zoneless 模式：精准信号驱动

Zoneless 模式（Angular v18+ 正式稳定）移除了 Zone.js，改用明确的"举手示意"机制。变更检测完全由 ChangeDetectionSchedulerImpl 统一调度——任何需要触发检测的操作，都必须显式通知它：

// packages/core/src/change_detection/scheduling/zoneless_scheduling_impl.ts
class ChangeDetectionSchedulerImpl implements ChangeDetectionScheduler {

  notify(source: NotificationSource): void {
    // 将通知来源映射为对应的脏标志，精确记录"哪类变更发生了"
    this.appRef.dirtyFlags |= sourceToDirtyFlag(source);

    if (this.shouldScheduleTick()) {
      // 选择调度策略：微任务（测试/SSR）或 rAF 竞速（生产环境）
      const schedFn = this.useMicrotaskScheduler
        ? queueMicrotask
        : scheduleCallbackWithRafRace;

      // 调度函数在 Zone 外执行，避免在 Zone.js 共存场景中重复触发
      this.cancelScheduledCallback = this.ngZone.runOutsideAngular(
        () => schedFn(() => this.tick()),
      );
    }
  }

  private tick(): void {
    // 只有 dirtyFlags 不为零时才真正执行 tick()
    if (this.appRef.dirtyFlags !== ApplicationRefDirtyFlags.None) {
      this.ngZone.run(() => this.appRef._tick());
    }
  }
}

谁会调用 notify()？主要有四个来源：

flowchart TD A["ChangeDetectorRef<br/>.markForCheck()"] --> E B["signal.set() /<br/>signal.update()"] --> E C["Input 绑定更新<br/>（父组件传入新值）"] --> E D["手动 ApplicationRef<br/>.tick()"] --> E E["ChangeDetectionSchedulerImpl<br/>.notify(source)"] --> F["dirtyFlags |= 对应标志"] --> G{"shouldScheduleTick()?<br/>是否已有待执行 tick?"} G -->|"是，调度"| H["queueMicrotask / rAF<br/>注册下一次 tick()"] G -->|"否，去重"| I["只更新 dirtyFlags，跳过调度"] H --> J["ApplicationRef.tick()"]

shouldScheduleTick() 的判断防止重复调度：如果已经有一个 tick() 在排队等待执行，新的通知只需更新 dirtyFlags，不需要再次注册调度。这是一个典型的"去抖动"设计。

心跳机制理解清楚了，但在第一次心跳之前，应用还有一道关卡要过。

APP_INITIALIZER——启动前的准备工作

在 bootstrap() 函数的调用序列中，appRef.bootstrap(rootComponent) 之前有一行关键的等待：

await initStatus.donePromise;

这是 APP_INITIALIZER 机制的核心体现——在根组件挂载之前，允许应用执行一批初始化工作。典型场景包括：从服务端加载运行时配置（API 地址、Feature Flag）、初始化国际化数据、预加载用户权限信息等。

初始化链的工作原理

// packages/core/src/application/application_init.ts
@Injectable({ providedIn: 'root' })
export class ApplicationInitStatus {
  private resolve!: () => void;
  done = false;
  // bootstrap() 等待这个 Promise resolve 后才继续
  readonly donePromise = new Promise<void>((res) => (this.resolve = res));

  runInitializers(): void {
    const promises: Promise<any>[] = [];

    for (const appInits of this.appInits) {
      // 在注入上下文中运行初始化函数（允许在函数体内使用 inject()）
      const initResult = runInInjectionContext(this.injector, appInits);

      if (isPromise(initResult)) {
        promises.push(initResult);                    // async/await 返回 Promise
      } else if (isObservable(initResult)) {
        promises.push(firstValueFrom(initResult));    // RxJS Observable
      }
      // void 返回（同步初始化）直接忽略，不阻塞
    }

    // 所有初始化器并行执行（Promise.all），全部完成后通知 bootstrap 继续
    this.initialized = Promise.all(promises).then(() => {
      this.done = true;
      this.resolve();
    });
  }
}

注册初始化器的方式：

// app.config.ts
export const appConfig: ApplicationConfig = {
  providers: [
    {
      provide: APP_INITIALIZER,
      // 工厂函数返回的函数才是初始化器本身
      useFactory: (configService: AppConfigService) =>
        () => configService.loadRemoteConfig(),
      deps: [AppConfigService],
      multi: true,  // 多个初始化器之间并行执行
    },
  ],
};

多个 APP_INITIALIZER（multi: true）被收集进一个数组，通过 Promise.all 并行等待。三个初始化器同时发起时，总等待时间取决于最慢的那个，而非三者之和——这是一个对启动性能有利的设计。

异步初始化的代价

flowchart LR A["bootstrap() 开始"] --> B["resolveInjectorInitializers()"] --> C["runInitializers()"] --> D{"有 Promise<br/>初始化器?"} D -->|"是"| E["⏳ await donePromise<br/>用户看到白屏"] E --> F["appRef.bootstrap(AppComponent)"] D -->|"否，直接继续"| F F --> G["tick() ✨"] G --> H["首次渲染完成"]

这张图揭示了一个直接的现实：APP_INITIALIZER 返回 Promise 的每一毫秒，都是用户面对空白页面的每一毫秒。

⚠️ APP_INITIALIZER 的成本直接累加到 FCP（首次内容绘制）时间：

情景FCP 估算
不加 APP_INITIALIZER≈ 200ms（正常 JS 解析 + 首帧渲染）
加 1 个 3 秒 API 请求的初始化器≈ 3,200ms
加 2 个并行初始化器（3s + 2.5s）≈ 3,200ms（Promise.all 取最慢，仍是 3s 级延迟）

在移动网络（3G）上，FCP 超过 3 秒时用户流失率显著上升（Web Vitals 核心指标）。

⚠️ 使用 APP_INITIALIZER 的三条原则

  1. 真正阻塞才使用：只有"没有这个数据，应用根本无法正确渲染"时才在此加载（如全局权限 token、主题颜色配置）；能做成路由级 Resolver 的，坚决不放这里
  2. 设置超时降级：网络请求必须有超时保护，失败时使用默认值降级，而不是让应用卡在白屏
  3. 能延迟就延迟：业务数据、用户偏好等完全可以在组件 ngOnInit 或路由 Resolver 中按需加载
// 带超时保护的初始化器示例
{
  provide: APP_INITIALIZER,
  useFactory: (configService: AppConfigService) => () =>
    Promise.race([
      configService.loadRemoteConfig(),
      // 3 秒后自动使用默认配置，不阻塞启动
      new Promise<void>((resolve) => setTimeout(resolve, 3000)),
    ]),
  deps: [AppConfigService],
  multi: true,
}

理解了启动流程和两种调度模式，我们可以从更宏观的角度，思考一个大版本迭代背后的设计决策。

设计思考：Zone.js 的历史使命与终结

Zone.js 在 2014 年是一个极具创意的工程方案。Angular 团队当时面临一个真实问题：如何让框架感知到任意异步操作的完成？在 Signals 尚未诞生、RxJS 无缝集成尚在探索的年代，Zone.js 的"全局拦截"策略是一个优雅的答案——开发者不需要做任何事情，框架自动处理一切。

然而，随着 Angular 使用规模扩大，Zone.js 的代价也越来越清晰。

根本性问题一：粒度失控

Zone.js 无法区分"哪些异步操作会影响 UI"。一个纯粹用于日志上报的 setTimeout、一个埋点统计的 fetch，都会触发完整的全树变更检测。开发者不得不反复学习 runOutsideAngular、NgZone.runTask 这些优化手段——本质上是在为框架设计缺陷打补丁。

根本性问题二：全局副作用

Zone.js 的 monkey-patch 是全局性的。它替换的是 window.setTimeout、全局 Promise 等对象，影响页面上所有代码，不只是 Angular 应用本身。在部分 Web 性能测试工具（如 Lighthouse 的特定检测项）、依赖原生 Promise 微任务顺序的第三方库中，Zone.js 会造成干扰，产生难以复现的 Bug。

根本性问题三：标准追赶压力

Web 平台每年都有新 API 出现。每次有新的异步机制（ResizeObserver、scheduler.postTask、AbortController 等）被引入，Angular 团队都需要评估是否为其添加 Zone.js patch。这是一项持续的维护负担，也是"用外挂库解决框架问题"这条路的天花板。

Signals 如何从根本上解决这些问题

Zoneless 模式本身只是"不用 Zone.js"，但它真正发挥威力，需要与 Signals 配合使用：

  • signal 的读写是同步且精确的：signal.set() 直接通知所有依赖它的视图，不需要等任何异步操作完成
  • computed 是惰性求值的：它的值只在被读取时才重算，不存在"无意义的中间状态"
  • signal 的变更通知是精确追踪的：只有真正依赖了某个 signal 的视图，才会在它变化时被标记为脏

当状态由 Signals 管理时，变更检测自然变成"精确的、按需的"。用之前的比喻来说：Zone.js 模式是"大楼里有任何访客按门铃，广播通知所有住户检查自己的门"；Zoneless + Signals 模式是"每套门铃都只连接自己对应的房间，只通知真正有访客的那一户"。

对于新项目，Angular 团队的推荐很明确：Zoneless + Signals 是 v18+ 的首选组合。对于遗留项目，Zone.js 模式仍然有效，Angular 会长期维护两种模式，迁移可以循序渐进。

🔗 Signals 的依赖追踪实现（ReactiveNode、Consumer/Producer 图、computed 的惰性求值机制）将在第7章深入分析。detectChangesInternal 的视图树遍历算法，以及 OnPush 策略与脏标志位的完整交互，将在第6章展开。

小结

我们追踪了 bootstrapApplication(AppComponent) 的完整旅程，覆盖了启动过程中每一个关键决策点：

  • 调用链：bootstrapApplication → internalCreateApplication（创建 PlatformRef + R3Injector）→ bootstrap()（NgZone 包裹 → APP_INITIALIZER 等待 → appRef.bootstrap → tick()）→ 第一帧渲染
  • ApplicationRef：脏标志位系统（ApplicationRefDirtyFlags）是变更检测的精确调度开关；synchronize() 的 while 循环确保应用在每次 tick() 后最终达到稳定状态；_views 数组是所有变更检测的遍历起点
  • Zone.js 心跳：monkey-patch 全局异步 API，任何异步操作完成后自动触发 ViewTreeGlobal 全树扫描；便利但粒度粗，存在过度检测的隐患
  • Zoneless 心跳：ChangeDetectionSchedulerImpl.notify() 接收精确的变更通知，只设置对应的脏标志位，按需调度 tick()；与 Signals 配合是 v18+ 的推荐模式
  • APP_INITIALIZER：在根组件挂载前运行初始化链；每个返回 Promise 的初始化器都直接延长 FCP 时间，使用需要权衡必要性

两种调度模式并无绝对优劣：Zone.js 提供开箱即用的便利，适合遗留项目和快速原型；Zoneless + Signals 提供更精确的控制与更小的包体（约少 10 KB gzip），是 v18+ 新项目的首选。这一选择直接影响第6章变更检测性能策略的适用范围——Zoneless 模式下 OnPush 的优化收益会放大，而许多 Zone.js 时代的 runOutsideAngular 补丁将不再必要。
