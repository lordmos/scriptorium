第4章：模板编译器——从 HTML 到指令序列

<div [class]="active">{{title}}</div>

这行你每天都在写的 HTML，Angular 看到它的第一反应不是"创建一个 div"——而是把它翻译成这样一段代码：

// 编译器生成，非手写
function MyComponent_Template(rf, ctx) {
  if (rf & 1) {
    ɵɵelementStart(0, "div");
    ɵɵtext(1);
    ɵɵelementEnd();
  }
  if (rf & 2) {
    ɵɵclassMap(ctx.active);
    ɵɵadvance(1);
    ɵɵtextInterpolate(ctx.title);
  }
}

这个翻译过程在你运行 ng build 时就已完成。浏览器打开页面时，它看到的不是那行 HTML，而是这段精确的函数调用序列。Angular 是一个 AOT（Ahead-of-Time，预编译）框架——模板在构建时就被彻底"消化"，运行时不再解析任何 HTML 字符串。

但"翻译"这个词掩盖了背后的复杂性。从一行 HTML 到一组 ɵɵ 前缀的函数调用，中间经历了词法分析、语法分析、语义转换、中间表示、优化流水线、代码生成——六道工序，每一道都有专属的数据结构和算法。

本章的目标，就是陪你走完这六道工序。

编译器全景——五层流水线

把模板比作一首用五线谱写就的钢琴曲，Angular 编译器的工作就是把它翻译成钢琴指法谱。演奏者（render3 运行时）只需按指法谱弹，不再需要五线谱——这就是"乐谱翻译"比喻的核心：编译时翻译，运行时只执行。

翻译过程分为五层：

flowchart TD A["HTML 模板字符串<br/>&lt;div [class]=&quot;active&quot;&gt;{{title}}&lt;/div&gt;"] B["ML AST<br/>html.Element / html.Text / html.Attribute<br/>packages/compiler/src/ml_parser/"] C["Template AST<br/>t.Element / BoundAttribute / BoundText<br/>packages/compiler/src/render3/r3_ast.ts"] D["IR（中间表示）<br/>OpList&lt;CreateOp&gt; + OpList&lt;UpdateOp&gt;<br/>packages/compiler/src/template/pipeline/"] E["IR 优化阶段（68 个 Phase）<br/>slot 分配 → vars 计数 → advance 生成 → reify"] F["输出：template 函数<br/>function MyComponent_Template(rf, ctx) { ... }"] A -->|"tokenize() + _TreeBuilder.build()"| B B -->|"htmlAstToRender3Ast()"| C C -->|"ingestComponent() / ingestNodes()"| D D -->|"transform() — 68 个处理阶段"| E E -->|"emitView() + emitTemplateFn()"| F

各层的职责一句话概括：

层名称职责
1ML AST词法 + 语法分析，产出通用 HTML 树，不理解 Angular 语义
2Template AST语义转换，把 [class]="x" 识别为绑定，输出 Ivy 专属节点
3IR中间表示，将 Template AST 转为操作序列，与具体指令解耦
4优化阶段68 个独立 Phase 依次处理：分配 slot、统计 vars、生成 advance…
5输出从优化后的 IR 生成最终 TypeScript 函数

🔗 交叉引用：第1章已经介绍了 @angular/compiler（模板编译 API）、@angular/compiler-cli（ngtsc，AOT 编译 CLI 插件）与 @angular/core 三者的分工关系。本章深入 @angular/compiler 的内部。

接下来，我们逐层拆解。

ML AST——HTML 的骨架

为什么 Angular 要自研 HTML 解析器

你可能会问：浏览器本身有 HTML 解析能力，Angular 为什么要自己造一个？

原因有三。

第一，Angular 在构建阶段运行，不在浏览器里。 ng build 在 Node.js 环境里执行，那里没有浏览器 DOM API，没有 document.createElement，没有 innerHTML 可以借用。

第二，浏览器的 HTML 解析会"自动修正"错误，这对编译器是有害的。 浏览器遇到 <div><p></div> 会悄悄补全闭合标签，开发者不知道自己写了有问题的模板。Angular 编译器需要严格报错，不需要这种容错性。

第三，Angular 模板里有浏览器不认识的语法。 @if (condition) { ... }、@for (item of items; track item.id) { ... } 这类控制流语法是 Angular 专有的，浏览器解析器遇到它们会直接跳过，而 Angular 编译器需要把它们当作一等公民处理。

Angular 自研的解析器定义在 packages/compiler/src/ml_parser/ 目录下，"ML"代表 Markup Language（标记语言），强调这一层是语言无关的——它同样可以解析 SVG、MathML 等标记语言。

词法分析：把字符串切成 Token

解析的第一步是词法分析（Lexer）。tokenize() 函数（ml_parser/lexer.ts:109）把 HTML 字符串切割成一个 Token 流。

对于 <div [class]="active">{{title}}</div>，词法分析的输出大致是：

TAG_OPEN_START   → "<div"
ATTR_NAME        → "[class]"
ATTR_VALUE_TEXT  → "active"
TAG_OPEN_END     → ">"
INTERPOLATION    → "{{title}}"
TAG_CLOSE        → "</div>"
EOF

TokenType 枚举（ml_parser/tokens.ts:11）共定义了 40 个类型，覆盖标签开闭、属性名值、插值表达式、HTML 注释、CDATA、以及 Angular v17+ 的 @block 控制流语法等。此时词法分析只是机械地切割字符，并不理解 [class] 这个方括号有什么 Angular 含义。

语法分析：Token 流 → HTML 树

词法之后是语法分析。_TreeBuilder.build() 方法（ml_parser/parser.ts:86）按 Token 类型分派处理，把扁平的 Token 序列重建为树形结构。

对于示例模板，最终得到的 ML AST 是（伪 JSON 风格，省略位置信息）：

[
  {
    "type": "Element",
    "name": "div",
    "attrs": [
      { "type": "Attribute", "name": "[class]", "value": "active" }
    ],
    "children": [
      {
        "type": "Text",
        "value": "{{title}}",
        "tokens": [{ "type": "INTERPOLATION", "parts": ["{{", "title", "}}"] }]
      }
    ]
  }
]

注意此刻 ML AST 仍然对 Angular 一无所知：[class]="active" 只是一个名字恰好带方括号的普通属性，{{title}} 是一段含有插值 Token 的文本节点。意义的赋予发生在下一层。

ML AST 的节点类型定义于 ml_parser/ast.ts:100（Element 类），除了 Element、Text、Attribute、Comment 这些经典 HTML 节点，还包括 Block（@if/@for 控制流块）、Component/Directive（Angular v19 引入的 Selectorless 语法）等现代扩展。

💡 提示：ML AST 不仅服务于模板编译，Angular 语言服务（Language Service）、i18n 消息提取工具都在这一层工作，这也是 Angular 将解析层单独抽象的原因——解析一次，多个工具复用。

到此为止，我们手里有了一棵标准的 HTML 语法树。接下来，编译器要赋予它 Angular 的"灵魂"。

Template AST——Angular 的语义层

从语法到语义的跨越

ML AST 是一棵"语法树"，而 Template AST 是一棵"语义树"。两者的核心区别在于：Template AST 知道什么是数据绑定、什么是事件处理器、什么是模板引用变量——这些都是 Angular 层面的概念，HTML 标准里没有。

转换的入口函数是 htmlAstToRender3Ast()（render3/r3_template_transform.ts:91）：

// packages/compiler/src/render3/r3_template_transform.ts 第 91-113 行
export function htmlAstToRender3Ast(
  htmlNodes: html.Node[],
  bindingParser: BindingParser,
  options: Render3ParseOptions,
): Render3ParseResult {
  const transformer = new HtmlAstToIvyAst(bindingParser, options);
  const ivyNodes = html.visitAll(transformer, htmlNodes, htmlNodes);
  // 合并 bindingParser 的错误（表达式解析错误）和 transformer 的错误（语义错误）
  const allErrors = bindingParser.errors.concat(transformer.errors);
  return { nodes: ivyNodes, errors: allErrors, ... };
}

HtmlAstToIvyAst 是核心访问者类，它实现了 html.Visitor 接口，对 ML AST 的每种节点类型都有对应的 visitXxx() 方法。html.visitAll() 驱动遍历，访问者在过程中将每个 ML 节点"翻译"为一个或多个 Template AST 节点。

属性的语义解析

Template AST 转换最精彩的部分发生在 visitElement() 方法里（r3_template_transform.ts:136）。当遇到一个 ML Element 节点时，访问者会对它的所有属性调用 prepareAttributes()，将它们分类成四种：

  • TextAttribute：纯文本属性，如 id="foo"、class="btn"
  • BoundAttribute：绑定属性，如 [class]="active"、[style.color]="color"
  • BoundEvent：事件绑定，如 (click)="handler()"
  • Variable / Reference：模板变量 let-x / 模板引用 #ref

分类的依据很简单：方括号 [xxx] 是属性绑定，圆括号 (xxx) 是事件绑定，[(...)] 是双向绑定——这些规则由 BindingParser 负责执行。

Template AST 的关键节点

对于 <div [class]="active">{{title}}</div>，转换后的 Template AST 如下：

[
  {
    "type": "t.Element",
    "name": "div",
    "attributes": [],
    "inputs": [
      {
        "type": "BoundAttribute",
        "name": "class",
        "bindingType": "BindingType.Class",
        "value": {
          "ast": "PropertyRead { name: 'active', receiver: ImplicitReceiver }"
        }
      }
    ],
    "outputs": [],
    "children": [
      {
        "type": "BoundText",
        "value": {
          "ast": "Interpolation { strings: ['', ''], expressions: [PropertyRead('title')] }"
        }
      }
    ]
  }
]

与 ML AST 相比，发生了三件事：

  1. [class]="active" 从一个普通 Attribute 变成了 BoundAttribute，其中 bindingType 是 BindingType.Class（区别于属性绑定 BindingType.Property），value 字段是已经解析好的表达式 AST——PropertyRead { name: 'active' }
  2. 文本节点 {{title}} 从 Text 变成了 BoundText，其 value 字段是 Interpolation 表达式 AST，包含插值字符串数组和表达式数组
  3. div 元素的 inputs 数组收纳了所有绑定属性，与静态 attributes 分离

Template AST 完整的节点类型定义于 render3/r3_ast.ts。除了上面出现的 Element、BoundAttribute、BoundText，还有对应 <ng-template> 和结构型指令宿主的 Template 节点，以及 Angular v17+ 控制流的 IfBlock、ForLoopBlock、DeferredBlock 等。

⚠️ 注意：BindingType.Class（不带具体类名）对应 [class]="expr" 整体赋值形式，最终生成 ɵɵclassMap()；而 [class.active]="flag" 是 BindingType.Class 带具体类名，最终生成 ɵɵclassProp("active", flag)。两者的 vars 消耗量不同。

Template AST 已经"知道"Angular 语义了。但如果直接从 Template AST 生成最终代码，优化空间会很受限。这就引出了编译流水线中最重要的一层——IR。

IR——优化的中间层

为什么需要中间层

为什么不从 Template AST 直接生成 JavaScript？因为代码生成不只是"翻译"，还需要"优化"。

想象一个更复杂的模板，里面有十几个绑定、几个结构型指令、嵌套的 @if 和 @for。如果直接从 Template AST 生成代码，你必须在一次遍历里同时计算 slot 分配、变量计数、上下文引用、advance 插入……这会让代码生成逻辑复杂到难以维护，更无法独立测试每一类优化。

IR（Intermediate Representation，中间表示）把这些关切彻底分离：每个优化是一个独立的 Phase 函数，输入一组 IR 操作，输出修改后的 IR 操作，可以独立测试，可以随时在任意两个 Phase 之间断点检查状态。Angular 的编译器里这样的 Phase 共有 68 个。

这个思路和传统编译器（如 LLVM）的 IR 设计如出一辙：把"理解程序"和"优化程序"的工作彻底分离。Angular 只是把这个思想应用在了模板编译上。

OpList：双向链表上的操作序列

IR 的核心数据结构是 OpList，定义于 template/pipeline/ir/src/operations.ts:55：

// packages/compiler/src/template/pipeline/ir/src/operations.ts 第 55-80 行
export class OpList<OpT extends Op<OpT>> {
  readonly head: OpT;   // 哨兵头节点（kind=ListEnd）
  readonly tail: OpT;   // 哨兵尾节点（kind=ListEnd）

  push(op: OpT | Array<OpT>): void { ... }
  prepend(ops: OpT[]): void { ... }
  *[Symbol.iterator](): Generator<OpT> { ... }

  static insertBefore<T extends Op<T>>(op: T, target: T): void { ... }
  static replace<T extends Op<T>>(op: T, newOp: T | T[]): void { ... }
  static remove<T extends Op<T>>(op: T): void { ... }
}

为什么是双向链表而不是数组？因为 Phase 处理时最频繁的操作是"在特定 Op 前后插入新 Op"——比如 generateAdvance Phase 需要在每个需要推进游标的 Update Op 前面插入一个 AdvanceOp。链表的这种插入是 O(1)，数组则是 O(n)。处理数千个 Op 时，这个差距不可忽视。

每个视图（ViewCompilationUnit）有两条独立的 OpList（compilation.ts:165-170）：

// packages/compiler/src/template/pipeline/src/compilation.ts 第 165-170 行
readonly create = new ir.OpList<ir.CreateOp>();  // 创建阶段操作
readonly update = new ir.OpList<ir.UpdateOp>();  // 更新阶段操作

Create 链表 对应 template 函数里的 if (rf & 1) 块，记录"如何构建 DOM 结构"的操作序列——创建元素、文本节点、注册事件监听、声明嵌入视图等，首次渲染时执行一次，此后不再执行。

Update 链表 对应 if (rf & 2) 块，记录"如何更新数据绑定"的操作序列——属性更新、插值刷新、条件判断等，每次变更检测时都可能执行。

这个"创建/更新双轨"设计是 Angular 高效的关键：首次渲染之后，框架只需执行 Update 序列，完全跳过 DOM 构建逻辑。

ingest：从 Template AST 到 IR

将 Template AST"喂入"IR 的过程称为 ingest，入口是 ingestComponent()（template/pipeline/src/ingest.ts:55），它递归遍历 Template AST 节点，对每种节点类型调用对应的 ingestXxx() 函数，向 Create/Update 链表追加相应的 IR 操作。

对于 <div [class]="active">{{title}}</div>，ingest 后 Create 链表的内容是：

ElementStartOp { xref: XrefId(0), tag: "div", handle: null (待分配) }
TextOp         { xref: XrefId(1), initialValue: "",  handle: null (待分配) }
ElementEndOp   { xref: XrefId(0) }

Update 链表是：

BindingOp      { kind: ClassName,   target: XrefId(0), expr: PropertyRead("active") }
InterpolateTextOp { target: XrefId(1), interpolation: Interpolation(["",""], [PropertyRead("title")]) }

注意 target: XrefId(0) 和 target: XrefId(1) 这两个跨引用。XrefId 是 IR 中的稳定标识符（类型安全的 number & { __brand: 'XrefId' }），Update Op 通过它引用对应的 Create Op。之所以不用直接的 JavaScript 对象引用，是因为 Phase 处理可能会删除或替换 Op 对象——XrefId 就像数据库主键，无论行被替换，键不变。

68 个 Phase 的流水线

ingest 完成后，transform() 函数（emit.ts:181）会依次调用 68 个 Phase 函数对 IR 进行优化和转换。下图展示了最关键的几个 Phase 及其位置：

flowchart LR A["ingest<br/>Template AST → IR 操作序列"] B["Phase 8<br/>specializeStyleBindings<br/>BindingOp → ClassMapOp / StylePropOp"] C["Phase 17<br/>orderOps<br/>调整操作顺序"] D["Phase 42<br/>optimizeVariables<br/>内联/消除冗余临时变量"] E["Phase 49<br/>allocateSlots<br/>分配 slot → unit.decls"] F["Phase 57<br/>countVariables<br/>统计 vars → unit.vars"] G["Phase 58<br/>generateAdvance<br/>插入 ɵɵadvance 调用"] H["Phase 67<br/>reify<br/>IR Op → o.Statement"] I["Phase 68<br/>chain<br/>合并调用链"] J["emitView<br/>生成最终 template 函数"] A --> B --> C --> D --> E --> F --> G --> H --> I --> J

让我们把 68 个 Phase 中对理解编译器最关键的四个拿出来详解。

Phase 49：allocateSlots（slot 分配，决定 decls）

文件：template/pipeline/src/phases/slot_allocation.ts

LView 是一个平坦数组（第5章会详细解剖它）。decls 告诉运行时这个数组的前多少个槽用于存 DOM 节点引用。每个需要占用 LView 槽的 Create Op 都实现了 ConsumesSlotOpTrait 接口。allocateSlots 遍历 Create 链表，按顺序为这些 Op 分配索引：

// packages/compiler/src/template/pipeline/src/phases/slot_allocation.ts 第 28-52 行
let slotCount = 0;
for (const op of unit.create) {
  if (!ir.hasConsumesSlotTrait(op)) continue;
  op.handle.slot = slotCount;       // 分配索引
  slotMap.set(op.xref, op.handle.slot);
  slotCount += op.numSlotsUsed;     // 通常步进 1
}
unit.decls = slotCount;             // 记录总 decls 值

需要占用 slot 的 Create Op 包括：ElementStart、Text、Template（嵌入视图）、Pipe、Projection 等。不占用 slot 的 Create Op 包括：ElementEnd（只是闭合标签，不对应 DOM 节点）、Listener（事件处理器不在数据区，在运行时通过其他机制存储）。

对于示例模板，Phase 49 后：ElementStartOp.handle.slot = 0，TextOp.handle.slot = 1，unit.decls = 2。

🔗 TView 关联：decls = 2 将在代码生成阶段写入 TView.blueprint 数组的长度预算——每个组件类共享一个 TView（运行时元数据蓝图），所有组件实例的 LView 都以此 blueprint 为模板分配内存。slot 分配的结果正是 TView blueprint 中"DOM 节点区"的边界标记（第5章详解）。

Phase 57：countVariables（变量计数，决定 vars）

文件：template/pipeline/src/phases/var_counting.ts

vars 告诉运行时 LView 数组中紧跟 DOM 节点之后，还需要多少个槽用于存储"上次变更检测时的旧值"——这是脏检查的物质基础。每次变更检测时，ɵɵclassMap(ctx.active) 需要把新值和旧值比较，只有不同时才真正更新 DOM，旧值就存在这里。

不同类型的绑定消耗的 vars 数量不同：

绑定类型消耗的 vars原因
property1只需存上次的值
classMap / styleMap2需要存上次的对象引用 + 字符串化后的值
classProp / styleProp2同上
textInterpolate（n 个表达式）n每个表达式各存一个旧值
conditional（@if）1存上次的条件结果
pipe（管道）1 + 参数数量存管道实例 + 每个参数的旧值

🔍 vars 消耗的本质：vars 槽的数量取决于变更检测需要多少"记忆"。property/textInterpolate 只需记住上次的值，1 个 var 够用。classMap/styleMap 需要两层记忆：var[i] 存旧的对象引用（用 === 判断对象身份是否改变），var[i+1] 存旧的字符串化值（对象引用不变但内容变时进行逐属性比较）——因此消耗 2 个 vars。这个设计让运行时既能快速跳过未变化的对象，又能精确检测对象内容变化。

对于示例模板：ClassMapOp 消耗 2 个 vars，InterpolateTextOp（1 个表达式）消耗 1 个 vars，合计 unit.vars = 3。

Phase 58：generateAdvance（生成 ɵɵadvance）

文件：template/pipeline/src/phases/generate_advance.ts

运行时维护一个隐式 slot 游标，初始为 0。Update 链表中的绑定指令（classMap、property、textInterpolate 等）不接受 slot 参数，而是隐式作用于游标当前指向的 slot。ɵɵadvance(n) 将游标向前推进 n 步。

为什么要这样设计？节省代码体积。一个简单的组件模板可能有数十个绑定指令，如果每个指令都需要传 slot 索引，生成的 JavaScript 文件会更大。利用游标，只在位置需要跳跃时插入 advance()，其余情况游标自然递增。

generateAdvance 模拟运行时执行，跟踪游标的当前位置，在需要跳跃的 Update Op 前插入 AdvanceOp：

// packages/compiler/src/template/pipeline/src/phases/generate_advance.ts 第 36-75 行（伪代码示意）
let slotContext = 0;  // 模拟运行时游标
for (const op of unit.update) {
  // 实际源码使用 ir.hasDependsOnSlotContextTrait(op) trait 检查
  // 及 ir.visitExpressionsInOp() 遍历嵌套表达式；此处简化为 findSlotDependency()
  const consumer = findSlotDependency(op);  // 伪代码，代表 DependsOnSlotContext trait 检查
  if (consumer === null) continue;

  const slot = slotMap.get(consumer.target)!;
  if (slotContext !== slot) {
    const delta = slot - slotContext;   // 需要跳跃的步数（只能为正）
    ir.OpList.insertBefore<ir.UpdateOp>(
      ir.createAdvanceOp(delta, consumer.sourceSpan), op
    );
    slotContext = slot;
  }
}

对于示例模板：ClassMapOp 作用于 slot 0（游标初始为 0，不需要 advance），InterpolateTextOp 作用于 slot 1（游标当前在 0，需要 advance(1)）。Phase 58 后，Update 链表变为：

ClassMapOp { target: XrefId(0) }
→ AdvanceOp { delta: 1 }
→ InterpolateTextOp { target: XrefId(1) }

Phase 67：reify（IR 具体化）

文件：template/pipeline/src/phases/reify.ts

reify 是最后一道实质性变换：将每个语义 IR Op 替换为对应的运行时指令调用（o.Statement）。这里"具体化"这个词很准确——之前的 IR 操作是抽象的（"创建一个元素"），reify 之后变成具体的（ɵɵelementStart(0, "div")）。

// packages/compiler/src/template/pipeline/src/phases/reify.ts 第 101 行（简化）
switch (op.kind) {
  case ir.OpKind.ElementStart:
    // IR Op → ɵɵelementStart(slot, tag, attrs, refs)
    ir.OpList.replace(op, ng.elementStart(op.handle.slot!, op.tag!, ...));
    break;
  case ir.OpKind.ClassMap:
    // IR Op → ɵɵclassMap(expr)
    ir.OpList.replace(op, ng.classMap(op.expression, ...));
    break;
  case ir.OpKind.Advance:
    // IR Op → ɵɵadvance(delta)
    ir.OpList.replace(op, ng.advance(op.delta, ...));
    break;
}

Phase 68（chain）在 reify 之后运行，将连续的同类指令调用合并为链式调用（如 ɵɵelementStart(0, "div").text(1).elementEnd() 这样的形式，在 bundle 体积上更紧凑）。

理解了 IR 的设计，代码生成就只剩下最后的"装配"了。

代码生成——最终输出

template 函数的结构

所有 68 个 Phase 执行完毕后，emitView() 函数（emit.ts:219）从 Create 和 Update 链表中提取所有 o.Statement，分别用 if (rf & 1) {...} 和 if (rf & 2) {...} 包裹，组装成最终的 template 函数：

// packages/compiler/src/template/pipeline/src/emit.ts 第 219-256 行（简化）
function emitView(view: ViewCompilationUnit): o.FunctionExpr {
  const createStatements: o.Statement[] = [];
  for (const op of view.create) {
    createStatements.push(op.statement);  // 所有 Op 已被 reify 转为 StatementOp
  }
  const updateStatements: o.Statement[] = [];
  for (const op of view.update) {
    updateStatements.push(op.statement);
  }
  const createCond = maybeGenerateRfBlock(1, createStatements); // if (rf & 1)
  const updateCond = maybeGenerateRfBlock(2, updateStatements); // if (rf & 2)
  return o.fn(
    [new o.FnParam('rf', o.NUMBER_TYPE), new o.FnParam('ctx', o.DYNAMIC_TYPE)],
    [...createCond, ...updateCond],
    undefined, undefined, view.fnName,
  );
}

rf 是 RenderFlags 的简写，是一个位掩码：1 代表 Create 模式（首次渲染），2 代表 Update 模式（变更检测）。ctx 是组件实例（this 的替代——函数里没有 this，通过参数传入）。

decls 和 vars：LView 的"预算表"

decls 和 vars 经常让初学者困惑。用一个比喻来理解：

LView 是一栋楼，decls 告诉运行时"这栋楼前 N 层住 DOM 节点"，vars 告诉运行时"接下来 M 层住脏检查旧值"。楼层分配在启动时就固定了，不会动态扩展。

如果 decls 算错了，DOM 节点引用会"住进"旧值层，下次变更检测取出的会是错误数据。如果 vars 算错了，脏检查会访问越界的内存，造成随机的更新失败。这两个数字的正确性对运行时的正确性至关重要。

🔗 交叉引用：第5章「Ivy 渲染引擎」会详细解剖 LView 数组的内存布局，届时 decls 和 vars 所描述的"楼层结构"会有完整的图示。

类型检查——模板错误的侦探

什么是 TCB

你有没有遇到过这样的报错：模板里用了一个不存在的属性，TypeScript 给了你一个准确的错误位置？这背后是 Angular 的模板类型检查（Template Type Checking）系统，核心是 TCB（Type Check Block，类型检查块）。

TCB 是 Angular 编译器额外生成的一段"虚假"TypeScript 代码。这段代码不会出现在最终 bundle 里，只用于类型检查：它把模板中的表达式翻译成 TypeScript 表达式，让 TypeScript 的类型检查器（tsc）能够理解它们的类型。

以 <div [class]="active">{{title}}</div> 为例，生成的 TCB 类似：

// TCB（类型检查块，不会出现在生产代码中）
function _tcb_MyComponent(ctx: MyComponent) {
  // [class]="active" → 检查 ctx.active 的类型是否可赋值给 class 属性
  const _t1 = ctx.active;
  // {{title}} → 检查 ctx.title 是否存在
  const _t2 = "" + ctx.title + "";
}

TypeScript 编译器对这段虚假代码做类型检查时，一旦发现 ctx.active 或 ctx.title 不存在，就会报错——Angular 语言服务把这个错误"映射"回原始模板的具体位置，你就看到了准确的模板错误提示。

strictTemplates 的影响

tsconfig.json 中的 angularCompilerOptions.strictTemplates 控制类型检查的严格程度。关闭时，Angular 对许多模板表达式使用 any 类型，放宽检查；开启时，Angular 对每个绑定表达式执行完整的 TypeScript 类型推断。

选项strictTemplates: falsestrictTemplates: true
[prop]="value"value 可以是任意类型value 必须与 prop 的类型兼容
{{expr}}不检查 expr 是否为 null/undefined必须处理 null/undefined（或使用 ??、?.）
管道类型不检查管道参数类型严格检查
Signal 解包—自动识别 signal() 返回值并解包类型

对于新项目，强烈建议开启 strictTemplates——它让模板类型错误在编译时暴露，而非留到运行时。

⚠️ 注意：TCB 不参与运行时代码生成，它的错误只影响开发体验，不影响应用能否运行。但 Angular 语言服务（Language Service）的智能提示、跳转定义、重命名重构等能力都依赖 TCB 的准确性。

设计思考

设计思考

为什么是 AOT 而非 JIT？

Angular 从 v9 开始将 AOT 设为默认模式，背后有几个相互加强的理由。

包体积。JIT 模式下，@angular/compiler（约 160KB gzip 后约 55KB）必须随应用一起下载，因为模板需要在浏览器里现场编译。AOT 模式下，编译器完全不进入生产 bundle。这不只是省了 compiler 的体积，更重要的是 AOT 让 tree-shaking 工具能够精确识别哪些 directive 和 pipe 被真正使用——未使用的代码可以被安全删除。

安全性。JIT 模式本质上是在运行时执行动态代码求值（eval()），在 Content Security Policy（CSP）严格的环境（如某些企业内网应用）中无法工作。AOT 生成的是静态函数，完全兼容 CSP。

IDE 支持。Angular 语言服务（Language Service）正是基于 AOT 编译流水线（包括 Template AST 和 TCB）实现的。JIT 路径下，编译器只在运行时"看到"模板，语言服务无法利用这些信息提供精准的类型推断和跳转定义。

预制菜 vs 现炒：AOT 是工厂预制好的成品——用户打开应用时直接"加热上桌"（执行编译好的函数），没有现场编译的开销；JIT 是到场点单现炒——灵活，但"上菜慢"（启动时间长、内存占用高）。

IR 层的历史背景

Angular v14 之前，模板编译器使用 TemplateDefinitionBuilder（TDB）——一个单遍的访问者，直接在 Template AST 遍历过程中输出代码。TDB 工作得不错，但有一个根本性的限制：单遍遍历意味着看到后面节点之前不知道 decls 和 vars 的值，代码生成逻辑和优化逻辑纠缠在一起，难以维护和扩展。

Angular v14 引入了新的 IR（Internal Representation）层，彻底替代 TDB。68 个 Phase 的多遍优化架构，使得每一类变换都可以独立实现、独立测试、独立扩展。@if/@for 控制流语法（v17）、@let 局部变量（v18）、Selectorless 组件语法（v19 实验性）等新特性，都是以新增 IR 操作和 Phase 的方式实现的，完全不影响已有的代码路径。

与其他框架的编译策略对比

从"有没有专属编译器"这个维度看，主流框架可以分为两类：

React + JSX Transform：JSX 本质上是语法糖，<div className={active}> 被 Babel/swc/tsc 直接转换为 React.createElement("div", { className: active })。这是一个通用的语法变换，不需要理解 React 语义。代价是运行时必须有一个虚拟 DOM 协调器（Reconciler）来处理数据绑定和 DOM 更新。React 编译器（React Compiler，前 React Forget）是近年的尝试，在 JSX 层面做记忆化优化，但它是可选的、不改变运行模型的。

Vue 模板编译：Vue 有一个专属的模板编译器，将 <div :class="active">{{ title }}</div> 编译为 h("div", { class: active }, title) 这样的虚拟 DOM 创建调用，再经过 Virtual DOM diff 更新 DOM。Vue 3 在编译器里引入了静态提升（Static Hoisting）等优化，但运行时仍然依赖虚拟 DOM diff。

Angular 的路径：Angular 编译器输出的不是"创建虚拟 DOM"的调用，而是"直接操作真实 DOM"的命令式函数。脏检查在 LView 的绑定槽里以旧值比较的方式完成，完全跳过虚拟 DOM。这个选择的代价是编译器更复杂（需要精确计算 decls/vars/advance），收益是运行时零虚拟 DOM 开销——对大型应用的性能而言，这是值得的权衡。

三者的本质差异在于编译器与运行时的解耦程度：React 把运行时 Reconciler 视为必须依赖；Vue 在编译层优化但运行层仍靠 Virtual DOM；Angular 则把尽可能多的决策前移到编译期，让运行时成为一台执行预先计算好指令的"机器"——编译器越聪明，运行时越轻薄。

小结

让我们用五层流水线回顾整个旅程：

  1. ML AST：Angular 自研的 HTML 解析器（不依赖浏览器 DOM）将模板字符串切割为词法 Token，再组装为通用 HTML 语法树。此时 [class]="active" 只是普通属性。
  1. Template AST：HtmlAstToIvyAst 访问者赋予 HTML 骨架以 Angular 语义。[class]="active" 变成 BoundAttribute { bindingType: Class, value: PropertyRead("active") }，{{title}} 变成 BoundText { value: Interpolation([PropertyRead("title")]) }。
  1. IR：ingestComponent() 将 Template AST 摊平为两条 OpList——Create 链表（DOM 构建指令）和 Update 链表（数据绑定指令），通过 XrefId 跨引用相互关联。
  1. 68 个 Phase：多遍优化流水线依次处理 IR——特化绑定类型、分配 slot（决定 decls）、统计变量（决定 vars）、插入 advance、消除临时变量，最后由 reify 将 IR Op 具体化为 ɵɵXxx 指令调用。
  1. 输出：emitView() 将所有语句包裹进 if (rf & 1) 和 if (rf & 2) 两个分支，生成最终的 ComponentTemplate 函数，写入 ComponentDef.template 字段。

特别要记住 decls 和 vars 的含义：decls 是视图中 DOM 节点的数量，决定 LView 数据区的大小；vars 是变量槽的数量，决定 LView 绑定区的大小。第5章解剖 LView 的内存布局时，这两个数字会变得更加直观——它们是连接编译器与运行时的桥梁。

衔接第5章：LView 与 TView 预览

本章讲的是编译时的决策，第5章讲的是运行时的执行。在正式进入第5章之前，先建立一个直观印象：

Angular 创建组件实例时，根据 decls 和 vars 分配一个固定大小的平坦数组，这就是 LView。以本章示例（decls=2, vars=3）为例，LView 的内存布局如下：

LView[0] = <div> 的 DOM 引用          ← elementStart(0) 写入
LView[1] = 文本节点的 DOM 引用         ← text(1) 写入
—————————— decls = 2 的边界 ——————————
LView[2] = classMap 旧对象引用（var 0）← ɵɵclassMap 脏检查时读写
LView[3] = classMap 旧字符串值（var 1）← ɵɵclassMap 逐属性比较时读写
LView[4] = textInterpolate 旧值（var 2）← ɵɵtextInterpolate 脏检查时读写
—————————— decls + vars = 5 的边界 ——
  • 首次渲染执行 if (rf & 1) 块，向 LView[0..decls-1] 存入 DOM 节点引用
  • 变更检测执行 if (rf & 2) 块，从 LView[decls..decls+vars-1] 取旧值与新值比较
  • TView（Template View）是编译器为每个组件类生成的共享元数据蓝图，不随实例复制；LView 是每个组件实例的独立运行时数组，按 TView 的 blueprint 分配

decls 和 vars 在编译时确定，LView 大小在实例化时一次性分配，运行期间不再动态扩展——这正是 Angular 脏检查高效的物质基础。

思考题

  1. 对于模板 <button (click)="submit()" [disabled]="isLoading">提交</button>，请手动计算其 decls 和 vars 的值。（提示：事件绑定 (click) 生成 Listener Op，Listener 不实现 ConsumesSlotOpTrait；[disabled] 是 PropertyOp，消耗 1 个 var）
  1. 为什么 ɵɵclassMap 需要 2 个 vars，而 ɵɵproperty 只需要 1 个？运行时需要存储哪两种信息？
  1. 如果你在模板中写了 @if (show) { <span>{{message}}</span> }，@if 块的 decls 和 vars 会如何计算？主视图的 decls 和 vars 是否会包含 <span> 这个子视图的节点？（提示：子视图是独立的 ViewCompilationUnit，有自己的 decls 和 vars；主视图的 decls 包含一个 ConditionalCreateOp 的 slot，但不包含 <span> 本身的 slot）
