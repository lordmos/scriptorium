第5章：Ivy 渲染引擎——LView 的生命

ɵɵelementStart(0, 'div') 执行的那一刻，Angular 把什么存在了哪里？

这行代码你可能在编译器生成的产物里见过，也可能在调试器的调用栈里偶然扫到过，但大多数时候它就像黑盒子里的机器齿轮——你知道它转，但不知道它咬合着什么。新建的 <div> DOM 节点要有地方住；你的组件实例（this）要有地方住；上一次渲染留下的属性绑定值，也要存在某个地方，等着下次变更检测来比对。

这个"地方"，就是 LView——一个用扁平数组模拟的组件运行时内存快照。

延续第1章建立的剧院比喻：LView 是今晚这场演出的舞台——舞台上有演员（DOM 节点）、道具（绑定值缓存）、幕布状态（视图标志位），每场演出（每个组件实例）都有一块独立的舞台。而 TView 是剧院的建筑设计图纸——描述这类舞台的结构：该有几盏灯、几个入场口、几处布景悬挂点。同一类舞台（同一组件类型）共用一份图纸，图纸本身无论演了多少场都不会改变。

理解 LView 与 TView，是读懂 Angular Ivy 渲染引擎的基础；而这两个数据结构，又是后续章节——变更检测（第6章）、Signals 集成（第7章）、SSR 水合（第10章）——一切讨论的起点。

还有第三个主角你需要认识：TNode（Template Node，模板节点）。如果说 TView 是剧院的建筑图纸，TNode 就是图纸里对每个座椅或道具架的精确描述——它的类型（元素/文本/容器）、静态属性、关联的指令列表。TNode 存储在 TView 的 data 数组中，与 LView 在相同索引处存储运行时的 DOM 节点形成严格对应。这三者——TView、LView、TNode——构成了 Ivy 渲染引擎的数据模型三角，理解它们的分工是本章的核心任务。

全景设计：蓝图与实例

一份图纸，无数舞台

TView 和 LView 的关系，可以借用面向对象语言的类比来理解：TView 是"类描述符"，LView 是"类实例"。

同一个组件类型（如 UserCardComponent）无论在页面上被实例化多少次，只有一个 TView。这个 TView 记录的是静态、编译期已知的结构信息——模板里有几个节点、每个节点是什么类型、注册了哪些指令、有多少个绑定、生命周期钩子注册在哪里。这些信息在第一个实例创建时固化，后续所有实例共享。

每次创建新的组件实例，Angular 都会克隆 TView 中预存的蓝图（blueprint），得到一个新的 LView 数组。LView 存储的是运行时变化的数据：当前帧的 DOM 节点引用、组件实例（this）、上次渲染后缓存的绑定值、视图的当前状态标志。

HEADER_OFFSET = 27：固定与动态的分界线

LView 是一个普通的 JavaScript 数组，但它的内部结构被划分为两个语义截然不同的区域。

固定头部（索引 0–26）：27 个语义固定的槽位，存储视图的元信息。无论模板有多复杂，这 27 个槽位的含义始终不变。常量 HEADER_OFFSET = 27 是固定头部的边界标志，也是动态体部的起始索引。

动态体部（索引 27 起）：由编译器根据模板内容在编译时静态计算的动态区域，存放 DOM 节点引用、指令实例、绑定值缓存。体部的大小写入 ComponentDef 的 decls（节点数）和 vars（绑定数）字段，在每次 createTView() 时用来分配蓝图。

理解这个分层设计有一个直观的角度：固定头部是框架自用的"系统区"，与具体模板内容无关；动态体部是"用户数据区"，完全由编译器按模板分配。两者合并成一个连续数组，框架在遍历时凭借 HEADER_OFFSET 这一个边界，便能同时访问两个区域的内容。

体部的三个子区段：

子区段索引范围存储内容
Decls 区[27, 27 + decls)DOM 节点引用、LContainer、指令实例
Vars 区[27 + decls, 27 + decls + vars)绑定值缓存，初始为 NO_CHANGE
Expando 区expandoStartIndex 起，动态扩展运行时匹配的指令实例、Host Binding 数据

HEADER_OFFSET = 27 这个数字随 Ivy 的演化而增长：每当框架新增一个需要在所有视图上追踪的全局状态（v17 加入的 REACTIVE_TEMPLATE_CONSUMER、v18 加入的 EFFECTS），头部就扩展一格。这也是为什么源码中始终使用 HEADER_OFFSET 常量，而不是硬编码数字 27。

LView 槽位解剖

这一节是本章最核心的部分。只有理解了固定头部每个槽位的语义，你才能在调试器里"认出"一个 LView，才能理解变更检测为何能以如此精准的方式工作。

在进入槽位列表之前，先建立一个直觉：LView 的固定头部可以被分为几个功能组。

身份与结构组（槽位 0–5）：回答"这个视图是谁、在哪、父子关系如何"。HOST 告诉你宿主元素，TVIEW 告诉你它属于哪类组件，PARENT 告诉你它在视图树中的位置。

状态与清理组（槽位 6–7）：HYDRATION 用于 SSR 水合（hydration），CLEANUP 记录需要在销毁时释放的资源（事件监听器、订阅等）。

执行上下文组（槽位 8–11）：回答"模板函数执行时，从哪里取组件实例、注入器、渲染器"。这四个槽位是每次执行 executeTemplate() 时最频繁被读取的。

树遍历组（槽位 12–16）：回答"如何找到子视图和声明位置"。变更检测和视图树操作高度依赖这几个槽位。

框架扩展组（槽位 17–26）：记录查询、Effect、Signal 消费者、动画等各功能模块在视图上的状态。每增加一个功能，这个区域就可能扩展一格，HEADER_OFFSET 随之增加。

理解这些功能分组后，槽位表就不再是一份需要死记硬背的清单，而是一张有内在逻辑的地图。

所有槽位常量都定义在同一个文件：

固定头部关键槽位详解

索引常量名类型含义与用途
0HOST`RElement \null`宿主 DOM 元素。对于组件视图，指向组件的宿主元素（如 <app-hello>）；根视图为 null
1TVIEWTView指向共享的 TView。所有同类型实例的此槽位都指向同一个对象，是 flyweight 模式的直接体现
2FLAGSLViewFlags位标志组合，编码了视图的完整状态：创建模式、脏标记、是否已附加、是否已销毁等
3PARENT`LView \LContainer \null`父级视图或容器。普通子组件指向父 LView；嵌入视图指向所在的 LContainer
4NEXT`LView \LContainer \null`兄弟视图/容器链表的下一节点，用于横向遍历
5T_HOST`TNode \null`此 LView 对应的 TNode，指向其所在模板节点；嵌入视图中指向容器 TNode，组件视图中指向宿主元素 TNode
6HYDRATION`DehydratedView \null`SSR 水合（hydration）相关序列化数据；非 SSR 环境为 null
7CLEANUP`any[] \null`DOM 事件监听器的清理函数列表。视图销毁时，框架遍历此数组移除所有事件监听
8CONTEXTT组件实例本身（即 this）。模板执行时，所有 ctx.xxx 访问的都是这里存储的对象
9INJECTORInjector后备环境注入器（原 Module Injector）。NodeInjector 在元素树中查找失败时，向此注入器委托

12CHILD_HEAD`LView \LContainer \null`子视图/容器链表头。变更检测遍历子视图的入口，refreshView 通过它触发子视图刷新
13CHILD_TAIL`LView \LContainer \null`子视图/容器链表尾。新子视图通过 addToEndOfViewTree 追加到此
14DECLARATION_VIEW`LView \null`模板声明处所在的 LView（与渲染位置可能不同）。ng-template 跨组件传递时，这个槽位让框架正确解析闭包变量

其中，FLAGS（槽位 2）值得单独展开。LViewFlags 是一个位枚举，视图的所有状态都编码在这一个整数里：

// packages/core/src/render3/interfaces/view.ts（LViewFlags 枚举，节选）
export const enum LViewFlags {
  CreationMode   = 1 << 2,   //   4：首次创建阶段，模板函数正在构建 DOM 树
  FirstLViewPass = 1 << 3,   //   8：首次完整创建+更新流程未结束
  CheckAlways    = 1 << 4,   //  16：Default 变更检测策略，每次 tick 都检查此视图
  Dirty          = 1 << 6,   //  64：标记需要重新检查（OnPush 时由 markForCheck 设置）
  Attached       = 1 << 7,   // 128：已附加到变更检测树，会被遍历到
  Destroyed      = 1 << 8,   // 256：已销毁，所有后续操作应跳过
  RefreshView    = 1 << 10,  //1024：需要强制刷新（Signal 视图、跨组件移动的视图）
  SignalView     = 1 << 12,  //4096：signal 驱动的组件（v19+，Zoneless 模式关键标志）
}

一个默认策略（Default）的新建视图，FLAGS 的初始值是：

CreationMode(4) | FirstLViewPass(8) | CheckAlways(16) | Dirty(64) | Attached(128) | RefreshView(1024)
= 4 | 8 | 16 | 64 | 128 | 1024 = 1244

变更检测算法通过位运算读写这个整数——例如，清除 Dirty 标志是 lView[FLAGS] &= ~LViewFlags.Dirty，检查是否处于创建模式是 (lView[FLAGS] & LViewFlags.CreationMode) !== 0。

🔗 关于 Dirty 标志何时被设置（属性绑定变化、Input 变化、事件处理等）以及 markForCheck() 如何向上标记祖先链，详见第 6 章。

一个真实的 LView 快照

来看一个具体例子。对于下面这个组件：

// 示例组件
@Component({
  selector: 'app-user-card',
  template: `<div class="card">{{ userName }}</div>`
})
export class UserCardComponent {
  userName = 'Alice';
}

编译器会生成 decls = 2（一个 <div> 元素节点 + 一个文本节点）、vars = 1（一个插值绑定）。 对应的 LView 在首次渲染后快照如下：

// LView 内存快照（UserCardComponent 实例，首次渲染后）
lView = [
  /*  0: HOST        */ <app-user-card>,          // 组件宿主元素
  /*  1: TVIEW       */ TView{ type: Component },  // 指向共享的 TView
  /*  2: FLAGS       */ 16,                        // CheckAlways（首次变更检测完成后的稳定状态；创建阶段初始值为 1244）
  /*  3: PARENT      */ rootLView,                 // 父级视图
  /*  ...            */
  /*  8: CONTEXT     */ UserCardComponent{userName:'Alice'}, // 组件实例 this
  /*  9: INJECTOR    */ EnvironmentInjector{},
  /* 10: ENVIRONMENT */ LViewEnvironment{…},
  /* 11: RENDERER    */ DefaultDomRenderer2{},
  /* 12: CHILD_HEAD  */ null,                      // 无子视图
  /* 13: CHILD_TAIL  */ null,
  /*  ...            */
  /* 27: Decls[0]    */ <div class="card">,        // TNode#27 对应的 DOM 元素
  /* 28: Decls[1]    */ #text "Alice",             // TNode#28 对应的文本节点
  /* 29: Vars[0]     */ 'Alice',                   // userName 上一次的绑定值（已由 NO_CHANGE 更新）
]

注意 TView.data 中的对应位置：

// TView.data 在相同索引处存储静态描述
tView.data = [
  /* 0-26: null */
  /* 27 */ TElementNode{ index:27, type:Element, value:'div', ... },
  /* 28 */ TTextNode{ index:28, type:Text, value:'', ... },
  /* 29 */ 'userName',   // 绑定属性名元数据
]

TView.data[i] 是节点的静态描述（TNode 对象），LView[i] 是节点的动态实例（DOM 节点）。同一个索引，两个数组，各司其职——这是 Ivy 最优雅的设计决策之一。

这个并排布局还有一个重要推论：框架从来不需要"查找"某个节点在哪里。节点的索引在编译时已经确定并写入 TNode.index，运行时访问 lView[tNode.index] 是 O(1) 的直接索引操作，没有哈希表查找，没有链表遍历，也没有树搜索。在一个有数千个绑定需要频繁更新的应用里，这个设计差异累积起来是显著的性能优势。

TView 的三种类型

Angular 的视图并非铁板一块。根据创建方式和使用场景，TView 分为三种类型：

// packages/core/src/render3/interfaces/view.ts
export const enum TViewType {
  Root = 0,       // 应用根视图，bootstrapApplication 创建
  Component = 1,  // 普通组件视图
  Embedded = 2,   // ng-template 创建的嵌入视图
}

Root TView：整棵视图树的起点，由 bootstrapApplication 在应用启动时创建。它没有父组件，HOST 槽位指向根组件的挂载元素（如 <body> 或 <app-root>）。全应用只有一个根视图。

Component TView：最常见的类型，每个组件类对应唯一一个。当 UserCardComponent 第一次被实例化时，框架调用 getOrCreateComponentTView(def)，如果 def.tView 尚未初始化则创建，否则复用。这意味着即便你的应用里有一千个 UserCardComponent 实例，也只有一个 Component TView 被创建。

Embedded TView：专为 <ng-template> 和结构型指令（*ngFor、@for 等）创建的视图。嵌入 TView 的存储位置特殊——它不挂在 ComponentDef 上，而是挂在其所在的容器节点（TContainerNode）的 .tView 字段上：

// packages/core/src/render3/instructions/template.ts（简化）
const embeddedTView = (tNode.tView = createTView(
  TViewType.Embedded,
  tNode,          // 声明此 embedded view 的容器 TNode
  templateFn,     // ng-template 编译生成的模板函数
  decls, vars,
  declarationTView.directiveRegistry,
  // ...
));

*ngFor 每次迭代创建一个新的 LView（Embedded 类型），但所有迭代共享同一个 Embedded TView——这与组件的情况完全一致。列表里有 100 条数据，就有 100 个 LView，但只有 1 个 TView。

这里有一个容易混淆的细节：Embedded TView 的 template 字段存储的是 <ng-template> 内部内容的模板函数，而非宿主组件的模板函数。当 *ngFor 创建一个新的迭代视图时，它用 Embedded TView 的 template 函数来渲染每一个列表项，同时把迭代上下文（NgForOfContext，包含 $implicit、index、first、last 等）存入新建 LView 的 CONTEXT 槽位（槽位 8）。这就是模板变量 let item of items; let i = index 能正确获取值的底层机制。

蓝图克隆机制：从图纸到舞台

TView 的关键字段 blueprint 是 LView 实例的原型数组。createTView() 调用 createViewBlueprint() 生成它：

// packages/core/src/render3/view/construction.ts
function createViewBlueprint(bindingStartIndex: number, initialViewLength: number): LView {
  const blueprint = [];
  for (let i = 0; i < initialViewLength; i++) {
    // Header 区（索引 0 到 bindingStartIndex-1）初始化为 null
    // Vars 区（bindingStartIndex 起）初始化为 NO_CHANGE 哨兵值
    blueprint.push(i < bindingStartIndex ? null : NO_CHANGE);
  }
  return blueprint as LView;
}

每次 createLView() 被调用，核心操作只有一行：

// packages/core/src/render3/view/construction.ts
const lView = tView.blueprint.slice() as LView;

不带参数的 Array.prototype.slice() 对整个数组做浅拷贝，V8 引擎对此做了高度优化（内部直接 memcpy）。对于一个典型组件（数十个节点 + 数十个绑定），这意味着数十次基础类型赋值，远比"为新对象逐字段初始化"的方式节省 GC 分配压力。

此外，NO_CHANGE 是 Angular 内部定义的一个特殊符号引用（const NO_CHANGE: unique symbol），Vars 区初始化为 NO_CHANGE 而非 null，确保首次绑定检查时 Object.is(NO_CHANGE, '真实值') 必然返回 false，强制触发首次更新——而不会误把 null 解读为合法的绑定值。

组件生命周期追踪

现在把所有知识串在一起，完整追踪一个组件从创建到首次渲染屏幕的全过程。

假设有如下组件：

@Component({
  selector: 'app-hello',
  template: `<div>Hello, {{ name }}!</div>`
})
export class HelloComponent {
  name = 'World';
}

编译器将为它生成：decls = 2，vars = 1，以及一个模板函数。

创建阶段：一次性搭建所有舞台布景

整个创建流程的起点是 createComponent()（或通过路由匹配、动态组件等路径间接触发）。创建阶段最重要的特征是 LViewFlags.CreationMode 标志被设置——所有渲染指令通过检查这个标志来决定是"创建并存储"还是"读取并比较"。

// 调用链（简化，packages/core/src/render3/）
createComponent(HelloComponent, options)
  └─ component.ts: ComponentFactory.create()
       └─ view/construction.ts: createLView(parentLView, tView, context, flags, host)
            │  tView.blueprint.slice()  ← 克隆蓝图
            │  lView[FLAGS] |= CreationMode | Attached | FirstLViewPass | Dirty | RefreshView
            │
            └─ instructions/shared.ts: renderView(tView, lView, context)
                 └─ executeTemplate(tView, lView, templateFn, RenderFlags.Create, context)
                      └─ templateFn(RenderFlags.Create, ctx)
                           ├─ ɵɵelementStart(0, 'div')   // 创建 TNode + DOM 节点
                           ├─ ɵɵtext(1)                  // 创建文本 TNode + 文本节点
                           └─ ɵɵelementEnd()

🔗 第4章回顾：关于 RenderFlags 的定义和位标志的工作原理，详见第 4 章。

更新阶段：只改动发生变化的部分

当 name 从 'World' 变为 'Angular'，变更检测被触发。此时 CreationMode 已清除，模板函数以 RenderFlags.Update 执行：

// 编译器生成，非手写
function HelloComponent_Template(rf: RenderFlags, ctx: HelloComponent) {
  if (rf & RenderFlags.Create) {     // 1：创建阶段
    ɵɵelementStart(0, 'div');
    ɵɵtext(1);
    ɵɵelementEnd();
  }
  if (rf & RenderFlags.Update) {     // 2：更新阶段
    ɵɵadvance(1);                    // 移动内部游标到节点索引 1
    ɵɵtextInterpolate1('Hello, ', ctx.name, '!');
  }
}

⚠️ 注意：ɵɵelementStart 本身不会在更新阶段重复执行——它被 if (rf & RenderFlags.Create) 分支保护，只在创建阶段运行一次。只有绑定指令（如 ɵɵproperty、ɵɵtextInterpolate1）才会在更新阶段执行。

更新路径的核心在 bindingUpdated()：

// packages/core/src/render3/bindings.ts（简化）
export function bindingUpdated(lView: LView, bindingIndex: number, value: unknown): boolean {
  const oldValue = lView[bindingIndex];
  if (Object.is(oldValue, value)) {
    return false; // 值未变化，跳过所有后续 DOM 操作
  }
  lView[bindingIndex] = value; // 更新缓存
  return true;                 // 值变化，调用方负责更新 DOM
}

Object.is() 只执行一次，若旧值与新值相同则函数立刻返回。没有 TNode 创建，没有 DOM 节点创建，没有属性设置——什么都没有。对于一个有 50 个属性绑定但只有 2 个实际变化的组件，48 次 bindingUpdated() 调用只消耗 48 次 Object.is() 比较，代价几乎为零。

两种模式下的完整行为对比：

操作创建模式（CreationMode 位为 1）更新模式（CreationMode 位为 0）
TNode 操作getOrCreateTNode 创建并存入 tView.data从 tView.data[i] 直接读取（已存在）
DOM 节点创建renderer.createElement() 调用完全跳过
指令匹配resolveDirectives() 扫描并注册指令跳过
绑定检查NO_CHANGE → 新值，必然触发更新Object.is() 比较，无变化则跳过
生命周期钩子执行 ngOnInit、ngAfterViewInit执行 ngDoCheck、ngOnChanges
RenderFlags 参数RenderFlags.Create（1）RenderFlags.Update（2）

这个双阶段设计有一个工程意义：编译器把"创建逻辑"和"更新逻辑"编译进同一个模板函数，通过 RenderFlags 参数的位标志分支，而不是生成两个独立函数。对于摇树优化（tree-shaking）来说，一个在 SSR 环境中只被创建而从不更新的组件，理论上可以移除更新分支。这是 Ivy 相比前代 View Engine 在 tree-shaking 粒度上的重要进步。

销毁阶段：清理舞台

组件销毁时，Angular 调用 destroyLView(tView, lView)：

// packages/core/src/render3/view/construction.ts（简化）
ViewRef.destroy()
  → destroyLView(tView, lView)
       ├─ 执行 tView.destroyHooks 中所有 ngOnDestroy 回调
       ├─ 遍历 lView[CLEANUP] 移除所有 DOM 事件监听器
       └─ lView[FLAGS] |= LViewFlags.Destroyed

lView[CLEANUP]（槽位 7）在创建阶段由 ɵɵlistener 等渲染指令逐条注册清理函数。销毁时一次性全部调用，这是 Angular 不产生内存泄漏的重要保证。

关键渲染指令深度解析

理解了数据结构，我们现在跟踪一条具体的渲染指令（instruction），看看它如何在 TView 和 LView 之间完成精准的读写。理解这条调用链，你就掌握了 Ivy 渲染指令的通用模式——其他所有渲染指令（ɵɵtext、ɵɵproperty、ɵɵlistener）都遵循类似的"首次创建做完整初始化，后续执行仅比较绑定值"的结构。

ɵɵelementStart：四层协作的起点

以模板 <div [class]="cardClass"> 为例，编译产物调用 ɵɵelementStart(0, 'div')（相对索引 0，标签名 'div'）。

首次创建时的完整调用链：

ɵɵelementStart(0, 'div')
│ // packages/core/src/render3/instructions/element.ts
│
├─ adjustedIndex = 0 + HEADER_OFFSET = 27    ← 相对索引 → 绝对索引
│
├─ [tView.firstCreatePass === true，首次创建]
│   └─ directiveHostFirstCreatePass(27, lView, TNodeType.Element, 'div', attrs)
│        // packages/core/src/render3/view/elements.ts
│        │
│        ├─ getOrCreateTNode(tView, 27, TNodeType.Element, 'div', attrs)
│        │    // packages/core/src/render3/tnode_manipulation.ts
│        │    ├─ tView.data[27] === null？
│        │    └─ [是] → createTNodeAtIndex(tView, 27, Element, 'div', attrs)
│        │              └─ createTNode(tView, parent, Element, 27, 'div', attrs)
│        │                   // 返回 TElementNode{index:27, type:Element, value:'div'}
│        │                   // tView.data[27] = tNode  ← 存入共享的 TView
│        │
│        └─ resolveDirectives(...)    // 匹配并注册指令（仅首次）
│

     └─ appendChild(tView, lView, native, tNode)

注意两次"存入"的目标完全不同：

  • tView.data[27] = tNode：写入共享的 TView，所有实例只写一次
  • lView[27] = nativeElement：写入当前实例的 LView，每次创建都执行

这是"静态元数据/动态实例"分离原则在最基层的体现。第二个组件实例创建时，tView.firstCreatePass 已为 false，getOrCreateTNode 直接从 tView.data[27] 读取已存在的 TNode，省去了所有 TNode 创建和指令匹配的开销。

从这个调用链还可以看出 Angular 渲染器（Renderer）的抽象价值：renderer.createElement('div') 是唯一一个真正"触碰"运行平台的调用——在浏览器里它调用 document.createElement()，在 Node.js（SSR）里它创建一个虚拟 DOM 节点对象。渲染指令层（ɵɵelementStart）对运行平台完全无感，只通过 lView[RENDERER] 持有的渲染器接口进行间接调用。这是 Angular 支持多平台（浏览器、SSR、Web Worker）的核心架构保证。

ɵɵproperty：精准绑定更新

更新阶段的 ɵɵproperty('class', ctx.cardClass)：

// packages/core/src/render3/instructions/property.ts（简化）
export function ɵɵproperty<T>(propName: string, value: T): typeof ɵɵproperty {
  const lView = getLView();
  const bindingIndex = nextBindingIndex(); // 从 bindingStartIndex 单调递增

  if (bindingUpdated(lView, bindingIndex, value)) {
    // 仅当 Object.is(旧值, 新值) 为 false 时才进入此分支
    const tNode = getSelectedTNode();
    setPropertyAndInputs(tNode, lView, propName, value, lView[RENDERER], sanitizer);
    //  ├─ 有 @Input('class')？→ 调用 setAllInputsForProperty 触发输入绑定
    //  └─ 无 @Input       ？→ renderer.setProperty(element, 'className', value)
  }
  return ɵɵproperty;
}

⚠️ 注意：ɵɵproperty 使用 Object.is() 而非 === 进行比较。两者的区别在于：Object.is(NaN, NaN) 返回 true（视为"未变化"，不更新 DOM），而 NaN === NaN 返回 false。这意味着如果你的绑定值是 NaN，Angular 不会反复更新 DOM——这是有意为之的优化。

💡 提示：ɵɵtext 和 ɵɵtextInterpolate1 的工作方式类似，只不过它们用 interpolation1() 拼接字符串后，最终调用 renderer.setValue(textNode, interpolatedString) 更新文本节点内容，而非 renderer.setProperty()。

LView 树：嵌套组件的世界

现实中的 Angular 应用是一棵组件树。LView 不是孤立存在的，而是通过多种链接方式形成一棵可遍历的树。

子组件的安置方式

当父模板中出现 <child-comp>，Angular 在处理该元素时检测到 isComponentHost(tNode) === true——这是一个组件宿主节点。框架为子组件创建 LView，并将其存放在父 LView 中该宿主节点的槽位：

// packages/core/src/render3/view/construction.ts（简化）
export function createComponentLView<T>(
  parentLView: LView,
  hostTNode: TElementNode,  // <child-comp> 对应的 TNode
  def: ComponentDef<T>,
): LView {
  const tView = getOrCreateComponentTView(def);
  const componentView = createLView(
    parentLView, tView, null, getInitialLViewFlagsFromDef(def), native, hostTNode, ...
  );

  addToEndOfViewTree(parentLView, componentView); // 加入 CHILD_HEAD/CHILD_TAIL 链表

  // 子组件 LView 存储在父 LView 主机节点对应的槽位
  return (parentLView[hostTNode.index] = componentView);
}

假设父模板是 <div><child-comp></child-comp></div>，TNode 索引 27 是 <div>，索引 28 是 <child-comp> 的宿主节点，那么：

  • parentLView[27] = <div> 的 DOM 元素引用
  • parentLView[28] = childLView（子组件的 LView 对象）

同一个槽位，在 TView.data 里是 TNode，在 LView 里是 DOM 节点或 LView——这是 Ivy 索引对应规则的一个微妙之处：如果该节点是普通 DOM 节点，存 DOM 引用；如果是组件宿主，存子组件的 LView。

CHILD_HEAD / CHILD_TAIL：变更检测的路线图

CHILD_HEAD（槽位 12）和 CHILD_TAIL（槽位 13）构成了一条单向链表，链接当前视图的所有直接子视图（子组件 LView 和 LContainer 都在其中）：

变更检测时，refreshView 通过遍历 tView.components 数组（存储所有子组件 TNode 的索引）触发子组件检测，同时遍历嵌入视图链表刷新结构型指令创建的视图：

// packages/core/src/render3/instructions/change_detection.ts（节选）
function refreshView<T>(tView: TView, lView: LView, templateFn, context: T) {
  // ... 执行模板函数（更新模式）...

  detectChangesInEmbeddedViews(lView, mode);     // 遍历 LContainer 内的嵌入视图
  refreshContentQueries(tView, lView);           // 刷新内容查询
  processHostBindingOpCodes(tView, lView);       // 更新 host bindings

  const components = tView.components;
  if (components !== null) {
    detectChangesInChildComponents(lView, components, mode); // 触发子组件检测
  }
}

LContainer：结构型指令的视图仓库

*ngIf、*ngFor、@if、@for 等结构型指令并不直接操作 LView——它们操作的是 LContainer（视图容器）。LContainer 同样是一个数组，前几个槽位与 LView 共享（FLAGS、PARENT、NEXT、T_HOST），从 CONTAINER_HEADER_OFFSET（= 10）起存放嵌入 LView 的数组。

从 parentLView[tNode.index] 的角度看，*ngIf 宿主节点的槽位存储的是 LContainer 对象，而非 DOM 节点。当 *ngIf 条件变为 true，框架创建一个新的嵌入 LView，推入 LContainer 数组，并将其中的 DOM 节点插入 DOM 树；条件变为 false 时，将 LView 从数组中移除，对应 DOM 也被移除。整个过程不需要重新创建 TNode 或重新解析模板。

内容投影：跨视图边界的借用

<ng-content> 的实现依赖两条配合工作的渲染指令：

ɵɵprojectionDef（在子组件模板函数的创建阶段执行）：扫描从父组件传入的内容节点（TNode 链表），按 <ng-content select="..."> 的 CSS 选择器将它们分组存入子组件宿主 TNode 的 projection 数组。

ɵɵprojection（在子组件模板函数的创建阶段，<ng-content> 占位处执行）：遍历对应选择器分组的 TNode 链，将它们对应的 DOM 节点实际插入子组件视图的指定位置。

关键特性：投影内容的 TNode 和 LView 归属于父组件，子组件只是借用了它们的 DOM 位置。这意味着：

  • 投影内容的变更检测由父组件负责触发（谁声明，谁负责）
  • 投影内容销毁时，其 DOM 节点的清理也属于父组件的职责

这个"归属权"的区别有一个实际的调试意义：如果你在父组件中修改了投影内容绑定的值，却发现它没有更新，问题往往不在子组件，而在父组件的变更检测策略（比如父组件是 OnPush 策略且没有被标记为 dirty）。

设计思考

设计思考

为什么用扁平数组，而不是对象树？

最直观的设计或许是：一个组件对应一个 JS 对象，拥有 domNodes、bindings、context、children 等属性。这样代码可读性更好，Angular 为什么选择了"数组 + 整数索引"这种看似原始的方案？

核心原因是内存局部性（memory locality）。CPU L1/L2 缓存对连续内存区域极为友好：遍历数组时，相邻元素通常在同一缓存行（cache line）中，访问速度极快；而对象树意味着大量指针间接寻址——每次访问子节点都需要解引用一个堆指针，极易引发缓存失效（cache miss）。变更检测需要对整棵视图树进行高频遍历，这个差异在规模化应用中是实测可见的。

另一个角度是 GC 压力。JavaScript 的垃圾回收器需要追踪所有对象引用关系。一个扁平数组只是"一个对象"，而对象树的每个节点是独立的堆分配对象。当一个大型应用有数千个组件时，这个差异对 GC 暂停时间（jank）有直接影响。此外，整数索引访问在 V8 等引擎中有专项优化（SMI 索引数组可以内部表示为原生 C 数组），访问速度接近命名属性访问的两倍。

Blueprint 克隆 vs 工厂函数

另一种选择是工厂函数——每次调用返回一个初始化好的 LView。工厂函数更易于理解和测试，但每次调用都要执行初始化逻辑，有额外的函数调用和分支开销。蓝图克隆是纯粹的内存复制语义：一次 slice()，得到一个正确初始化的数组，V8 内部将其降为 memcpy 级别的操作。蓝图还充当了一份"活文档"——直接看 tView.blueprint 就能知道 LView 的初始化形态，无需执行任何逻辑。

工厂函数的一个潜在优势是灵活性——可以根据运行时条件动态调整初始化。但对于 Angular 来说，LView 的初始布局在编译时完全确定，不需要运行时灵活性，因此蓝图克隆是更合适的选择：简单、可预测、高性能。

HEADER_OFFSET = 27 的演化史

这个数字不是一开始就是 27 的。Ivy 早期（v9 发布时），HEADER_OFFSET 只有 17 左右；随着框架功能增加，头部不断扩展：i18n 支持新增了 HYDRATION 槽，Signals 集成新增了 REACTIVE_TEMPLATE_CONSUMER、EFFECTS 和 EFFECTS_TO_SCHEDULE，afterRender API 新增了 AFTER_RENDER_SEQUENCES_TO_ADD……每增加一个全局视图状态，所有历史代码访问的动态体部索引都要相应偏移。Angular 团队通过将 HEADER_OFFSET 作为常量导出，确保无论数字如何变化，框架内外的代码都只依赖常量名，不依赖具体数值。

如果你在 Angular 源码中看到 lView[27] 而非 lView[HEADER_OFFSET]，那几乎可以判定是历史遗留的 tech debt——这样的代码在下一次头部扩展时就会静默地出错，产生极难排查的偏移 bug。

小结

从一个问题出发：ɵɵelementStart(0, 'div') 执行时数据存在哪里？答案是两个地方，各司其职——TNode 存入共享的 TView（静态元数据），DOM 节点存入当前实例的 LView（动态状态）。

本章核心要点：

  • LView 是扁平数组，索引 0–26 为固定头部（元信息），27 起为动态体部（节点/绑定）；HEADER_OFFSET = 27 是两者的分界常量
  • TView 是共享蓝图，同类型所有 LView 实例指向同一 TView；蓝图克隆（blueprint.slice()）是 LView 创建的核心优化
  • TViewType 分为 Root、Component、Embedded 三种，*ngFor 的每次迭代是 Embedded LView，但共享同一个 Embedded TView
  • TNode 与 LView 并排布局：TView.data[i] 存 TNode（静态结构描述），LView[i] 存 DOM 节点（动态实例），同一索引，两类数据
  • CreationMode 标志区分首次创建与后续更新，两种模式下模板函数走完全不同的代码路径，避免重复创建 TNode 和 DOM 节点
  • 嵌套组件的 LView 存于父 LView 主机节点槽位，同时链入 CHILD_HEAD/CHILD_TAIL 链表供变更检测遍历
  • NO_CHANGE 哨兵值与 Object.is() 配合，保证绑定检查的正确性与最低开销

这些设计决策不是孤立的——它们都服务于同一个目标：在频繁的变更检测循环中，以最小的计算和内存代价，精准地更新发生变化的 DOM。Ivy 的性能优势，归根结底来自于这个数据模型的精心设计。

🔗 交叉引用：第6章将从变更检测的视角重新审视 LView——Dirty 标志如何被设置和清除、refreshView 如何递归遍历 CHILD_HEAD/CHILD_TAIL 链表、OnPush 策略如何利用 CheckAlways/Dirty 位组合跳过整棵子树的检查。LView 是变更检测的"操作对象"，本章建立的数据结构认知是理解下一章算法的直接前提。

变更检测将按深度优先顺序遍历 CHILD_HEAD/CHILD_TAIL 链表，Dirty 标志决定哪些节点需要被实际检查——这是第 6 章的核心议题。此外，EFFECTS（槽位 23）和 REACTIVE_TEMPLATE_CONSUMER（槽位 24）是 Signals 运行时架构的两个支点，第 7 章将深入展开它们与 LView 的协作机制。
