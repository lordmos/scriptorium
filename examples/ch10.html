第10章：服务端渲染与水合——让 Angular 应用「秒开」

用户打开一个 Angular SPA，在网络稍差的环境下，屏幕上出现的是空白——先是几百毫秒的 HTML 解析等待，然后是 JavaScript 下载、解析、执行，最后才是框架启动、数据加载、视图渲染。这一连串操作叠加下来，白屏时间轻易超过 2 秒。对用户来说，这 2 秒是焦虑与放弃的临界点；对搜索引擎爬虫来说，一张空白的 <div id="root"> 也几乎毫无价值。

服务端渲染（Server-Side Rendering，SSR）提供了一条出路：在服务器上预先运行 Angular，生成完整的 HTML 字符串，让浏览器在首字节到达后就能立即呈现可见内容。但这只解决了可见性问题——Angular 仍然需要在客户端重新启动，接管页面的交互逻辑。这个"接管"的过程叫水合（Hydration）。

早期水合的做法非常粗暴：Angular 直接销毁服务端 HTML，从头构建 DOM 树——等于把服务端的工作推翻重来，用户依然会看到页面内容短暂消失再出现。从 v16 起，Angular 引入了基于 DOM 复用的水合机制：不再推翻重建，而是让客户端识别服务端遗留的每一个 DOM 节点，将其直接纳入组件树管理——这才是真正的"秒开"。

本章将沿着两条线——从 HTTP 请求到 HTML 字符串的服务端路径，以及从静态 HTML 到交互就绪的客户端水合路径——深入 packages/platform-server/ 和 packages/core/src/hydration/ 的源码，揭开 Angular SSR 的完整运作机制。

📝 在阅读本章之前，建议在本地搭建一个启用了 SSR 的 Angular 项目（ng new --ssr），用浏览器 DevTools 观察：<body> 的第一个子节点是否是 `、根组件上是否有 ngh 属性、页面源码中是否有 <script type="application/json">` 标签。有了这些直观感受，后续的源码分析会更容易落地。

⚠️ 版本说明：本章源码引用基于 Angular main 分支（约 v19.x）。增量水合（withIncrementalHydration）在 v19 进入稳定版。函数签名和行号可能随版本演进，建议以函数名在当前版本中搜索定位。

§10.1 SSR 概览：双轨渲染架构

在浏览器中，Angular 的渲染对象是原生 DOM；在服务端，Node.js 中没有原生 DOM。Angular SSR 的解决方案是引入 Domino——一个纯 JavaScript 实现的 DOM 仿真库，让同一套 Angular 代码能在 Node.js 环境里"以为自己在操作浏览器 DOM"。

可以把 SSR 想象成一场完整的"彩排录像"：服务端让 Angular 把整台戏完整演一遍，将演出结果录制成 HTML 字符串发给用户。浏览器收到录像后，无需等待现场搭台，直接放映给用户看（FCP 极快）；与此同时，真正的演员（Angular 框架）赶来现场，对口型接管表演——这就是水合。

flowchart LR subgraph server["服务端（Node.js + Domino）"] S1["HTTP 请求"] --> S2["renderApplication()"] S2 --> S3["bootstrapApplication<br/>Domino DOM"] S3 --> S4["whenStable()"] S4 --> S5["annotateForHydration()<br/>写入 ngh 属性"] S5 --> S6["TransferState 序列化<br/>→ script 标签"] S6 --> S7["renderToString()<br/>→ HTML 字符串"] end subgraph browser["浏览器端（原生 DOM）"] B1["接收 HTML<br/>FCP 极快"] --> B2["Angular Bundle 加载"] B2 --> B3["bootstrapApplication<br/>原生 DOM"] B3 --> B4["读取 ngh 属性<br/>+ TransferState"] B4 --> B5["locateOrCreate<br/>认领 DOM 节点"] B5 --> B6["绑定事件监听器"] B6 --> B7["水合完成<br/>应用就绪"] end S7 -->|"HTTP 响应"| B1

🔗 这里调用的 bootstrapApplication() 启动链路与第2章完全一致——服务端启动 Angular 的方式和客户端无异，差别只是注入的平台级 provider 不同。

ngServerMode 的传播效应：这个全局布尔值不只是一个简单开关，它在源码中被十数个位置检查，影响从平台提供者注册到 HttpClient 缓存策略的各种行为。例如：TransferState 工厂函数用它判断是否需要从 DOM 读取初始状态；HttpClient 的缓存拦截器用它判断是否需要将响应写入 TransferState；部分动画和浏览器 API 调用在 ngServerMode = true 时被静默跳过，避免 Domino 环境下的不兼容错误。

RenderMode 与混合渲染：v19 引入了 RenderMode 枚举，允许为不同路由指定渲染策略（Server、Client、Prerender）。设置 RenderMode.Client 的路由会在宿主元素上添加 ngcm 属性，客户端水合时检测到该属性后跳过水合、回退到纯客户端渲染——为特定页面（如用户仪表盘）提供了精细控制粒度。

§10.2 服务端渲染流程：renderApplication()

入口函数位于 packages/platform-server/src/utils.ts，整个服务端渲染分为四个阶段：

// packages/platform-server/src/utils.ts（约 L260）
export async function renderApplication(
  bootstrap: (context: BootstrapContext) => Promise<ApplicationRef>,
  options: {document?: string | Document; url?: string; platformProviders?: Provider[]},
): Promise<string> {
  const platformRef = createServerPlatform(options);        // ① 创建服务端平台
  const applicationRef = await bootstrap({platformRef});    // ② 启动应用
  await applicationRef.whenStable();                        // ③ 等待应用稳定
  return await renderInternal(platformRef, applicationRef); // ④ 序列化输出
}

阶段①：创建服务端平台。createServerPlatform() 注册 INITIAL_CONFIG（含初始 HTML 文档字符串和请求 URL），初始化 Domino Document，提供 PlatformState 服务持有 Domino 的 Document 实例。

阶段②：启动应用。与浏览器端走完全相同的初始化链路（DI 容器、组件树、变更检测），只是所有 DOM 操作落在 Domino 上，对上层代码完全透明。

阶段③：等待稳定。whenStable() 等待所有 PendingTask、HTTP 请求、Zone.js 宏任务完成。这是"数据直出"的关键——服务端等 API 数据就绪后再序列化，确保 HTML 中包含完整内容而非加载骨架。

值得注意的是：对于使用 Zoneless + Signals 架构的应用，whenStable() 依赖的是 PendingTasksService 而非 Zone.js 宏任务追踪。resource() API 在内部会注册 PendingTask，确保数据加载完成前 whenStable() 不会提前返回。

阶段④：renderInternal() 序列化，这是水合标注写入的时机：

// packages/platform-server/src/utils.ts（约 L95）
async function renderInternal(platformRef, applicationRef): Promise<string> {
  prepareForHydration(platformState, applicationRef);
  // prepareForHydration 做了三件事：
  // 1. 在 <body> 首位插入 （完整性校验标记）
  // 2. 调用 annotateForHydration(appRef, doc)，给所有组件写 ngh 属性
  // 3. 若有需要重放的事件，注入 JSAction Bootstrap 脚本
  appendServerContextInfo(applicationRef); // 写入 ng-server-context 属性
  // 触发 BEFORE_APP_SERIALIZED 多播回调（TransferState 序列化在此执行）
  return platformState.renderToString();   // Domino DOM → HTML 字符串
}

` 这个注释节点（SSR_CONTENT_INTEGRITY_MARKER）是一道安全阀：CDN 有时会压缩 HTML、删除注释节点，这会让水合算法完全失效。客户端启动时，verifySsrContentsIntegrity()` 检查该标记是否存在——一旦缺失就放弃水合，降级为完整的客户端渲染，而非静默失败。

§10.3 DOM 标注机制：ngh 属性的秘密

想象导演在彩排结束后，给每位演员的剧本标上场景编号——现场演员对口型接管时，能精确知道每句台词对应哪个布景位置。Angular 的 DOM 标注机制做的正是这件事：在服务端序列化阶段，为每个组件的宿主元素写入一个 ngh 属性，作为指向 TransferState 中序列化数据的索引。

标注入口是 annotateForHydration()（packages/core/src/hydration/annotate.ts，L:243），它遍历所有根 LView，对每个组件调用 annotateHostElementForHydration()：

// packages/core/src/hydration/annotate.ts（约 L806）
function annotateHostElementForHydration(element, lView, parentDeferBlockId, context) {
  // 两类组件自动跳过水合标注：
  if (hasI18n(lView) && !isI18nHydrationSupportEnabled()
      || componentUsesShadowDomEncapsulation(lView)) {
    renderer.setAttribute(element, SKIP_HYDRATION_ATTR_NAME, '');
    return null;
  }
  const ngh = serializeLView(lView, parentDeferBlockId, context);
  // 去重：相同结构的视图共用同一索引（*ngFor 列表项可极大节省体积）
  const index = context.serializedViewCollection.add(ngh);
  renderer.setAttribute(element, NGH_ATTR_NAME, index.toString()); // ngh="3"
  return index;
}

serializedViewCollection.add() 通过对 SerializedView 做 JSON.stringify 后比较实现去重——相同的视图描述只存一份，仅记录一个 multiplier（重复次数）。对于 *ngFor 生成的列表，这个优化可以显著减小 TransferState 的体积。

serializeLView() 遍历 TView 中所有 TNode，为每类节点生成不同描述：

节点类型序列化方式字段
Element无（DOM 树可隐式导航）—
ElementContainer（<ng-container>）记录子节点数c
Container（*ngFor 等）序列化所有嵌入视图c
Projection（投影节点）显式路径字符串n
文本节点处理空/相邻文本注释标记

路径压缩算法：当节点无法隐式导航时（如投影打乱了 DOM 顺序），使用压缩路径字符串表示位置。路径由 f（firstChild）和 n（nextSibling）组合，例如 "fn" 表示"取 firstChild 再取 nextSibling"，"fnfn" 表示两次这样的操作。这比完整 XPath 更紧凑，比序号索引更稳健。

⚠️ 文本节点陷阱：浏览器会合并相邻文本节点或丢失空文本节点，导致水合时 DOM 结构与服务端序列化不一致。Angular 用两个特殊注释标记绕过这个问题：`（空文本节点占位符）和 `（文本节点分隔符）。

序列化完成后，所有数据嵌入 HTML 的一个 <script> 标签：

<script id="my-app-state" type="application/json">
{"__nghData__":[{"c":{"0":3},"t":{"1":"t0"}},{"n":{"5":"fn"}}],
 "__nghDeferData__":{"d0":{"s":2,"r":5,"t":[1,{"trigger":3}]}}}
</script>

字段缩写（interfaces.ts）：c=ELEMENT_CONTAINERS，t=TEMPLATES，n=NODES，r=NUM_ROOT_NODES，x=MULTIPLIER，s=DEFER_BLOCK_STATE，i=TEMPLATE_ID，di=DEFER_BLOCK_ID，l=I18N_DATA。

SerializedView 的完整接口定义揭示了序列化的全貌：

// packages/core/src/hydration/interfaces.ts
interface SerializedView {
  [ELEMENT_CONTAINERS]?: {[index: number]: number};  // 'c' - ng-container 子节点数
  [TEMPLATES]?: Record<number, string>;               // 't' - 模板指令索引 → SSR ID
  [CONTAINERS]?: Record<number, SerializedContainerView[]>; // 'c' - ViewContainer
  [NODES]?: Record<number, string>;                   // 'n' - 节点导航路径
  [DISCONNECTED_NODES]?: number[];                    // 'd' - 断连节点索引
  [I18N_DATA]?: Record<number, number[]>;             // 'l' - i18n ICU 活跃 case
  [DEFER_BLOCK_ID]?: string;                          // 'di' - defer 块 ID
}

interface SerializedContainerView extends SerializedView {
  [TEMPLATE_ID]: string;    // 'i' - 模板 SSR ID（如 "t3"）
  [NUM_ROOT_NODES]: number; // 'r' - 视图根节点数
  [MULTIPLIER]?: number;    // 'x' - 重复次数（*ngFor 去重优化）
}

客户端运行时对应的是 DehydratedView——它在 SerializedView 的静态描述基础上，附加了真实 DOM 引用作为导航起点：

// packages/core/src/hydration/interfaces.ts
interface DehydratedView {
  data: Readonly<SerializedView>; // 来自 TransferState 的静态描述（只读）
  firstChild: RNode | null;       // DOM 中该视图的第一个节点（水合导航起点）
  segmentHeads?: {[index: number]: RNode | null}; // ViewContainer 各段起始节点
  disconnectedNodes?: Set<number> | null;          // 断连节点集合（快速查找）
  i18nNodes?: Map<number, RNode | null>;           // i18n 节点索引映射
}

SerializedView 是服务端写、客户端读的静态描述；DehydratedView 是客户端水合过程中的"导航仪"——随着节点逐一被认领，firstChild 等指针会随之推进，直到整个视图水合完毕。

§10.4 水合算法：如何"认领"服务端 DOM

水合的核心设计极为精巧：通过函数指针替换，在不改变渲染指令调用方式的前提下，切换 DOM 操作的行为——从"创建节点"变为"定位并复用节点"。演员们不重新搭台，而是直接走进已有的舞台，在彩排录像对应的位置就位。

启用水合运行时

withDomHydration() 最终调用 enableHydrationRuntimeSupport()（packages/core/src/hydration/api.ts，约 L95）：

// packages/core/src/hydration/api.ts（约 L95）
function enableHydrationRuntimeSupport() {
  if (!isHydrationSupportEnabled) {
    isHydrationSupportEnabled = true;
    enableRetrieveHydrationInfoImpl();            // 启用 ngh 属性读取
    enableLocateOrCreateElementNodeImpl();        // 元素节点：创建 → 定位
    enableLocateOrCreateTextNodeImpl();           // 文本节点同上
    enableLocateOrCreateElementContainerNodeImpl();
    enableLocateOrCreateContainerAnchorImpl();
    enableLocateOrCreateContainerRefImpl();
    enableFindMatchingDehydratedViewImpl();
    enableApplyRootElementTransformImpl();
  }
}

这些 enable* 函数将全局函数指针从"创建模式"切换为"水合模式"。如果应用没有调用 provideClientHydration()，整套水合代码被摇树移除——引入 SSR 功能不带来任何额外客户端包体积。

水合入口：读取 ngh 属性

组件初始化时，retrieveHydrationInfoImpl() 读取宿主元素的 ngh 属性，从 TransferState 中取出对应的 SerializedView，构造 DehydratedView 并存入 LView[HYDRATION]（槽位 6）：

// packages/core/src/hydration/utils.ts
export function retrieveHydrationInfoImpl(rNode, injector, isRootView = false) {
  const nghAttrValue = rNode.getAttribute(NGH_ATTR_NAME); // 读 "3"
  const transferState = injector.get(TransferState);
  const nghData = transferState.get(NGH_DATA_KEY, []);
  const data = nghData[Number(nghAttrValue)];             // 取 nghData[3]
  const dehydratedView: DehydratedView = {
    data,                          // 静态描述（只读，来自服务端序列化）
    firstChild: rNode.firstChild,  // DOM 第一个子节点（水合导航起点）
  };
  rNode.removeAttribute(NGH_ATTR_NAME); // 消费完毕，清除属性
  return dehydratedView;               // 存入 LView[HYDRATION]
}

DehydratedView 就是演员手中的台词本：data 是服务端记录的场景信息，firstChild 是现场布景的起点。

节点认领：locateOrCreateElementNodeImpl()

// packages/core/src/render3/instructions/element.ts（约 L329）
function locateOrCreateElementNodeImpl(tView, lView, tNode, name, index): RElement {
  const isNodeCreationMode = !canHydrateNode(lView, tNode);
  lastNodeWasCreated(isNodeCreationMode);

  if (isNodeCreationMode) {
    return createElementNode(lView[RENDERER], name, getNamespace()); // 创建模式
  }
  // 水合模式：从现有 DOM 定位节点，createElement() 从未被调用
  const hydrationInfo = lView[HYDRATION]!;
  const native = locateNextRNode<RElement>(hydrationInfo, tView, lView, tNode)!;
  ngDevMode && validateMatchingNode(native, Node.ELEMENT_NODE, name, lView, tNode);
  ngDevMode && markRNodeAsClaimedByHydration(native); // 标记已认领（仅开发模式）
  return native; // 直接返回已存在的 DOM 节点引用
}

水合模式下，locateNextRNode() 使用四层降级策略来定位目标节点：

flowchart TD A["locateNextRNode()"] --> B{"i18nNodes Map<br/>有记录?"} B -->|是| C["从 i18nNodes Map<br/>直接取节点"] B -->|否| D{"NODES 字段<br/>有显式路径?"} D -->|是| E["解压路径字符串<br/>按 f/n 组合导航"] D -->|否| F{"是 TView<br/>firstChild 对应节点?"} F -->|是| G["取 DehydratedView<br/>.firstChild"] F -->|否| H["相对导航：基于前一<br/>兄弟节点或父节点定位"] C --> Z["返回现有 DOM<br/>节点引用（不创建）"] E --> Z G --> Z H --> Z

这四层策略覆盖了所有复杂模板结构：i18n、投影、普通元素树、ng-container。

水合后清理：应用稳定时（appRef.whenStable()），cleanupDehydratedViews() 遍历所有 LContainer，删除服务端产生但客户端条件渲染不需要的"孤儿"DOM 节点——它们是条件分支不满足时服务端渲染的内容。

事件监听器绑定：水合阶段认领 DOM 节点后，渲染引擎继续执行 ɵɵlistener 渲染指令，将 Angular 的事件处理函数绑定到已有 DOM 节点上。这是水合的最后一步，也是让"录像"变为"直播"的关键动作——DOM 已就位，现在装上"电路"（事件处理），组件就真正活了。

🔗 LView[HYDRATION] 槽位 6 的存在，以及这里涉及的 TView/LView 数据结构，在第5章有完整的剖析。

§10.5 TransferState：数据的跨环境旅行

TransferState（packages/core/src/transfer_state.ts）是横跨服务端和浏览器端的状态传输桥梁——把它想象成彩排时准备好的道具清单，随剧本一起移交给正式演出的现场演员。它同时承载两类数据：水合必要的 DOM 结构描述（__nghData__），以及任意业务数据（如 API 响应）。

服务端序列化（由 BEFORE_APP_SERIALIZED 令牌触发）：

// packages/platform-server/src/transfer_state.ts
function serializeTransferStateFactory() {
  return () => {
    const content = transferStore.toJson(); // 触发所有 onSerialize 回调
    if (transferStore.isEmpty) return;
    const script = doc.createElement('script');
    script.id = appId + '-state';            // 如 "my-app-state"
    script.setAttribute('type', 'application/json');
    script.textContent = content;            // JSON（< 转义为 \u003C 防 XSS）
    doc.body.appendChild(script);
  };
}

客户端反序列化：TransferState 工厂函数在浏览器端自动从 <script id="my-app-state" type="application/json"> 读取并解析——业务代码无需感知环境差异。

// packages/core/src/transfer_state.ts，TransferState 工厂
factory: () => {
  const transferState = new TransferState();
  if (!ngServerMode) {
    // 浏览器端：从 DOM 中的 script 标签读取数据
    transferState.store = retrieveTransferredState(inject(DOCUMENT), inject(APP_ID));
  }
  return transferState;
}

💡 HttpClient 与 TransferState 的自动协同：当应用配置了 provideHttpClient(withFetch()) 和 SSR 时，Angular 的 HttpTransferCacheInterceptor 会在服务端将 HTTP 响应自动写入 TransferState，客户端直接从缓存读取，不再重复发起相同请求。对于 Signal-based 的数据加载（如 toSignal(httpClient.get('/api/users'))），这个机制完全透明——Signal 的初始值就是服务端已拿到的数据。

如果你需要手动传递某个 Signal 的状态，可以在服务端通过 onSerialize 回调写入，客户端通过 get() 读取：

// 手动将 Signal 状态通过 TransferState 传递
const DATA_KEY = makeStateKey<User[]>('users');

// 服务端（或服务类中，运行于两端）
if (isPlatformServer(this.platformId)) {
  this.transferState.onSerialize(DATA_KEY, () => this.users());
}
// 客户端：直接读取，覆盖 signal 初始值
const serverUsers = this.transferState.get(DATA_KEY, null);
if (serverUsers) this.users.set(serverUsers);

Signals 在 SSR 中：signal 在服务端正常工作（signal.set、computed 均可执行）。SSR 渲染时 Angular 读取 signal 当前值写入 DOM；水合阶段直接认领 DOM 节点，不重新执行读取。signal 的数据恢复依赖 TransferState（通过 HttpClient 自动传递，或手动注册 onSerialize 回调）。

§10.6 ngSkipHydration：优雅地放弃

并非所有组件都适合水合。第三方组件库可能在内部直接操作 DOM，其渲染结果与 Angular 的 TView/LView 描述不符；使用随机 ID 或时间戳的动态内容每次渲染结果都不同，水合时会触发节点不匹配错误（NG0500）。

ngSkipHydration 属性允许将某个组件及其子树整体排除在水合之外——就像在房间门上贴了"此处翻新"标牌，另行处理：


<third-party-chart ngSkipHydration />

服务端：annotateHostElementForHydration() 跳过序列化，不写 ngh 属性，仅保留 ngSkipHydration 属性在宿主元素上。

客户端：locateOrCreateElementNodeImpl() 检测到该属性后进入跳过路径：

// packages/core/src/render3/instructions/element.ts 及 skip_hydration.ts
if (hasSkipHydrationAttrOnTNode(tNode) || hasSkipHydrationAttrOnRElement(native)) {
  if (isComponentHost(tNode)) {
    enterSkipHydrationBlock(tNode); // 设置 TNodeFlags.inSkipHydrationBlock
    clearElementContents(native);   // 清空 SSR HTML 内容（避免与新建 DOM 重叠）
    // 后续渲染指令回到"创建模式"，重新构建该组件子树
  }
}

作用域传播：isInSkipHydrationBlock() 通过遍历 TNode 的祖先链传播跳过标志，确保被跳过的组件的所有子组件也都不尝试水合。

两类组件会被自动标记跳过：使用 ViewEncapsulation.ShadowDom 的组件（Domino 不支持 Shadow DOM），以及包含 i18n 且 isI18nHydrationSupportEnabled() 返回 false 的组件（即未调用 withI18nSupport() 时）——启用了 i18n 水合支持的组件将正常水合，不会自动跳过。

💡 ngSkipHydration 应当是最后手段，不是第一选择。过度使用会让 SSR 的 FCP 提升效果打折——被跳过的组件区域在水合时会清空后重建，期间可能出现内容抖动。如果某个区域有较多组件需要跳过，优先考虑通过 RenderMode.Client 让整个路由走纯客户端渲染，比逐一添加 ngSkipHydration 更清晰。

§10.7 增量水合与 @defer

全量水合的局限在于：即使视口之外的组件，Angular 也要在启动时完整水合整棵组件树，白白消耗主线程时间。增量水合（Incremental Hydration，v19 稳定）改变了这一策略：@defer 块在 SSR 时以完整内容渲染（保障 SEO 可抓取），但被标记为"脱水状态"，客户端按触发条件（视口进入、空闲时、用户交互）按需激活。

// main.ts 或 app.config.ts
bootstrapApplication(AppComponent, {
  providers: [
    provideClientHydration(
      withIncrementalHydration() // 内含 withEventReplay()（必须）
    )
  ]
});

💡 普通水合与增量水合的事件重放配置有所不同，请勿混淆：

// packages/core/src/hydration/api.ts の使い方
// 普通水合 + 事件重放：需要显式添加 withEventReplay()
provideClientHydration(withEventReplay())

// 增量水合（自动包含事件重放，不需要再添加 withEventReplay()）
provideClientHydration(withIncrementalHydration())
// ❌ 错误：provideClientHydration(withIncrementalHydration(), withEventReplay())

withIncrementalHydration() 在内部已调用 withEventReplay()——若再重复添加，事件监听器会被注册两次，造成事件重放的重复执行。

服务端：defer 块脱水序列化

SSR 时，每个 @defer 块以当前状态渲染，但宿主注释节点被标注为 `，块的元数据写入 __nghDeferData__`：

// packages/core/src/hydration/annotate.ts（serializeDeferBlock）
comment.textContent = `ngh=${deferBlockId}`;   // 
context.deferBlocks.set(deferBlockId, {
  [DEFER_BLOCK_STATE]: currentState,   // 渲染状态（主/占位符/加载中/错误）
  [NUM_ROOT_NODES]: rootNodeCount,     // 块根节点数
  [DEFER_HYDRATE_TRIGGERS]: triggers,  // idle/viewport/interaction/timer
  [DEFER_PARENT_BLOCK_ID]: parentId,   // 嵌套块的父块 ID
});

客户端：注册、触发与水合

应用启动（APP_BOOTSTRAP_LISTENER）时，Angular 扫描 DOM 中所有 ` 注释节点，根据 __nghDeferData__ 为每个块设置触发器监听器（IntersectionObserver、requestIdleCallback` 等）。触发水合时，流程从顶层祖先块向下递归：

DehydratedBlockRegistry（packages/core/src/defer/registry.ts）管理所有待水合块的状态，awaitParentBlock() 确保嵌套块按父子顺序水合——子块必须等待父块水合完成，才能接管父块渲染出来的真实 DOM 节点。

// packages/core/src/defer/registry.ts
class DehydratedBlockRegistry {

  get(blockId: string): DehydratedDeferBlock | null;
  cleanup(hydratedBlocks: string[]): void;  // 水合完成后移除注册、清理 JSAction 监听
  awaitParentBlock(topmostParentId: string, callback: Function): void;
}

水合完成后，cleanup() 从注册表中移除已激活的块，同时解绑 JSAction 全局监听器（避免内存泄漏），将触发器控制权移交给 Angular 的常规事件系统。

🔗 增量水合与第9章的懒加载路由形成互补：loadComponent 控制页面级组件的按需加载，@defer 控制单个路由内部组件的按需激活，两者可以叠加使用以实现最细粒度的代码分割。

// app.routes.ts — 路由级懒加载 + 组件内增量水合的组合
export const routes: Routes = [{
  path: 'dashboard',
  loadComponent: () =>
    import('./dashboard/dashboard.component').then(m => m.DashboardComponent),
}];



@defer (on viewport; hydrate on viewport) {
  <analytics-chart [data]="metrics()" />
} @placeholder {
  <div class="chart-skeleton" style="height: 300px" />
}

loadComponent 控制的是页面级按需加载——导航到 /dashboard 时才下载该路由的 JavaScript；@defer 的 hydrate on viewport 确保图表组件只在进入视口时才被水合激活。两种机制叠加，前者减少初始包体积，后者减少水合时的主线程占用，共同实现最细粒度的代码分割。

§10.8 事件重放：交互不丢失

SSR 页面在浏览器展示后，Angular Bundle 仍在下载和解析，水合可能需要数百毫秒。这段时间内用户已经能看到页面，他们会点击按钮、填写表单——如果这些事件丢失，用户会以为操作无效而重复点击，体验彻底崩溃。

事件重放（Event Replay）机制通过三个阶段解决这个问题：

sequenceDiagram participant U as 用户 participant S as SSR HTML participant J as JSAction 脚本 participant A as Angular S->>J: 页面加载时立即执行 J->>J: 在 document.body<br/>注册全局事件监听器 U->>J: 点击（Angular 尚未就绪） J->>J: 事件入队<br/>window._ejsas[appId] Note over U,A: Angular Bundle 加载中... A->>A: bootstrapApplication() + 水合 A->>A: whenStable() A->>J: initEventReplay() J->>A: 读取积压事件队列 A->>A: replayEarlyEventInfos()<br/>按原始顺序重放 A->>U: 事件处理函数正常执行

服务端在 HTML 中注入 JSAction Bootstrap 脚本（Google 内部事件框架）：

<script type="text/javascript" id="ng-event-dispatch-contract">
  /* JSAction 事件捕获库内联代码 */
</script>
<script>
  window.__jsaction_bootstrap(document.body, "my-app", ["click", "input"], []);
</script>

水合完成后，initEventReplay() 从 window._ejsas[appId] 读取积压事件并重放：

// packages/core/src/hydration/event_replay.ts
const initEventReplay = (eventDelegation, injector) => {
  const earlyJsactionData = window._ejsas[appId];
  const eventContract = new EventContract(
    new EventContractContainer(earlyJsactionData.c)
  );
  // 读取所有积压事件，按原始顺序重放
  eventContract.replayEarlyEventInfos(
    getAppScopedQueuedEventInfos(appId)
  );
  clearAppScopedEarlyEventContract(appId); // 清空队列，避免重复执行
};

增量水合中的特殊情况：如果用户点击的正好是一个未水合的 @defer 块，invokeRegisteredReplayListeners() 会先触发块水合，事件进入队列，水合完成后按顺序重放。事件处理函数感知不到任何延迟，用户操作必然生效。

🔍 JSAction 的工程背景：Angular 选用 Google 内部的 JSAction 框架（而非自研方案），这是 Angular 与 Google 基础设施深度融合的体现。JSAction 最初是 Google Search 前端处理"交互前点击"的基础设施，核心优势是体积极小（可内联到 HTML）、无依赖、在任何 JS 框架之前运行。v17 将其引入 Angular 生态。收到事件重放的组件完全感知不到差异——Event 对象的 type、target、currentTarget 与原始事件完全一致。

§10.9 设计思考

为什么不重新渲染？

Angular SSR 早期版本正是重新渲染的：客户端收到 SSR HTML，Angular 启动后清空 innerHTML，重新构建 DOM 树。实现简单，代价是：用户看到内容短暂消失再出现（闪烁），服务端工作完全浪费。DOM 复用水合的实现复杂度远高于重新渲染，代码量多出数千行。Angular 团队选择承担这份复杂性，换来的是：可见内容不闪烁、布局不抖动、Core Web Vitals 持续优良。

函数指针替换的设计哲学

水合代码通过函数指针机制与核心渲染引擎完全解耦：ɵɵelementStart 渲染指令始终调用同一个函数变量（_locateOrCreateElementNode），水合功能的启用只是切换该变量指向的实现。这个设计有两个深层好处：

  • 零成本抽象：不使用水合时，相关代码被摇树移除，不影响客户端包体积
  • 可组合性：增量水合、i18n 水合、事件重放是独立选项，彼此不耦合，按需叠加

🔍 这与第6章分析的 Signals 集成方式如出一辙——Angular 在多处使用函数指针 + 条件启用的模式，将可选功能与核心路径彻底分离。

性能权衡：水合的代价与收益

DOM 复用水合并非没有代价：水合阶段 Angular 需要遍历整棵组件树、读取 TransferState、执行节点认领，这本身会消耗主线程时间（通常在 50~200ms 之间，取决于组件树复杂度）。对于交互极少的纯展示页面，有时直接 CSR 甚至比 SSR + 水合的总体 TTI 更低。工程师需要根据实际 Core Web Vitals 数据来决策，而非一刀切地引入 SSR。

双重渲染检查

ngDevMode 下，validateMatchingNode() 对每个水合节点验证服务端 DOM 节点与客户端期望的节点类型、标签名完全一致。这个检查在生产构建中完全剥离。遇到 NG0500（节点不匹配）错误，通常是第三方库在构造函数中操作了 DOM，或服务端/客户端的时区、语言环境不同导致渲染结果差异。ngSkipHydration 是急救方案，根本解法是消除这种环境差异。

以下是 NG0500 最常见的四类根因及诊断方法：

  1. 第三方组件在 constructor/ngOnInit 中直接操作 DOM：服务端 Domino 与浏览器 DOM 操作结果不一致，渲染节点数量或顺序可能有差异。排查方法：在组件 constructor 添加断点，查看是否有 document.querySelector 等调用。解决：改用 afterNextRender() 将 DOM 操作推迟到浏览器水合后执行，或对该组件添加 ngSkipHydration。
  1. 动态内容（随机值、时间戳）：如 Math.random() 或 new Date().toLocaleDateString()（受时区影响），服务端与客户端的输出字符串不同。诊断：检查模板中是否有 Date/Math.random 等非确定性调用。解决：通过 TransferState 将服务端生成的值传给客户端，确保两端渲染结果完全一致。
  1. i18n 动态内容（ICU 表达式）：多语言复数/选择表达式在 SSR 与 CSR 的处理路径不同，未启用 withI18nSupport() 时相关组件会被自动跳过水合（触发 ngSkipHydration），可能影响 SEO 可抓取性。解决：添加 withI18nSupport() 启用 i18n 水合支持，或接受对相关组件的 ngSkipHydration 标注。
  1. CDN/代理层修改了 HTML：部分 CDN 会压缩或删除 HTML 注释（包括 ` 完整性标记），导致 NG0501（而非 NG0500）。诊断方法：用 curl` 对比直连服务器与经 CDN 的 HTML 响应，确认注释节点是否被保留。解决：在 CDN 配置中禁用 HTML 注释压缩，或使用不删除注释的压缩选项。

从 CSR 到 SSR 的迁移成本

Angular SSR 的一个工程挑战是，现有应用中可能充斥着对 window、document、navigator 的直接引用——这些 API 在 Domino 环境中或缺失或行为不同。Angular 提供了 isPlatformBrowser() 和 PLATFORM_ID 令牌来做环境判断，但更优雅的方案是通过 DI 注入 DOCUMENT 令牌（服务端返回 Domino Document，浏览器端返回原生 Document），将环境差异封装在框架层，业务代码保持统一。

从剧本到巡演：全书知识体系的回顾

第1章展开了整座剧院的鸟瞰图：二十余个 package 分三层叠落，编译时与运行时各司其职，monorepo 将所有角色统一在同一剧本之下。那是一张蓝图，让我们在进入任何一个细节之前，先知道自己身处何处，以及这座建筑为什么非得这样搭才稳。

第2章拉开了帷幕。bootstrapApplication() 的一声令下，平台创建、根注入器初始化、根组件首次渲染——这一系列动作链的每一环都环环相扣，没有一步是多余的，也没有一步可以省略。

第3章进入道具部门。依赖注入系统维护着两棵并行的注入器树：模块注入器链向上溯源，节点注入器随组件树生长。每一个 inject() 调用都是一次精确的寻访，沿树向上，找到就停。框架以这套机制将"谁提供"与"谁消费"彻底解耦，让每个角色只需关心自己需要什么，不必知道它从何而来。

第4章翻译了剧本。模板编译器将 HTML 字符串变为一串渲染指令序列——ɵɵelement、ɵɵproperty、ɵɵlistener。人写的是声明式模板，机器执行的是命令式函数调用。这道翻译发生在构建时，运行时不再保留任何模板字符串，也不再需要任何解析开销。

第5章搭建了舞台。LView 是运行时快照——每个组件实例的真实状态；TView 是蓝图——同一组件类的所有实例共享的静态描述。两者一动一静，以最经济的内存布局支撑起整棵渲染树，第10章的水合算法正是在这两块数据结构之上运作的。

第6章描绘了导演巡场。变更检测从根组件出发，沿组件树深度优先走遍每个节点，检查绑定值是否改变，决定是否更新 DOM。调度器控制时机，OnPush 策略控制粒度，这是框架在正确性与性能之间反复权衡后画出的那条线。

第7章带来了精准传呼系统。Signals 以生产者—消费者图替代了 Zone.js 的全局广播，computed 只在依赖变化时重算，effect 只在真正读过的 signal 改变时触发。从粗粒度的"什么都可能变"到细粒度的"这个确实变了"，是框架响应式模型的一次根本性升级，也是 Zoneless 架构的理论基础。

第8章描摹了演员档案。ComponentDef（ɵcmp）是组件完整的运行时身份证：模板函数、属性绑定、输入输出、生命周期钩子、封装策略——全部以静态字段编码，供渲染引擎在启动时直接读取，无需反射、无需运行时解析。这是 AOT 编译哲学在数据结构层面的体现。

第9章编排了场景调度。路由器是一台状态机，NavigationStart 到 NavigationEnd 之间经历守卫检查、数据预加载、组件懒加载、视图切换。每一次 URL 变化都是一次完整的换幕，而换幕的代价被精心控制在最小范围——只加载当前场景真正需要的代码。

第10章则带剧团走上了巡回演出。服务端先演完整场，将成果录制进 HTML；客户端收到录像，不推翻、不重建，只是悄悄走进现场，认领每一块布景，装上事件电路，让"录像"在不知不觉间变成了"直播"。水合的本质不是从零开始，而是接过接力棒，继续往前跑。

十章下来，变化的是场景，不变的是一个贯穿始终的设计哲学：把复杂性留在框架内部，把简洁性暴露给使用者；把可选功能藏在函数指针后面，把核心路径打磨到极简。Angular 从未试图掩盖它的复杂，但它用层层抽象，让每一层的复杂性都只在它应该存在的地方存在。

小结

本章是全书的终章，也是所有前序内容汇聚的地方。SSR 依赖第2章的应用启动机制在服务端运行，水合认领 DOM 时操作的是第5章的 LView/TView 数据结构，增量水合触发的变更检测更新走的是第6章的调度路径，Signal-based 数据恢复借助第7章的响应式原语。把这些拼在一起，你就拥有了 Angular 全栈渲染的完整图景。

本章梳理了 Angular SSR 和水合的完整机制：

  • 服务端：renderApplication() 在 Domino DOM 上运行完整 Angular 应用，等待稳定后通过 ngh 属性和 TransferState 将 DOM 结构信息嵌入 HTML
  • 客户端水合：函数指针替换使渲染指令进入"定位模式"，locateNextRNode() 用四层策略认领服务端 DOM 节点，Angular 组件树直接与现有 DOM 绑定，不创建任何新节点
  • TransferState：同时承载水合元数据（__nghData__）和业务数据，HttpClient 自动利用它避免重复请求，对 Signal-based 数据加载透明
  • ngSkipHydration：为第三方组件和动态内容提供逃生门，自动应用于 ShadowDom 和 i18n 组件（未启用 withI18nSupport() 时）
  • 增量水合：@defer 块以脱水状态存在，按触发条件按需激活，与懒加载路由互补实现最细粒度代码分割
  • 事件重放：JSAction 在水合完成前捕获用户交互，水合后精确重放，保证零事件丢失

⚠️ 常见生产问题速查：遇到 NG0500（水合节点不匹配）检查服务端/客户端渲染差异（时区、随机值、第三方库 DOM 操作）；遇到 NG0501（找不到水合标记 `）检查 CDN 是否删除了 HTML 注释；遇到水合后事件无响应，确认是否启用了 withEventReplay()`（增量水合模式下默认包含，普通水合模式下需单独添加）。

至此，全书十章的旅程告一段落。从第1章的宏观架构到第10章的全栈渲染，我们始终沿着"源码是最好的文档"这条线索，一层一层拨开 Angular 框架的面纱。框架还在演进，但理解了这些核心机制，你就拥有了持续学习的地基。

思考题

Q1：打开 packages/core/src/hydration/node_lookup_utils.ts，观察 locateNextRNode() 的四层定位策略。当 Angular 模板使用 ng-content 投影内容时，被投影的节点为什么必须走"显式路径"（NODES 字段），而不能用隐式的相邻节点导航？（提示：思考投影是否改变了节点在 DOM 树中的物理位置，以及 TView 中节点的逻辑顺序与 DOM 树中节点的物理顺序是否一致。）

Q2：你的应用中有一个 <date-display> 组件，它在模板里直接调用 new Date().toLocaleDateString() 显示当前日期。服务端在北京运行（UTC+8），一位纽约用户（UTC-5）访问。水合时会发生什么？ngDevMode 下会抛出什么错误？应该如何修复这类"环境依赖"问题？

Q3（实战）：在 Zoneless + Signals 架构的应用中，你用 resource() API 加载用户列表，并通过 provideHttpClient(withFetch()) 启用了 SSR 缓存。请追踪一次完整的 SSR + 水合过程：这个 resource() 的 Signal 状态（isLoading、value）在服务端经历了哪些变化？水合时经历了哪些变化？哪个具体机制让客户端 resource() 直接拿到了服务端已加载的数据，而没有再次发起 HTTP 请求？

下一步学习路径

掌握了 Angular SSR 与水合的源码机制之后，以下六个方向值得深入探索：

  1. 生产环境 SSR 运维：围绕 Node.js 服务部署与容器化（Docker/Kubernetes），重点关注优雅降级策略——当服务端渲染超时或出错时如何平滑回退到纯客户端渲染（CSR Fallback），以及追踪 TTFB、渲染成功率、错误率等关键监控指标。
  1. Core Web Vitals 深度优化：使用 PageSpeed Insights 和 Chrome UX Report（CrUX）追踪真实用户的 LCP、CLS、INP 数据，量化 SSR + 水合带来的改善幅度，找出仍在拖累指标的慢组件并针对性优化。
  1. 增量水合的高级场景：探索 @defer (hydrate on interaction) 与复杂表单、大型图表组件结合时的精细分片策略——在保证 SEO 可抓取的前提下，将首屏水合工作量压缩到极致。
  1. Zoneless + Signals 完整迁移：追踪从 Zone.js 应用迁移到 provideExperimentalZonelessChangeDetection() 的完整路径，理解哪些副作用代码依赖 Zone.js 的自动包裹、如何逐步替换为 Signals 驱动的显式更新。
  1. Angular DevTools 调试：利用 Angular DevTools 的 Profiler 分析水合性能瓶颈，结合 Node.js 调试器在 SSR 阶段断点追踪 renderApplication() 各阶段耗时，定位慢组件或阻塞稳定的异步任务。
  1. 参与 Angular 开源社区：订阅 Angular RFC（github.com/angular/angular/discussions），关注 packages/core/src/hydration/ 的变更日志，尝试向该目录提交你的第一个 PR——修复一个 TODO 注释，或补充一个边界测试用例。
