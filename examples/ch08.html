第8章：组件模型——从装饰器到运行时对象

你写的 @Component({ selector: 'app-root', template: '...' }) 最终变成了什么？

在第1章的剧院比喻里，我们把编译器比作"剧本作者"——它提前把你的声明式模板翻译成一份供演员（render3 运行时）照单执行的指法谱。但有一个问题没有充分回答：那份"剧本"长什么样？它存在哪里？运行时拿到它之后，又是如何决定何时调用 ngOnInit，何时触发 ngAfterViewInit 的？

答案就藏在每个组件类上的一个静态字段里：ɵcmp。

这个字段不是你写的，是编译器生成的。它的类型是 ComponentDef<T>——Angular 运行时对一个组件的完整身份证。Angular 真正执行的不是装饰器，而是编译器从装饰器里提炼出来的这份静态描述。一旦你理解了 ComponentDef 的结构，你就拥有了读懂 Angular 任何组件行为的钥匙。

本章沿着一个组件的"一生"展开：从 @Component 装饰器，到编译器输出的 ComponentDef；从 LView 被创建的那一刻，到每个生命周期钩子被调用的精确时机；从 @Input() 的属性赋值，到内容投影的链表匹配算法。这是一段让装饰器"开口说话"的旅程。

ComponentDef——组件的运行时身份证

ComponentDef<T> 继承自 DirectiveDef<T>，两者合计约 42 个字段，构成 Angular 运行时对一个组件的完整描述。它挂载在组件类的 ɵcmp 静态属性上，由编译器在构建期通过 ɵɵdefineComponent() 生成。

源码位置：packages/core/src/render3/interfaces/definition.ts

关键字段一览

字段来源用途
typeDirectiveDef组件类本身（同时作为 DI token）
selectorsDirectiveDefCSS 选择器列表，用于模板匹配

ɵcmp 是静态字段，而非实例字段。这意味着同一个组件类的所有实例共享同一份 ComponentDef——实例的可变状态（绑定值、DOM 节点引用）存在各自的 LView 里，ComponentDef 只描述"这类组件长什么样"，不描述"这个实例现在的状态"。

编译器如何生成 ɵcmp

编译器的入口是 compileComponentFromMetadata()（packages/compiler/src/render3/view/compiler.ts:188）。它把 @Component 元数据转化为一个 DefinitionMap，最终输出对 ɵɵdefineComponent() 的调用：

// 编译器生成，非手写
// 源文件路径：packages/core/src/render3/definition.ts（运行时部分）
class MyComponent {
  static ɵcmp = /* @__PURE__ */ ɵɵdefineComponent({
    type: MyComponent,
    selectors: [['app-my']],
    inputs: {
      title: [0 /* InputFlags.None */, 'title'],
      count: [1 /* InputFlags.SignalBased */, 'count'],
    },
    outputs: { clicked: 'clicked' },
    standalone: true,
    features: [ɵɵNgOnChangesFeature],  // 有 ngOnChanges 时由编译器注入
    decls: 3,
    vars: 1,
    template: function MyComponent_Template(rf, ctx) {
      if (rf & 1) { /* 创建阶段：ɵɵelementStart, ɵɵtext ... */ }
      if (rf & 2) { /* 更新阶段：ɵɵproperty, ɵɵtextInterpolate ... */ }
    },
    encapsulation: ViewEncapsulation.Emulated,
    changeDetection: ChangeDetectionStrategy.OnPush,
  });
}

/* @__PURE__ */ 注释是关键。它告诉打包工具（Terser / esbuild）：如果这个类没有被引用，可以安全地将整个 ɵɵdefineComponent({...}) 调用树摇掉。这是 Angular 摇树优化（tree-shaking）的基础。

运行时的 ɵɵdefineComponent() 接收这份配置，执行 initFeatures(def) 遍历 features 数组、计算唯一 ID、填充 directiveDefs 和 pipeDefs，最终返回一个不可变的 ComponentDef 对象。

🔗 交叉引用：tView 字段初始为 null，在组件首次实例化时才填充。TView 与 LView 的分离设计已在第5章详细分析。

有了身份证，运行时才知道"这个组件如何工作"。接下来的问题是：当这个组件被创建、检测变更、最终销毁时，它的生命周期钩子是在哪个精确时刻被调用的？

生命周期钩子的调用时序

这是本章最值得细读的一节。Angular 的生命周期钩子不是随意触发的——它们的调用顺序由 refreshView() 的执行结构严格决定。

钩子存储在 TView 中

Angular 在组件首次渲染时，通过 registerPreOrderHooks() 和 registerPostOrderHooks() 将钩子函数扫描并注册到 TView 的几个数组里：

源码位置：packages/core/src/render3/hooks.ts

TView 属性存储的钩子执行场景
preOrderHooksngOnChanges、ngOnInit、ngDoCheck首次创建
preOrderCheckHooksngOnChanges、ngDoCheck后续每次检测
contentHooksngAfterContentInit、ngAfterContentChecked首次创建
contentCheckHooksngAfterContentChecked后续每次检测
viewHooksngAfterViewInit、ngAfterViewChecked首次创建
viewCheckHooksngAfterViewChecked后续每次检测
destroyHooksngOnDestroy视图销毁

这里有一个精妙的设计：数组里的元素值如果是负数（存储的是 -directiveIndex），就代表 Init 钩子（只执行一次）；正数（存储的是 directiveIndex）代表 Check 钩子（每次变更检测都执行）。ngOnInit 只进入 preOrderHooks，而不进入 preOrderCheckHooks——这保证了它永远只运行一次。

refreshView 的执行顺序

变更检测的核心函数 refreshView()（packages/core/src/render3/instructions/change_detection.ts:225）按固定顺序执行：

1. 模板执行（创建或更新 DOM 节点）
2. 前序钩子：ngOnChanges → ngOnInit → ngDoCheck
3. 运行当前视图的 effect
4. 嵌入视图（ng-if/ng-for 生成的视图）变更检测
5. 内容查询刷新（ContentChild/ContentChildren）
6. 内容钩子：ngAfterContentInit → ngAfterContentChecked
7. 宿主绑定 OpCodes 执行
8. 子组件变更检测（递归进入子组件的 refreshView）
9. 视图查询刷新（ViewChild/ViewChildren）
10. 视图钩子：ngAfterViewInit → ngAfterViewChecked

步骤 8（子组件变更检测）发生在步骤 10（父组件视图钩子）之前——这决定了父子嵌套时的钩子顺序。

父子嵌套的精确时序

假设组件结构为 Parent 包含 Child，首次渲染时的钩子调用如下：

sequenceDiagram participant P as Parent participant C as Child Note over P: refreshView() 开始 P->>P: constructor P->>P: ngOnChanges（若有 @Input） P->>P: ngOnInit P->>P: ngDoCheck Note over P: 步骤 8：子组件变更检测 P->>C: 进入 Child 的 refreshView() C->>C: constructor C->>C: ngOnChanges C->>C: ngOnInit C->>C: ngDoCheck C->>C: ngAfterContentInit C->>C: ngAfterContentChecked C->>C: ngAfterViewInit C->>C: ngAfterViewChecked C-->>P: Child 检测完成 Note over P: 步骤 10：父组件视图钩子 P->>P: ngAfterContentInit P->>P: ngAfterContentChecked P->>P: ngAfterViewInit P->>P: ngAfterViewChecked

图例：P->>P 表示组件自身同步调用，P->>C 表示父调用子，C-->>P 表示子异步返回父。

规律很清晰：前序钩子从外到内（父先于子），视图钩子从内到外（子先于父）。

当组件结构更深（如三层嵌套 Parent → Child → GrandChild）时，GrandChild 的 ngAfterViewInit 最先完成，Parent 的 ngAfterViewInit 最后完成——因为只有当所有子孙的视图都"就绪"了，父组件的视图才算真正完整。

后续变更检测的时序差异

上图描述的是首次渲染的完整钩子序列。从第 N 次（N > 1）变更检测开始，Init 类钩子不再执行，序列缩减为：

ngOnChanges（仅当输入引用发生变化）→ ngDoCheck → ngAfterContentChecked → ngAfterViewChecked

ngOnInit、ngAfterContentInit、ngAfterViewInit 各只执行一次——由 LView FLAGS 中的状态机保证（详见下文"Init 钩子的单次执行保证"）。这个差异在性能调优时尤为关键：ngDoCheck 与 ngAfterViewChecked 在每次检测中都会执行，是性能敏感的钩子，应避免在其中放置耗时的计算或 DOM 操作。

两个最容易踩的误区

误区一：ngOnInit 等于 constructor。这两者执行时机不同：constructor 在 Angular 注入系统创建实例时触发（可能比 DOM 创建早很多），而 ngOnInit 在首次变更检测期间执行，此时输入属性（@Input）已经被赋值完毕。如果你在 constructor 里读取 @Input 的值，通常会得到 undefined。

误区二：ngOnChanges 只在值"变化"时触发。更准确的说法是：只要父组件模板里绑定的表达式在本次变更检测中产生了与上次不同的引用或原始值，Angular 就会调用 ngOnChanges。对于对象和数组，如果你修改了其内部属性而没有创建新引用，ngOnChanges 不会被触发。

⚠️ 注意：ngAfterViewInit 才是"DOM 就绪"的信号。如果你需要访问 @ViewChild 查询到的 DOM 元素或子组件实例，ngOnInit 里拿到的通常是 undefined——正确的时机是 ngAfterViewInit。

📝 练习：在一个组件同时实现 constructor、ngOnChanges、ngOnInit、ngAfterViewInit，并在每个钩子里打印时间戳和当前 @Input 的值。用 Angular DevTools 的性能面板观察第1次与第2次变更检测期间被调用的钩子有何不同。

Init 钩子的单次执行保证

Init 类钩子（ngOnInit、ngAfterContentInit、ngAfterViewInit）如何保证只执行一次？答案是 LView FLAGS 字段里维护的一个状态机：

OnInitHooksToBeRun
  → AfterContentInitHooksToBeRun
    → AfterViewInitHooksToBeRun
      → InitPhaseCompleted

callHook() 函数在调用 Init 钩子前会检查当前 LView 处于哪个阶段，以及该阶段已调用的 Init 钩子数量。两个条件同时满足才执行，并在执行后更新计数器。这三重保护确保了每个 Init 钩子在视图的整个生命周期中只被调用一次。

🔍 深入：callHookInternal() 在执行每个生命周期钩子前会调用 setActiveConsumer(null)，暂停 Signals 响应式追踪。这意味着在钩子函数里读取 signal 值不会建立响应式依赖——这是有意的设计，防止钩子被 signal 变更意外重新触发。

@Input() 与 ngOnChanges

编译输出：inputs 映射表

当你写 @Input() title: string，编译器会在 ComponentDef.inputs 中生成一条映射：

// packages/core/src/render3/interfaces/definition.ts（第 119 行附近）
// inputs 的值类型：[混淆后的私有属性名, InputFlags, 变换函数 | null]
inputs: {
  title: [0 /* InputFlags.None */, 'title', null],
}

三元组的含义：第一个值是混淆后的私有属性名（用于访问实例字段，在生产构建中会被混淆以缩短体积），第二个是 InputFlags 位掩码，第三个是可选的变换函数（来自 @Input({ transform: ... })）。

InputFlags 是一个位掩码枚举，具体定义如下：

// packages/core/src/render3/interfaces/input_flags.ts
export enum InputFlags {
  None                       = 0,
  SignalBased                = 0b001,  // 值为 1：信号输入（input()）
  HasDecoratorInputTransform = 0b010,  // 值为 2：有 @Input({ transform }) 函数
  Required                   = 0b100,  // 值为 4：必需输入（input.required()）
}

位掩码可以组合使用。例如 SignalBased | Required（值为 5）表示必需的信号输入，对应 input.required<T>()；HasDecoratorInputTransform | SignalBased（值为 3）表示带有 transform 函数的信号输入。

这些标志位是 writeToDirectiveInput() 内部分支处理的依据，也是思考题第1题的背景知识。

输入赋值的完整链路

从父组件模板 <child [title]="name"> 到子组件属性赋值，要经过五个环节：

① ɵɵproperty('title', name)        ← 父模板更新阶段
② bindingUpdated()                  ← 检测值是否变化
③ setAllInputsForProperty()         ← 查找匹配的指令输入
④ writeToDirectiveInput()           ← 解构三元组，处理变换
⑤ applyValueToInputField()          ← 实际赋值（属性 or 信号）

writeToDirectiveInput()（packages/core/src/render3/instructions/write_to_directive_input.ts）是这条链路的核心。如果组件有 ngOnChanges，编译器会通过 ɵɵNgOnChangesFeature 将 def.setInput 替换为 ngOnChangesSetInput，在每次赋值时同时构造 SimpleChange 对象并存入实例上的 __ngSimpleChanges__ 暂存区。待变更检测执行到前序钩子阶段，再统一读取暂存区、调用 ngOnChanges(changes)，然后把 current 合并到 previous。

firstChange 的判断非常直接：previous === EMPTY_OBJ——首次赋值时 previous 是一个空对象哨兵，此时 firstChange 为 true。

signal-based input() vs 传统 @Input()

Angular v17 引入了函数式的信号输入 API：

// packages/core/src/authoring/input/input.ts
export class ProductCard {
  // 传统装饰器——普通属性，每次检测都可能触发 ngOnChanges
  @Input() productId: string = '';

  // 信号输入——可调用函数，读取时自动注册响应式依赖
  productName = input<string>('');

  // 必需信号输入——未传值时运行时抛出 NG0XXX 错误
  price = input.required<number>();
}

信号输入的底层是一个 InputSignalNode，InputFlags.SignalBased = 1 被设置在 flags 上。读取 this.productName() 时，producerAccessed(node) 会在当前响应式上下文中注册依赖——这使得 computed 和 effect 能响应输入变更，而无需 ngOnChanges。

input.required() 的运行时检查是通过一个哨兵值 REQUIRED_UNSET_VALUE 实现的：信号初始值被设为这个哨兵，当组件在没有父绑定的情况下被渲染时，第一次读取会抛出 RuntimeError。编译时检查则由 TypeScript 的类型系统完成——input.required<T>() 的返回类型不包含 undefined，模板编译器会确保绑定表达式类型匹配。

💡 提示：信号输入不触发 ngOnChanges。如果你同时使用信号输入和 ngOnChanges，Angular 不会为信号输入的变更调用 ngOnChanges——两套系统是独立的，请用 effect() 替代。

⚠️ 注意：混合使用 @Input() 与 input() 的陷阱

当组件同时使用 @Input() 装饰器和 input() 函数时，两套变更感知机制互相独立：

- ngOnChanges 的 changes 参数只包含 @Input() 装饰器声明的属性变更，input() 信号输入的变更不会出现在其中。
- 这意味着若你将某个输入从 @Input() 迁移为 input()，依赖 ngOnChanges(changes) 做条件判断的代码会静默失效——不会报错，逻辑却不再触发。
- 推荐做法：在过渡期，用 effect() 替代 ngOnChanges 监听信号输入的变更；待全部输入迁移完成后，移除 ngOnChanges 方法，编译器将不再注入 ɵɵNgOnChangesFeature，减少一层运行时开销。

@Output() 与 EventEmitter

outputs 字段与事件绑定

@Output() 装饰器在 ComponentDef.outputs 中生成一条简单映射：

// 压缩属性名 → 公开事件名
outputs: { clicked: 'clicked' }

当父模板绑定 (clicked)="handler($event)" 时，Angular 会通过这张映射表找到对应的 EventEmitter 实例，订阅它的事件流。

EventEmitter 的本质

EventEmitter 继承自 RxJS Subject，这是它与普通 Subject 的主要差异在于多了一层与 Angular 运行时的集成：

// packages/core/src/event_emitter.ts（第 114 行附近）
class EventEmitter_ extends Subject<any> {
  emit(value?: any) {
    const prevConsumer = setActiveConsumer(null); // 暂停信号追踪
    try {
      super.next(value);  // 委托给 RxJS Subject
    } finally {
      setActiveConsumer(prevConsumer);
    }
  }
}

⚠️ 注意：有些开发者会用 Subject 替代 EventEmitter 直接作为 @Output 值暴露。这技术上可以工作，但有两个隐患：Subject 没有 emit() 方法（需要用 next()），更严重的是外部调用者可以拿到 Subject 并调用 complete() 或 error()，破坏事件流。Angular 团队建议始终使用 EventEmitter。

output() 信号输出

v17 同步引入了函数式输出 API，底层与 RxJS 完全无关：

// packages/core/src/authoring/output/output_emitter_ref.ts
export class ProductCard {
  // 传统：依赖 RxJS Subject
  @Output() priceChange = new EventEmitter<number>();

  // 信号输出：独立实现，体积更小
  priceUpdate = output<number>();
}

OutputEmitterRef 用一个普通数组 listeners 管理订阅，通过 DestroyRef 实现自动清理。它不依赖 RxJS，也不支持管道操作符——如果你确实需要对输出流使用 RxJS 操作符，outputToObservable() 工具函数可以将 OutputRef 适配为 Observable。

内容投影

🔗 交叉引用：内容投影的比喻已在比喻注册表中注册——子组件就像一块预留了布景槽的舞台，父组件提供的内容正好插进对应的槽里。

两条渲染指令的协作

内容投影由两条渲染指令（instruction）配合完成：

  • ɵɵprojectionDef()：在组件初始化阶段调用一次，扫描父组件传入的子节点，按 select 选择器分配到各投影槽，构建 TNode 链表。
  • ɵɵprojection()：在模板每个 <ng-content> 的位置调用，从对应槽中取出链表头，遍历并将 TNode 插入到当前位置的 DOM 中。
// packages/core/src/render3/instructions/projection.ts（第 96 行附近）
// ɵɵprojectionDef 构建的数据结构：
// componentNode.projection = [slot0Head, slot1Head, slot2Head]
//                               ↓           ↓
//                            TNode_A     TNode_C
//                               ↓ (projectionNext)
//                            TNode_B

每个投影槽是一条通过 projectionNext 指针串联的 TNode 单向链表。这种结构意味着投影内容没有被复制，只是被"重新排列"了引用——DOM 节点仍然只有一份。

多槽投影与选择器匹配

当模板中有多个带 select 的 <ng-content> 时：


<header><ng-content select=".title"></ng-content></header>
<main><ng-content select=".body"></ng-content></main>
<footer><ng-content></ng-content></footer>

编译器将选择器信息编译进 ɵɵprojectionDef() 的参数，匹配算法（matchingProjectionSlotIndex()）遵循三条规则：

  1. 若节点有 ngProjectAs 属性，以该属性值作为选择器匹配目标（允许运行时控制投影归属）
  2. 否则对节点的标签名、类名、属性名做 CSS 选择器匹配
  3. 首个匹配的槽"捕获"该节点——不会进入多个槽；无匹配则进入通配符（默认）槽

投影内容的生命周期归属

这是一个经常令人困惑的问题：被投影的内容，其生命周期钩子属于父组件还是子组件？

答案是父组件。被投影内容的 ngOnInit、ngOnDestroy 均在父组件的变更检测上下文中执行。ngAfterContentInit 是子组件的钩子，在"内容投影完成后"触发——这里的"完成"指 ɵɵprojectionDef() 完成分配，而不是父组件的生命周期完成。被投影内容对子组件的 viewProviders 也是不可见的，只能访问父组件 providers 中注册的服务。

Angular v17 为 <ng-content> 引入了后备内容（fallback content）支持：当没有任何内容被投影到该槽时，显示 <ng-content> 标签内部的内容。这对"可选插槽"的场景很有用。

宿主绑定与宿主指令

hostBindings 函数

@HostBinding() 和 @HostListener() 最终会被编译为 ComponentDef.hostBindings 字段——一个接收 rf（RenderFlags）和 ctx（指令实例）的函数：

// 编译器生成，非手写
// packages/core/src/render3/interfaces/definition.ts（第 568 行）
hostBindings: function MyComponent_HostBindings(rf, ctx) {
  if (rf & RenderFlags.Create) {
    // 注册宿主监听器
    ɵɵlistener('click', function() { return ctx.onClick(); });
  }
  if (rf & RenderFlags.Update) {
    // 更新宿主属性绑定
    ɵɵhostProperty('class.active', ctx.isActive);
    ɵɵattribute('aria-label', ctx.label);
  }
}

宿主绑定不是自由执行的——它们被编码为 TView.hostBindingOpCodes 数组，在 refreshView() 的步骤 7（内容钩子之后、子组件检测之前）统一执行。这保证了宿主绑定在组件内容准备好之后、但在子组件开始渲染之前完成。

hostDirectives：组合而非继承

hostDirectives 让一个组件或指令在被选择器匹配时，自动附加其他指令的能力：

@Component({
  selector: 'app-button',
  hostDirectives: [
    { directive: CdkButton, inputs: ['disabled'], outputs: ['cdkButtonClick: click'] }
  ]
})
export class AppButton {}

底层实现是 ɵɵHostDirectivesFeature（packages/core/src/render3/features/host_directives_feature.ts）。它在 ComponentDef.features 数组里注册一个特性函数，该函数将宿主指令的定义存入 def.hostDirectives。匹配阶段，resolveHostDirectives() 按优先级顺序（先宿主指令，后组件本身）依次实例化并初始化。

宿主指令有两个硬性约束：必须是 standalone（不能来自 NgModule），且不能再嵌套 hostDirectives（防止无限递归）。

这种"指令组合"的设计比继承更灵活——你可以把通用交互行为（焦点管理、无障碍属性、键盘响应）封装成独立指令，然后通过 hostDirectives 组合进任意组件，而不需要改变组件的继承链。

Standalone 组件

standalone: true 对运行时的影响

从 Angular v19 起，standalone 的默认值是 true（definition.ts:633）。这不只是一个标志位的变化，它改变了组件的依赖解析路径。

NgModule 组件的依赖（可用的指令和 pipe）来自所在模块的声明区，是模块级别的共享作用域。Standalone 组件则把依赖直接写在 @Component({ imports: [...] }) 里，编译器将 imports 列表编译到 ComponentDef.dependencies。

在 ɵɵdefineComponent() 中，这个差异产生了两条不同的分支：

// packages/core/src/render3/definition.ts（第 355 行附近）
dependencies: (baseDef.standalone && componentDefinition.dependencies) || null,
getStandaloneInjector: baseDef.standalone
  ? (parentInjector) =>
      parentInjector.get(StandaloneService).getOrCreateStandaloneInjector(def)
  : null,  // NgModule 组件此处为 null

StandaloneService 的注入器缓存

当 standalone 组件首次被创建时，Angular 通过 getStandaloneInjector 闭包调用 StandaloneService.getOrCreateStandaloneInjector()：

// packages/core/src/render3/standalone_service.ts（第 21 行附近）
getOrCreateStandaloneInjector(componentDef: ComponentDef<unknown>) {
  if (!this.cachedInjectors.has(componentDef)) {
    const providers = internalImportProvidersFrom(false, componentDef.type);
    const standaloneInjector =
      providers.length > 0
        ? createEnvironmentInjector([providers], this._injector, `Standalone[...]`)
        : null;  // 无 providers 时返回 null，避免创建空注入器
    this.cachedInjectors.set(componentDef, standaloneInjector);
  }
  return this.cachedInjectors.get(componentDef);
}

缓存以 ComponentDef 为键——同一类型的多个组件实例共享一个注入器，而非每个实例创建一个。StandaloneService 本身实现了 OnDestroy，在父注入器销毁时自动销毁所有缓存的子注入器。

编译器优化：DomOnly 模式

如果一个 standalone 组件的 imports 数组里没有任何指令或组件依赖（只有 providers），编译器会检测到 !meta.hasDirectiveDependencies，使用 TemplateCompilationMode.DomOnly 生成更精简的模板代码——省去了指令匹配的运行时开销。

设计思考

为什么 ComponentDef 是静态字段而非实例属性？

想象一个应用里有一个列表，渲染了 500 个 <product-card> 组件。如果 ComponentDef 是实例属性，那 500 个实例就有 500 份完全相同的模板函数、选择器、封装策略——这是纯粹的内存浪费。静态字段让所有实例共享同一份描述，而实例的可变状态（DOM 引用、绑定值）存在各自的 LView 里。TView 与 LView 的分离思路贯穿了整个 render3 设计（第5章详述），ComponentDef 的静态字段是同一设计哲学在组件层面的体现。

生命周期钩子为什么不用 Observable 或 Signal？

这是一个取舍问题。Observable（如 RxJS）会引入订阅管理的复杂性，并要求开发者手动取消订阅；Signal 依赖是惰性追踪的，而生命周期回调需要命令式的确定性调用顺序，不适合"只在被读取时触发"的响应式模型。当前的接口方法设计有一个显著优势：运行时可以通过检查 prototype 上是否存在该方法来决定是否注册钩子，不存在的钩子零运行时成本。Angular 选择了简单和可预测性，代价是 API 稍显"过时"——事实上，Angular 团队已在探索用 DestroyRef、afterNextRender() 等函数式 API 替代部分生命周期钩子的场景。

Standalone 组件的推出背景

NgModule 最初的设计目标是两个：提供模板编译作用域（确定模板里可以用哪些指令），以及提供 DI 作用域（确定组件树里哪些服务可用）。问题是这两个职责耦合在一起，导致简单组件也必须挂靠一个模块，测试时要搭建大量脚手架。Ivy 的局部编译原则（Locality Principle，ch01）让每个组件的编译产物自给自足——不再需要模块上下文。Standalone 是这一原则的自然延伸：组件直接声明自己的依赖，模块只是一个可选的组织层，而非强制的架构约束。

小结

本章沿着"装饰器 → ComponentDef → LView 生命周期"的主线，拆解了 Angular 组件模型的运行时基础。几个值得记住的关键点：

  • ɵcmp 是共享的、不可变的身份证，同类型所有实例共用；可变状态存在 LView 里
  • 生命周期钩子的顺序由 refreshView() 的结构决定：前序钩子外到内，视图钩子内到外；ngAfterViewInit 才是 DOM 就绪的真正信号；首次渲染后 Init 类钩子不再执行
  • @Input() 赋值经过五个环节，NgOnChangesFeature 在链路中拦截赋值来构造 SimpleChanges；信号输入绕过这套机制，直接接入响应式图
  • 内容投影的本质是 TNode 链表的重新排列，投影内容的生命周期归属于父组件
  • Standalone 组件用 StandaloneService 按 ComponentDef 缓存注入器，同类型多个实例共享注入器，不是每实例一个

本章建立的组件模型基础将在后续两章中得到直接运用：

  • 第9章（路由系统）——路由激活一个组件，本质上是创建其 LView 并触发完整的首次生命周期序列；路由切换时 ngOnDestroy 的精确触发时机（在新组件初始化之前还是之后？），以及路由守卫如何在生命周期钩子序列的前后介入，都以本章的 refreshView() 执行结构为基础。
